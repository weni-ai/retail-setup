## Phase 0 — Outline & Research

No `[NEEDS CLARIFICATION]` markers came out of `/speckit-specify`, and the
stack is fixed (Markdown). The only resolved decision worth recording:

- **Decision**: place the doc under `docs/SPEC_KIT.md` rather than at the
  repository root.
- **Rationale**: keeps the top-level free of further `*.md` clutter (the repo
  already has `CHANGELOG.md`, `DEPLOYMENT_NOTES.md`, `PR-DESCRIPTION.md`,
  `LICENSE`). `docs/` is the conventional location.
- **Alternative considered**: a `CONTRIBUTING.md` at the root — rejected
  because a future `CONTRIBUTING.md` will likely cover more than Spec Kit, and
  splitting it later costs more than placing the doc correctly now.

No `research.md` is produced; the table above is the entire Phase 0 output.

## Phase 1 — Design Artifacts

- **data-model.md**: not produced; no entities.
- **contracts/**: not produced; no external APIs.
- **quickstart.md**: not produced; the deliverable IS the quickstart.
- **Agent context**: `.cursor/rules/specify-rules.mdc` already exists and
  points the agent at the current plan; no further wiring needed.

Additional design decisions:

- The current `.gitignore` excludes `docs/` (line 143). For this feature to
  ship the doc inside `docs/`, the ignore rule MUST be replaced with a more
  precise rule that preserves the existing intent (ignore generated docs
  artefacts) while explicitly tracking `docs/SPEC_KIT.md`. The chosen
  approach: convert the blanket `docs/` ignore into a negated rule pair
  (`docs/` followed by `!docs/SPEC_KIT.md`) so only this file is tracked.
  Future doc additions follow the same pattern.

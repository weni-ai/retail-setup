# Feature Specification: Template Sample Validation Endpoint for Direct Send

**Feature Branch**: `004-template-sample-validation`

**Created**: 2026-05-26

**Status**: Draft

**Input**: User description: "We need to create an endpoint that our frontend will call to send a sample with updated content to meta and check if the template is a valid one for the direct send api. The endpoint body should be similar to the `/api/v3/templates/<template_uuid>` PATCH, the frontend has the same informations and instead of requesting a template update directly, it will request a sample analysis, if the analysis category response is utility then we should automatically update the template content internally so that future direct sends use the new content. The endpoint will receive a template-like body and should convert it to Meta's required message-like body before sending the request for the samples; we have a few examples in `docs/direct-send-api-beta-integration.md:567-685` and additional reference at <https://developers.facebook.com/documentation/business-messaging/whatsapp/messages/interactive-cta-url-messages>. We should save the content locally in the format that our direct send broadcast supports — Meta API and Flows API have differences, so the locally saved template information must remain consistent with what the Direct Send broadcast consumes when calling Flows."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Operator validates a content edit against Meta before mutating the local template (Priority: P1)

A retail operator edits the body, header, footer, or buttons of a Direct Send-eligible template in the frontend. Instead of pushing the change straight to the local store (which is what `PATCH /api/v3/templates/<template_uuid>` does today and is what makes the template start broadcasting with the new content on the next dispatch attempt), the operator clicks "Validate sample" — and the frontend posts the same edit payload to a NEW dedicated sample-validation endpoint. The endpoint translates the local-shape payload into Meta's `message_samples` wire shape (`type: text`, `type: interactive` with `cta_url` or `button` action), forwards a sample to Meta against the WABA that owns the template, and waits for Meta's synchronous response. If Meta classifies the sample as `UTILITY`, the local template's `metadata` (and a new `Version` row, matching the existing `PATCH` behavior) is updated with the proposed content so the next Direct Send broadcast uses the new wording; the operator gets the `UTILITY` verdict on screen and knows the template will keep broadcasting. If Meta classifies the sample as `MARKETING` / `AUTHENTICATION` / anything other than `UTILITY`, the local template is NOT mutated (no `metadata` rewrite, no new `Version`), the existing approved content keeps being used by Direct Send broadcasts, and the operator sees the rejected category in the response and can revise the content before resubmitting.

**Why this priority**: This is the entire reason for the feature. Without it, the operator's only options today are (a) push the edit through the existing `PATCH /api/v3/templates/<template_uuid>` endpoint and hope Meta doesn't downgrade the resulting Direct Send template's category later — at which point the `template_correct_category_detection` webhook (spec 003) will FLAG the version and silently kill the next broadcast — or (b) sit blind. The sample endpoint moves the category-check from a post-broadcast surprise to a pre-broadcast green-light, eliminating the FLAGGED-after-the-fact failure mode for content edits the operator can preview.

**Independent Test**: With a Direct Send-eligible template (an OrderStatus / abandoned-cart agent's template whose `Version.status == "APPROVED"`) and a proposed body edit that a human would obviously classify as UTILITY (e.g. "Your order #{{1}} has been shipped"), post the edit to the new sample endpoint. Verify: (a) the endpoint returns HTTP 200 with `{"category": "UTILITY", ...}` and an updated `metadata`; (b) the local `Template.metadata.body` now contains the new body; (c) a new `Version` row exists for the template with the new content; (d) the next Direct Send broadcast attempt against that template renders the new body in the outgoing payload. Repeat with a body edit that a human would classify as MARKETING (e.g. "Get 50% off this weekend only!"); verify the endpoint returns HTTP 200 with `{"category": "MARKETING", ...}`, the local `Template.metadata.body` is UNCHANGED, no new `Version` row was created, and the next Direct Send broadcast renders the old body.

**Acceptance Scenarios**:

1. **Given** a Direct Send-eligible Template `T` with `current_version.status == "APPROVED"` and a `metadata.body` of `"Olá {{1}}, seu pedido foi confirmado"`, **When** the operator posts the new endpoint with `{template_body: "Olá {{1}}, seu pedido #{{2}} foi enviado e chega amanhã", template_body_params: ["João", "1234"], app_uuid: A, project_uuid: P}` and Meta's `message_samples` endpoint returns `{"success": true, "category": "UTILITY"}` (Meta SAW the substituted body `"Olá João, seu pedido #1234 foi enviado e chega amanhã"` per A7's outbound substitution rule), **Then** the response is HTTP 200 with body `{"category": "UTILITY", "template_updated": true, ...}`, the local `Template.metadata.body` is rewritten to the new value with `{{N}}` placeholders PRESERVED (NOT substituted — A7's substitution is outbound-only), a new `Version` row is created with `status = "APPROVED"` directly (per FR-006d — Meta's sample verdict IS the Direct Send approval; no transitional `PENDING` state), `Template.current_version` is repointed to the new Version in the same write, and NO `task_create_template` Integrations push fires (per A10 — Direct Send templates do not use the Integrations engine as a content source; the dispatch path reads local `metadata` directly).
2. **Given** the same setup as scenario 1, **When** Meta returns `{"success": true, "category": "MARKETING"}` (or `"AUTHENTICATION"`), **Then** the response is HTTP 200 with body `{"category": "MARKETING", "template_updated": false, ...}`, the local `Template.metadata` is UNCHANGED, NO new `Version` row is created, and `Template.current_version` is NOT advanced. The operator sees the rejected category and revises the content client-side before resubmitting.
3. **Given** the same setup as scenario 1, **When** the payload also carries `template_header`, `template_footer`, and `template_button` fields (the same fields accepted by `UpdateTemplateContentSerializer`), **Then** the endpoint translates them into the matching `interactive`-typed Meta sample shape per `docs/direct-send-api-beta-integration.md:588-628` (CTA URL header / button) or `docs/direct-send-api-beta-integration.md:630-674` (reply buttons), sends a single sample request, and applies the same conditional update rule from scenarios 1 and 2 to the FULL payload (header, footer, buttons all update together — or none of them do).

### User Story 2 - Local content stays in lockstep with what the Direct Send broadcast renders (Priority: P1)

When the sample is approved as `UTILITY`, the local template's `metadata` MUST converge on the EXACT same canonical local shape that the Direct Send broadcast payload builder (`retail.agents.domains.agent_webhook.services.direct_send_payload_builder`) reads at dispatch time — so the next dispatch renders the new content correctly with zero additional changes. The Meta `message_samples` shape (`type: interactive`, `interactive.action.parameters.{display_text, url}`) and the local Direct Send canonical metadata shape (`metadata.buttons[*]` carrying `type=URL|QUICK_REPLY` entries with separate `text` / `url` fields, `metadata.header={header_type: TEXT|IMAGE, text: ...}`, etc.) DIFFER — the endpoint MUST persist the LOCAL canonical shape, NOT Meta's wire shape. The wire shape is computed ONLY for the outbound sample call and discarded after Meta responds. The legacy Flows broadcast path is not in play here: per FR-002a the endpoint refuses non-Direct-Send-eligible templates, and per `Broadcast.build_message` (`retail/agents/domains/agent_webhook/services/broadcast.py:932`) Direct Send-eligible templates ALWAYS dispatch via `build_direct_send_message`, never via the legacy `build_broadcast_template_message`.

**Why this priority**: A schema drift between what's persisted and what the Direct Send broadcast renderer expects would silently break dispatches for the validated template — the operator's "UTILITY" green-light would be followed by a broken broadcast. This is the same priority as US1 because US1 without US2 produces a worse outcome than today's `PATCH` endpoint: it would convince the operator the content is safe and then break it.

**Independent Test**: With a template that has a body + header (TEXT) + URL button + QUICK_REPLY button, post a UTILITY-classifying edit. After the endpoint returns HTTP 200 with `category=UTILITY`, fetch the same template via the existing `GET /api/v3/templates/<uuid>` endpoint. Verify that the response shape matches what `PATCH /api/v3/templates/<template_uuid>` would have produced for the same payload — same `metadata.body`, same `metadata.header.{header_type, text}`, same `metadata.buttons[*].{type, text, url}` shape. Trigger a Direct Send broadcast against the template (e.g. via the order-status dispatch path) and confirm the broadcast payload's `msg.body`, `msg.header`, `msg.cta_message`, and `msg.quick_replies` fields render the new content correctly.

**Acceptance Scenarios**:

1. **Given** a template whose current `metadata.buttons` is `[{"type": "URL", "text": "Track", "url": "https://example.com/{{1}}"}]`, **When** the operator posts the endpoint with `template_button = [{"type": "URL", "text": "Track order", "url": {"base_url": "https://example.com", "url_suffix_example": "abc"}}]` (the same shape `UpdateTemplateContentSerializer` accepts today) and Meta classifies the sample as `UTILITY`, **Then** the local `Template.metadata.buttons[0]` is rewritten to the canonical local shape `{"type": "URL", "text": "Track order", "url": "https://example.com/{{1}}"}` (the same shape produced by `UpdateNormalTemplateStrategy` → `TemplateTranslationAdapter`), and the on-wire Meta sample carried `interactive.action.parameters.{display_text: "Track order", url: "https://example.com/{{1}}"}` (the Meta shape per the docs ref).
2. **Given** a template with an IMAGE header whose new sample carries a base64-encoded image, **When** the operator posts the endpoint with `template_header = <base64-data-uri>` and Meta classifies the sample as `UTILITY`, **Then** the image is uploaded to S3 by `TemplateMetadataHandler.post_process_translation` (the same upload path used by `UpdateNormalTemplateStrategy`), the local `metadata.header.text` is rewritten to the resulting S3 URL (NOT the base64 blob), and the on-wire Meta sample's `interactive.header.image.link` carried the resolved URL (NOT the base64 — Meta expects a `link` per the CTA URL docs).
3. **Given** any UTILITY-classified sample (any combination of body / header / footer / buttons / parameters / language), **When** the endpoint completes the conditional update, **Then** NO `task_create_template.delay(...)` Integrations push fires (per A10 — Direct Send templates skip the Integrations push because dispatch reads local `metadata` directly and because pushing would create a conflicting legacy template at Meta with the same `weni_<name>_<timestamp>` name, triggering Meta's error 132021 "A template with the same name already exists" at the next Direct Send dispatch per `docs/direct-send-api-beta-integration.md:976`). The next Direct Send broadcast against the template reads `Template.metadata` and `Template.current_version.template_name` directly and renders the new content on the very next dispatch attempt — no asynchronous convergence window, no waiting on a status webhook.

### User Story 3 - Sample submission failures don't corrupt local template state (Priority: P2)

Meta's `message_samples` endpoint is an external dependency and can fail synchronously (HTTP 4xx / 5xx, network timeout) or return a body that doesn't carry a `category` field (e.g. `{"success": false, "error": ...}`). When this happens, the local template MUST NOT be mutated (no `metadata` rewrite, no new `Version` row, no `current_version` advance), and the endpoint MUST surface a deterministic error response to the frontend so the operator knows the validation didn't complete and that they can retry safely.

**Why this priority**: The whole point of US1 is to use Meta's verdict as the gate. If a Meta-side failure silently degrades into "treat as not-UTILITY" (so we skip the update — the safe direction) the operator would think their edit was rejected when it wasn't actually evaluated; if the failure silently degrades into "treat as UTILITY" (so we apply the update — the unsafe direction) the operator would think their edit was approved when Meta never saw it. Both outcomes are worse than an explicit "couldn't validate" error. This is P2 (not P1) because the dominant flow is the success path; the failure path must be correct but is exceptional.

**Independent Test**: Configure the Meta client to raise a `CustomAPIException` (the existing infra-error type used by `MetaService`). Post a sample to the endpoint. Verify the response is HTTP 502 with a body like `{"detail": "Meta sample submission failed", "error_code": "meta_unavailable"}`, the local template's `metadata` and `Version` rows are untouched (verified by reading the template before and after the call), `Template.current_version` is unchanged, and an audit log line tagged `meta_error` was emitted.

**Acceptance Scenarios**:

1. **Given** any well-formed sample payload, **When** the Meta client raises `CustomAPIException` (HTTP 5xx / network failure on Meta's side), **Then** the response is HTTP 502 with a body indicating the upstream failure, the local template is unchanged, and an audit log entry tagged `meta_error` carries the Meta error code / message and the template identifier.
2. **Given** any well-formed sample payload, **When** Meta returns HTTP 200 with a body that does NOT carry a `category` field (e.g. `{"success": false, "error": {...}}` — same error envelope shape used elsewhere in the document), **Then** the response is HTTP 502 with body indicating a category was not returned, the local template is unchanged, and an audit log entry tagged `meta_invalid_response` carries the raw Meta response body for debugging.
3. **Given** any well-formed sample payload, **When** the project's WABA id cannot be resolved (e.g. the Project's onboarding config is missing the `wpp-cloud` channel's `waba_id` field — see Assumption A2), **Then** the endpoint returns HTTP 400 with body indicating the WABA is not configured for this project, no Meta call is made, and the local template is unchanged.
4. **Given** a Template whose `integrated_agent` is `null` OR whose `integrated_agent.config["direct_send"]` is falsy (e.g. a legacy assignment from before the Direct Send rollout), **When** the operator posts any well-formed sample payload, **Then** the endpoint returns HTTP 400 with `{"detail": "Template is not Direct Send-eligible", "error_code": "not_direct_send_eligible"}` (per FR-002a), no Meta sample-API call is made (sparing Meta-side quota), and the local template is unchanged. An audit log entry tagged `not_direct_send_eligible` carries the template and IntegratedAgent identifiers so operators can identify why the call failed.

### User Story 4 - Frontend gets a payload it can render without translation (Priority: P3)

The frontend posts the SAME body it would post today to `PATCH /api/v3/templates/<template_uuid>` (per the user's description: "the endpoint body should be similar to the `/api/v3/templates/<template_uuid>`, the frontend has the same informations"). The endpoint's HTTP 200 response also carries the same `Template` representation the frontend already knows how to render (the `ReadTemplateSerializer` shape) so the frontend can refresh the displayed template in-place after a UTILITY classification — no second `GET` round-trip is required.

**Why this priority**: P3 because the UX win is marginal (a saved round-trip) and any deviation from this story (e.g. the frontend has to refetch) is annoying but not broken. Pinning the request/response shape to mirror the existing `PATCH` endpoint is the cheap path and prevents accidental schema divergence in the frontend.

**Independent Test**: Diff the JSON schema of the new endpoint's request body against `UpdateTemplateContentSerializer`'s schema. The fields MUST match (`template_body`, `template_header`, `template_footer`, `template_button`, `template_body_params`, `app_uuid`, `project_uuid`, `parameters`, `language`). Diff the JSON schema of the new endpoint's HTTP 200 response (on a UTILITY classification) against `ReadTemplateSerializer`'s schema; the `template` field of the response MUST conform to `ReadTemplateSerializer`. The response MUST additionally carry a small wrapper with at minimum the `category` Meta returned and a boolean `template_updated`.

**Acceptance Scenarios**:

1. **Given** the frontend has a working call site for `PATCH /api/v3/templates/<template_uuid>`, **When** the operator switches the call to the new sample endpoint (same URL path or a path that differs only in the trailing segment), **Then** the request body schema is byte-for-byte compatible (no new required fields, no removed required fields, no renamed fields) and the existing client-side validation continues to pass.
2. **Given** a UTILITY-classified sample response, **When** the frontend reads the response body, **Then** the `template` key contains a `ReadTemplateSerializer`-shaped object the frontend can substitute into its template state without re-fetching, and the wrapper carries `{"category": "UTILITY", "template_updated": true, "meta_sample_response": <raw_meta_body>}` so the frontend can also display Meta's raw verdict to the operator for transparency.
3. **Given** a non-UTILITY-classified sample response, **When** the frontend reads the response body, **Then** the `template` key is the UNCHANGED `ReadTemplateSerializer` object (so the frontend can confirm visually that no edit was applied) and the wrapper carries `{"category": "MARKETING", "template_updated": false, "meta_sample_response": <raw_meta_body>}`.

### Edge Cases

- **Template UUID does not exist**: the endpoint MUST return HTTP 404 with a body matching the shape returned by `UpdateTemplateContentUseCase._get_template`'s `NotFound` (DRF default). No Meta call is made. (Parity with the existing `PATCH /api/v3/templates/<template_uuid>` endpoint.)
- **Template is not Direct Send-eligible** (`template.integrated_agent` is `null` OR `template.integrated_agent.config.get("direct_send", False)` is falsy — e.g. a legacy-path OrderStatus IntegratedAgent assigned before the Direct Send rollout, or a custom template with no IntegratedAgent FK): the endpoint MUST refuse the request at HTTP 400 with `error_code = "not_direct_send_eligible"` BEFORE making any Meta call (per FR-002a). No Meta sample-API quota is consumed and no local state changes. Operators are expected to use the existing `PATCH /api/v3/templates/<uuid>/` endpoint for content edits on non-Direct-Send templates — that endpoint already handles the legacy Flows approval cycle correctly. (See Assumption A8.)
- **Template's `current_version.status == "FLAGGED"`** (the template was flagged by the spec 003 webhook): the endpoint still allows sample submission. A UTILITY-classified sample writes a new `Version` row with `status = "APPROVED"` directly (per FR-006d) AND advances `Template.current_version` to the new Version in the same write — the previously-FLAGGED Version is RETAINED in the `versions` history (it's NOT mutated) but is no longer the current one. The Direct Send dispatch gate (spec 002's FR-007 / FR-012) re-admits the template on the next broadcast attempt because the gate reads `template.current_version.status`, which is now `APPROVED`. This makes the sample-validation endpoint a THIRD recovery channel for FLAGGED templates, joining the spec 003 auto-demote webhook and the operator-driven `UpdateTemplateUseCase` PATCH path described in spec 003's FR-014. Operators can clear a FLAG by submitting revised content that Meta accepts as UTILITY without needing any other intervention.
- **Sample carries a header but the header is missing required fields for the chosen Meta sample shape** (e.g. an `IMAGE` header without a `link` AND without a `media_id` per `docs/direct-send-api-beta-integration.md:603-609`): the endpoint MUST return HTTP 400 with a field-level error BEFORE calling Meta. The local template is unchanged.
- **Sample carries BOTH a CTA URL button AND reply buttons in the same payload**: the Meta docs treat these as two disjoint `interactive.type` shapes (`cta_url` vs `button`). The endpoint MUST reject this combination at the serializer level with HTTP 400. Operators MUST choose ONE button mode per sample.
- **Sample carries MORE than the Meta cap of buttons** (Direct Send supports max 1 CTA button OR max 3 reply buttons per `docs/direct-send-api-beta-integration.md:174-176`): the endpoint MUST reject with HTTP 400 at the serializer level. Local template state is unchanged.
- **Payload's `template_body` exceeds the Meta cap of 1024 characters** (per `docs/direct-send-api-beta-integration.md:167`): the endpoint MUST reject with HTTP 400 at the serializer level. The same length caps apply to `template_header` (60 chars), `template_footer` (60 chars), and each button label (20 chars) per the same docs reference.
- **Body / header / footer / button text or url contains `{{N}}` placeholder variables**: placeholders are SUBSTITUTED with operator-supplied values (`template_body_params`, positional) BEFORE the sample body is sent to Meta — per A7's outbound substitution rule. Meta classifies the sample based on the rendered content (the realistic customer-facing string), not the placeholder skeleton, so the verdict reflects what end users would actually receive. The LOCAL persistence path keeps placeholders verbatim (the Version's stored `metadata.body` retains `{{1}}`, `{{2}}`, etc.) so the Direct Send broadcast renderer (`substitute_template_variables`) can re-substitute against real per-contact variable values at dispatch time. When a `{{N}}` placeholder has no matching index in `template_body_params`, the substitution helper logs a WARNING and substitutes to empty string — same behavior the production dispatcher exhibits today (`retail.agents.domains.agent_webhook.services.direct_send_payload_builder.substitute_template_variables`). Operators are expected to provide values for every placeholder they include; the warning-and-empty behavior is the existing safe-default and is NOT changed by this feature.
- **Meta returns `{"success": true, "category": "UTILITY"}` but the local update fails partway through** (e.g. DB write fails, S3 image upload fails — note that `task_create_template.delay` is never invoked on this endpoint per A10, so Celery-enqueue failures are not a relevant failure mode here): the endpoint MUST raise the local-side error (HTTP 500) AFTER the Meta call succeeded — Meta saw the sample but the local mutation didn't land. The operator should retry; the second attempt will re-call Meta. Meta's sample endpoint is idempotent in the sense that submitting the same sample twice produces the same verdict, so the retry is safe. The audit log MUST emit a `local_update_failed_after_meta_approval` line so operators can investigate.
- **Concurrent sample submissions for the same template UUID**: two operators submit different edits for the same template at the same time. The endpoint does NOT lock the template row — both Meta calls fire, both verdicts return, and both updates apply in `last-writer-wins` order (the standard Django concurrency model under default isolation). The operator UX should typically serialize this via the frontend (only one edit dialog open at a time per template) but the backend does NOT enforce it. The audit log carries both calls so operators can reconstruct the timeline.
- **Operator submits a sample whose content is BYTE-FOR-BYTE identical to the current `metadata`**: the endpoint still calls Meta (Meta's verdict is content-dependent so an identical resubmission could legitimately produce a different verdict — though in practice Meta's classifier is deterministic for identical inputs). On a UTILITY classification the local update still creates a new `Version` row with `status = "APPROVED"` and advances `Template.current_version` to it (per FR-006d — `_create_version` always creates a row, it does not dedup on content). The previous current_version is retained in `Template.versions` history. This is consistent with the existing `PATCH /api/v3/templates/<uuid>/` row-creation behavior — only the terminal `status` and the in-line `current_version` advance deviate per FR-006d.
- **Project is `is_blocked=True`** (broadcasts are paused): the endpoint still processes the sample submission. Blocking gates outbound CUSTOMER deliveries (broadcasts) — it does NOT gate operator preview / content-validation flows. The local update fires normally on a UTILITY classification.

## Requirements *(mandatory)*

### Functional Requirements

#### Endpoint surface

- **FR-001**: The system MUST expose a new HTTP POST endpoint under the existing `retail.templates.urls` namespace (the same router that owns `PATCH /api/v3/templates/<uuid>/`). The new endpoint MUST sit on the same `TemplateViewSet` (e.g. as a detail-route action) so the URL path mirrors the existing template-update path. The recommended URL is `POST /api/v3/templates/<template_uuid>/sample/` (DRF detail action `sample`).
- **FR-002**: The endpoint MUST use the SAME authentication / authorization stack the existing `PATCH /api/v3/templates/<uuid>/` endpoint uses today: `permission_classes = [IsAuthenticated, HasProjectPermission]`. No new permission class is introduced. (Rationale: the new endpoint is a less-destructive cousin of the existing PATCH; gating it identically lets the same frontend session call it without an extra auth-token round trip.)
- **FR-002a**: The endpoint MUST be gated on Direct-Send-eligibility AT THE USE-CASE LAYER (not as a DRF permission class — the check requires the Template row, which the permission layer cannot reach). After loading the Template by `template_uuid`, the use case MUST refuse the request when `template.integrated_agent` is `null` OR when `template.integrated_agent.config.get("direct_send", False)` is falsy — returning HTTP 400 with body `{"detail": "Template is not Direct Send-eligible", "error_code": "not_direct_send_eligible"}`. No Meta call is made and no local mutation occurs. The eligibility flag is the SAME `IntegratedAgent.config["direct_send"]` snapshot set at assignment time by `retail.agents.domains.agent_integration.usecases.assign.AssignAgentUseCase` and read at dispatch time by `Broadcast.build_direct_send_message` per spec 002 — so eligibility for this endpoint is exactly the same predicate the dispatch path uses, no parallel definition is introduced. The check is free: `template.integrated_agent` is already loaded via `select_related("integrated_agent")` on the existing Template lookup, which the planned implementation already needs (the existing `UpdateNormalTemplateStrategy._update_common_metadata` invokes `get_agent_config(template.integrated_agent)`, so the relationship is hot regardless).
- **FR-002b**: The endpoint MUST refuse the request when the `Project-Uuid` HTTP header (the trust source consumed by `HasProjectPermission` per FR-002) does NOT match the `project_uuid` field in the request body. The check runs at the serializer layer (so it fires BEFORE any DB lookup, any WABA resolution, and any Meta call) and returns HTTP 400 with body `{"detail": "Project-Uuid header does not match body project_uuid", "error_code": "project_uuid_mismatch"}`. Rationale: `HasProjectPermission` (`retail/internal/permissions.py:67`) authorizes against the `Project-Uuid` header, while FR-005a uses `dto.project_uuid` from the body for WABA-id resolution. Without this equality check, an operator authorized for project A could submit a body with `project_uuid=B` and route a Meta sample call to project B's WABA — a SC-008 cross-tenant isolation violation. The legacy `PATCH /api/v3/templates/<uuid>/` endpoint inherits the same theoretical vector but predates this guarantee; the new endpoint closes it at the source. The serializer reads the header via the request's `context["request"]` (the standard DRF pattern) and emits the field-level error before `is_valid(raise_exception=True)` would return cleanly.
- **FR-003**: The endpoint's request body MUST be byte-for-byte compatible with `retail.templates.serializers.UpdateTemplateContentSerializer` — the SAME serializer (or a literal subclass that adds nothing new and removes nothing) MUST validate the input. The required and optional fields are: `template_body` (string, optional but at least one of `template_body` / `template_header` / `template_footer` MUST be present — same `validate` clause as today), `template_header` (string, optional — can be plain text, an HTTP(S) URL, a base64-data-URI, or an existing S3 URL), `template_footer` (string, optional), `template_button` (list, optional), `template_body_params` (list, optional), `app_uuid` (string, REQUIRED), `project_uuid` (string, REQUIRED), `parameters` (list of `ParameterSerializer`, optional), `language` (string, optional). No new required fields are added by this feature.
- **FR-003a**: Field-level length caps from `docs/direct-send-api-beta-integration.md:167-176` MUST be enforced at the serializer layer BEFORE the Meta call: `template_body` ≤ 1024 chars, `template_header` ≤ 60 chars (when it's a TEXT header — IMAGE headers carry a URL and the URL itself is not capped here), `template_footer` ≤ 60 chars, each button's `text` ≤ 20 chars, `template_button` carrying type `URL` MUST contain ≤ 1 entry of that type, `template_button` carrying type `QUICK_REPLY` MUST contain ≤ 3 entries of that type, and a single `template_button` payload MUST NOT mix `URL` entries with `QUICK_REPLY` entries (Meta's `cta_url` and `button` interactive types are disjoint per the docs ref at `docs/direct-send-api-beta-integration.md:594-674`). Length-cap failures MUST surface as HTTP 400 field-level errors. (Rationale: rejecting locally avoids burning Meta sample-API quota on payloads Meta would reject anyway.)

#### Local-shape → Meta-shape translation (outbound only)

- **FR-004**: The endpoint MUST translate the validated local-shape payload into a Meta `message_samples`-compatible JSON body BEFORE calling Meta. The translation MUST handle:
  - **Body-only** payloads (only `template_body` set, no header / footer / buttons) → Meta shape `{"type": "text", "text": {"body": <template_body>}}` per `docs/direct-send-api-beta-integration.md:579-586`.
  - **Payloads with a CTA URL button** (a single `template_button` entry whose `type == "URL"`) → Meta shape `{"type": "interactive", "interactive": {"type": "cta_url", "header": {...}?, "body": {"text": <template_body>}, "footer": {"text": <template_footer>}?, "action": {"name": "cta_url", "parameters": {"display_text": <button.text>, "url": <resolved_url>}}}}` per `docs/direct-send-api-beta-integration.md:588-628` and the dev-docs ref at <https://developers.facebook.com/documentation/business-messaging/whatsapp/messages/interactive-cta-url-messages>.
  - **Payloads with reply buttons** (one or more `template_button` entries whose `type == "QUICK_REPLY"`, max 3) → Meta shape `{"type": "interactive", "interactive": {"type": "button", "header": {...}?, "body": {"text": <template_body>}, "footer": {"text": <template_footer>}?, "action": {"buttons": [{"type": "reply", "reply": {"id": "<title-derived>", "title": <button.text>}}, ...]}}}` per `docs/direct-send-api-beta-integration.md:630-674`.
- **FR-004a**: The `header` sub-object inside the interactive shape MUST follow Meta's discriminated union (per `docs/direct-send-api-beta-integration.md:599-610`): for a TEXT header, emit `{"type": "text", "text": <header_text>}`; for an IMAGE header, emit `{"type": "image", "image": {"link": <resolved_image_url>}}` (and `media_id` is NOT used by this endpoint — we always resolve to a public URL via the existing S3 upload path before sending the sample). When the local `template_header` is a base64-data-URI, the endpoint MUST resolve it to an S3 URL using the SAME `TemplateMetadataHandler._upload_header_image` helper that `UpdateNormalTemplateStrategy` uses today, BEFORE building the Meta sample body — so the Meta sample carries a real URL Meta can fetch.
- **FR-004b**: The `interactive.action.parameters.url` field for CTA URL buttons MUST carry the FULLY-RESOLVED URL — `base_url` + `{{1}}` placeholder substitution if `url_suffix_example` was supplied — using the same `ButtonTransformer._is_button_format_already_translated` / `ensure_protocol` / `append_placeholder_if_needed` / `normalize_url_if_needed` helpers already used by `UpdateNormalTemplateStrategy`. The endpoint MUST NOT bypass these helpers; the canonical URL string MUST be identical to what `TemplateTranslationAdapter.adapt` would produce for the same input.
- **FR-004c**: The `interactive.action.buttons[i].reply.id` field for reply buttons (Meta requires a unique `id` per button per the docs ref at `docs/direct-send-api-beta-integration.md:665-668`) MUST be derived deterministically from the button's `text` (e.g. lowercased + non-alphanum stripped + truncated). The `id` is for Meta's internal sample categorization only; it is NEVER persisted locally. If the derivation produces duplicates across the buttons of a single payload, the endpoint MUST append a positional suffix to break the tie (`<base>_2`, `<base>_3`).
- **FR-004d**: The translation MUST be a pure function (no DB access, no I/O) — it takes the validated DTO and returns a plain Python dict that mirrors Meta's wire shape. This separation ensures the Meta shape can be unit-tested without database setup and lets the Meta call boundary handle errors uniformly.
- **FR-004e**: The translation MUST substitute every `{{N}}` placeholder in the body, header text, footer, and each button's `text` and `url` BEFORE building the Meta sample body, using the same `substitute_template_variables` helper that `Broadcast.build_direct_send_message` uses at dispatch time (`retail.agents.domains.agent_webhook.services.direct_send_payload_builder`). The variable source is `template_body_params` — a positional list mapped to `{{1}}`, `{{2}}`, ... — matching the contract the existing Direct Send broadcast path already uses for the body. Substitution is OUTBOUND-ONLY: the LOCAL persisted `metadata` MUST retain raw `{{N}}` placeholders (NOT the substituted strings) so the dispatch-time renderer can re-substitute against real per-contact values. Missing indices substitute to empty string and emit a WARNING (same behavior as the dispatch-time helper).

#### Outbound sample call to Meta

- **FR-005**: The endpoint MUST call Meta's `POST /<WABA_ID>/message_samples` endpoint (per `docs/direct-send-api-beta-integration.md:567-685`) via a new method on the existing `MetaClient` / `MetaService` (e.g. `MetaService.submit_template_sample(waba_id, sample_body)`). The new client method MUST use the same `_json_headers` / `make_request` plumbing as `create_flow` / `register_public_key`, the same `META_API_URL` setting, and the same `CustomAPIException` envelope on infrastructure failures.
- **FR-005a**: The WABA id passed to Meta MUST be resolved from the Project's `ProjectOnboarding.config["channels"]["wpp-cloud"]["channel_data"]["waba_id"]` (the same path `configure_wpp_cloud.py` reads — see Assumption A2). When the project has no `ProjectOnboarding` row, no `wpp-cloud` channel in the onboarding config, or the `waba_id` field is empty / missing, the endpoint MUST return HTTP 400 with a deterministic error indicating "WABA not configured for this project" — no Meta call is made and the local template is unchanged.
- **FR-005b**: The endpoint MUST treat the following Meta responses as the AUTHORITATIVE classification surface:
  - Response body containing a top-level `category` field of EXACTLY the string `"UTILITY"` → treat as UTILITY (the conditional-update gate fires).
  - Response body containing a top-level `category` field of any other non-empty string (`"MARKETING"`, `"AUTHENTICATION"`, or any future value Meta may introduce) → treat as NOT-UTILITY (the conditional-update gate does NOT fire).
  - Response body NOT containing a top-level `category` field (or `category` is `null` / empty string), or `success` field is explicitly `false` → treat as a Meta-side failure per FR-005c.
- **FR-005c**: The endpoint MUST translate Meta-side error responses into deterministic HTTP responses:
  - `CustomAPIException` raised by the Meta client (HTTP 5xx, network timeout) → HTTP 502 with body `{"detail": "Meta sample submission failed", "error_code": "meta_unavailable"}`.
  - Meta returns HTTP 200 but the body lacks a `category` field or carries `success: false` → HTTP 502 with body `{"detail": "Meta did not return a category", "error_code": "meta_invalid_response", "meta_response": <raw>}`.
  - The `MetaService.submit_template_sample(...)` method MUST surface the raw Meta response body to the use case (the existing `MetaService` pattern of swallowing exceptions into `None` is NOT followed here — the use case needs the raw body to surface to the frontend on errors per US3 AS2).

#### Conditional local update

- **FR-006**: The endpoint MUST update the local template ONLY when Meta's response body carries `category == "UTILITY"`. The update path MUST reuse the EXISTING metadata-building, S3-upload, and version-creation machinery from `UpdateNormalTemplateStrategy` / `UpdateCustomTemplateStrategy` (`TemplateTranslationAdapter.adapt`, `TemplateMetadataHandler.build_metadata`, `TemplateMetadataHandler.post_process_translation`, `TemplateBuilderMixin._create_version`) — the use case MUST NOT reimplement these. The Integrations push (`task_create_template.delay`) is the ONE step from the existing strategy that this endpoint MUST NOT invoke; see FR-006d for the rationale (per A10, Direct Send templates do not use the Integrations engine as a content source, and pushing would actively break Direct Send dispatch via Meta error 132021). The use case's job is the SAMPLE call and the GATING decision; the metadata-adapt + version-create + abandoned-cart-sync steps are delegated. The behavioral deviations from the existing `PATCH /api/v3/templates/<uuid>/` strategy are pinned by FR-006d below.
- **FR-006a**: When `category == "UTILITY"`, the use case MUST invoke the metadata-build + version-create steps with the validated payload (same shape `PATCH /api/v3/templates/<uuid>/` produces, after `template_uuid` is injected from the URL path). Because the legacy `PATCH` path runs `metadata-build → version-create → notify-integrations` as a single `update_template` method and the new endpoint needs `metadata-build → version-create (APPROVED + current_version advance)` WITHOUT the Integrations push, the recommended composition is to EXTRACT the existing `UpdateNormalTemplateStrategy.update_template` body into smaller building-block helpers — for example `_apply_metadata_update(template, payload) -> translation_payload` (encapsulates `_update_common_metadata` + `template.save(metadata)` + `_sync_abandoned_cart_image_config`) and `_create_version(template, payload, *, status, advance_current_version)` (encapsulates the existing `_create_version` plus the optional `current_version` repointing). The PATCH endpoint composes `_apply_metadata_update → _create_version(status="PENDING", advance_current_version=False) → _notify_integrations`; the new sample endpoint composes `_apply_metadata_update → _create_version(status="APPROVED", advance_current_version=True)` and STOPS. Parameterization on the helpers (not on `update_template` itself) keeps each method single-purpose and the new endpoint never invokes the Integrations push.
- **FR-006b**: When `category != "UTILITY"`, the use case MUST NOT invoke any update machinery, MUST NOT mutate the local `Template.metadata`, MUST NOT create a new `Version` row, MUST NOT advance `Template.current_version`, and MUST NOT clear any cache. The local template is returned to the frontend UNCHANGED. (The `task_create_template.delay(...)` Integrations push is structurally not fired by this endpoint regardless of category — see FR-006 / A10 — so the non-UTILITY path inherits that guarantee trivially.)
- **FR-006c**: When the local update fails AFTER a UTILITY classification (DB write fails, S3 upload fails — the Integrations-push Celery enqueue is not a failure mode here per A10), the use case MUST propagate the local-side exception as HTTP 500 (Django/DRF default). The Meta-side state (the submitted sample) is left as-is; the operator's retry produces a fresh sample submission. The audit log MUST emit a `local_update_failed_after_meta_approval` event with the original error so operators can investigate. (Rationale: rolling back the Meta sample is not possible — Meta's sample API is fire-and-forget for the categorization slot. The safer behavior is to let the operator retry and trust idempotency on the Meta side.)
- **FR-006d**: When `category == "UTILITY"`, the new `Version` row MUST be persisted with `status = "APPROVED"` (NOT the default `"PENDING"` that the existing `UpdateNormalTemplateStrategy` produces) AND `Template.current_version` MUST be advanced to the new Version in the same logical write. This rule applies ONLY to Direct Send-eligible templates — non-eligible templates never reach this branch because FR-002a refuses them upstream. The deviation from the existing `PATCH /api/v3/templates/<uuid>/` behavior is justified by the dispatch-path contract: the Direct Send broadcast reads content directly from `Template.metadata` and the template name from `Template.current_version.template_name` at dispatch time (`Broadcast.build_direct_send_message` per spec 002), so Meta's sample-API `category == "UTILITY"` verdict IS the meaningful approval signal for that path — the local "pending until `TemplatesStatusWebhook` confirms" state has no operational purpose and would only delay the next Direct Send broadcast from rendering the new content. The Integrations push (`task_create_template`) is NOT fired (per FR-006 / A10): Direct Send templates do not use the Integrations engine as a content source, AND firing the push would actively break Direct Send dispatch by creating a non-Direct-Send legacy template at Meta with the same `weni_<name>_<timestamp>` name — the next Direct Send dispatch with `direct_send_config.template_name = "weni_<name>_<timestamp>"` would collide with that legacy template and Meta would return error 132021 "A template with the same name already exists" (per `docs/direct-send-api-beta-integration.md:976`). Skipping the push mirrors how `AssignAgentUseCase` already creates Direct Send templates at assignment time (`retail/agents/domains/agent_integration/usecases/assign.py:421-438` — `status = "APPROVED"` written locally, no Integrations push). The previously-current Version row (if any) is RETAINED in `Template.versions` history — only the `current_version` pointer advances; no rows are deleted.

#### Response shape

- **FR-007**: On a UTILITY classification (post-update), the response body MUST be HTTP 200 with at minimum:
  - `category`: the string `"UTILITY"`.
  - `template_updated`: `true`.
  - `template`: the `ReadTemplateSerializer`-shaped object for the updated template (the SAME shape `PATCH /api/v3/templates/<uuid>/` returns today on success).
  - `meta_sample_response`: the raw JSON body Meta returned (so the frontend can display the verdict and the operator can see Meta's exact answer).
- **FR-007a**: On a NOT-UTILITY classification (no update applied), the response body MUST be HTTP 200 with at minimum:
  - `category`: the non-UTILITY value Meta returned (verbatim — `"MARKETING"`, `"AUTHENTICATION"`, or any future Meta value).
  - `template_updated`: `false`.
  - `template`: the `ReadTemplateSerializer`-shaped object for the UNCHANGED template (so the frontend can confirm visually that the edit was NOT applied).
  - `meta_sample_response`: the raw JSON body Meta returned.
- **FR-007b**: On a Meta-side error (FR-005c), the response is HTTP 502 with `{"detail": <human_message>, "error_code": <stable_token>}` plus an optional `meta_response` field when one was received. The local template is unchanged.
- **FR-007c**: On a serializer-level validation failure (FR-003a length caps, missing required fields, mixed button modes, etc.), the response is HTTP 400 with the DRF default field-by-field error body. No Meta call is made.
- **FR-007d**: On a "WABA not configured" failure (FR-005a), the response is HTTP 400 with `{"detail": "WABA not configured for this project", "error_code": "waba_not_configured"}`. No Meta call is made.
- **FR-007e**: On a "not Direct Send-eligible" failure (FR-002a), the response is HTTP 400 with `{"detail": "Template is not Direct Send-eligible", "error_code": "not_direct_send_eligible"}`. No Meta call is made and the local template is unchanged.
- **FR-007f**: On a "Project-Uuid header / body mismatch" failure (FR-002b), the response is HTTP 400 with `{"detail": "Project-Uuid header does not match body project_uuid", "error_code": "project_uuid_mismatch"}`. The error is surfaced as a field-level serializer error on `project_uuid`. No DB lookup, no WABA resolution, no Meta call is made.

#### Audit log

- **FR-008**: Every log line emitted by this endpoint MUST follow the structured format `[TemplateSampleValidation] <event_name>: <key=value> <key=value> ...` — the same `[TAG] event_name: k=v` shape used by `[DirectSendCategoryWebhook]` (spec 003 / FR-009), `[CART_SERVICE]`, `[ORDER_STATUS]`, `[CONVERSION_TRACKING]`, `[BroadcastDispatch]`, and `[DirectSend]`. The literal `[TemplateSampleValidation]` tag is the stable dashboard-filter token; `event_name` is a snake_case discriminator drawn from the closed enumeration in FR-008a; `key=value` pairs carry per-event context.
- **FR-008a**: The `event_name` token MUST be exactly one of:
  - `received` — request entry; emitted once at request start with the request payload (PII-redacted: `template_body` / `template_header` / `template_footer` / button text values are logged length-only, NOT verbatim, to avoid logging customer-facing content drafts).
  - `meta_sample_submitted` — outbound Meta call is about to fire; carries the resolved `waba_id`, the Meta sample `type` (`text` / `interactive` + interactive sub-type), and the local `template_uuid`.
  - `meta_sample_response` — Meta returned a body; carries the parsed `category` value (or `null` if absent), the `success` flag, and the HTTP status code.
  - `template_updated` — local update applied (UTILITY path); carries the new `version_uuid` (with `new_version_status=APPROVED` per FR-006d), the `previous_current_version_uuid` (the `Template.current_version` BEFORE the advance — `null` when the template had no prior current version), and the `previous_current_version_status` (the prior current_version's status — useful for tracking when the sample-validation channel un-flags a previously-FLAGGED template, in which case this field carries `"FLAGGED"`). No Integrations-push task id is carried because no Celery task is enqueued by this endpoint (per A10).
  - `update_skipped` — Meta returned a non-UTILITY category; carries the category value and the template's unchanged identifier (so operator dashboards can compute "non-UTILITY rate per template").
  - `meta_error` — `CustomAPIException` raised by the Meta client; carries the exception class name, the HTTP status code if available, and the Meta error code / message.
  - `meta_invalid_response` — Meta returned HTTP 200 but no `category` field; carries the raw Meta body (truncated to a safe length cap).
  - `waba_not_configured` — the project's WABA id could not be resolved; carries the `project_uuid` and the missing config path.
  - `not_direct_send_eligible` — the template's IntegratedAgent does not have `config["direct_send"] = True` (FR-002a); carries the `template_uuid`, the `integrated_agent_uuid` (or `null` when the template has no IntegratedAgent at all), and the resolved eligibility flag value.
  - `project_uuid_mismatch` — the `Project-Uuid` header did not match the body's `project_uuid` (FR-002b); carries `header_project_uuid` and `body_project_uuid` (both verbatim — these are tenant identifiers, NOT customer-facing content, so no PII redaction applies). Emitted by the serializer-level rejection BEFORE the use case runs; the view emits it directly from its serializer-error translation block.
  - `local_update_failed_after_meta_approval` — the local update raised AFTER Meta classified the sample as UTILITY; carries the local exception class name and the Meta sample id (when Meta returned one).
- **FR-008b**: Log-level discipline MUST be:
  - **INFO** for `received`, `meta_sample_submitted`, `meta_sample_response`, `template_updated`, `update_skipped` — happy-path and "expected-and-handled" events.
  - **WARNING** for `meta_invalid_response`, `waba_not_configured`, `not_direct_send_eligible`, and `project_uuid_mismatch` — expected-but-unhappy operational issues that need operator visibility but don't indicate a bug on Retail's side (a frontend that calls the endpoint with a non-eligible template or with mismatched tenant identifiers is a UX bug, but it's also a recoverable user error — the operator can retry against the legacy `PATCH` endpoint or fix the header).
  - **ERROR with `exc_info=True`** for `meta_error` and `local_update_failed_after_meta_approval` — genuine failures that warrant alerting.
- **FR-008c**: The audit log MUST NOT log the full `template_body` / `template_header` / `template_footer` / button text values verbatim. These fields may contain customer-facing drafts that operators have NOT yet approved for production; logging them at INFO level would expose draft content to anyone with log access. Log lengths and field-presence flags only (`template_body_len=42`, `template_header_present=true`).

#### Architectural boundaries

- **FR-009**: The new endpoint MUST follow the existing layered architecture: a thin view that validates input via the serializer, builds a DTO, calls a single use case, and returns a Response; a use case that orchestrates the Meta call + the conditional update + the audit log; a new method on the existing `MetaService` / `MetaClient` for the outbound Meta call; a new pure-function module (or a small class) for the local-shape → Meta-shape translation.
- **FR-009a**: The new use case MUST be framework-agnostic — no `rest_framework` imports, no `Request` / `Response` handling, no permission checks. It receives the DTO, returns a result dataclass with a `to_dict()` method, and raises domain exceptions (or DRF exceptions when they're already the lingua franca — e.g. `NotFound` for the missing template UUID, matching `UpdateTemplateContentUseCase`'s precedent).
- **FR-009b**: The new use case MUST inject its dependencies (Meta service, the extracted shared helpers from `UpdateNormalTemplateStrategy` per FR-006a, audit logger) via `__init__` with `Optional` parameters that default to concrete instances — the same DI pattern used by `UpdateTemplateContentUseCase(rule_generator=..., template_adapter=...)`. This makes the use case testable without monkey-patching.

#### Out of scope

- **FR-010**: The feature does NOT introduce a new database migration. No new table, no new column on `Template` / `Version` / `IntegratedAgent` / `Project` / `ProjectOnboarding` is added by this feature. The Meta sample id (if Meta returns one) is NOT persisted — the audit log carries it for debugging but no foreign-key reference is stored on `Version`. The PR-time verification step is `poetry run python manage.py makemigrations --check --dry-run` — the command MUST exit with code 0 (no migration would be generated). Any non-zero exit indicates an accidental schema change and the PR is blocked.
- **FR-011**: The feature does NOT change the existing `PATCH /api/v3/templates/<uuid>/` endpoint. That endpoint continues to apply edits directly without sample-validation, for operator workflows that need the older behavior (e.g. a content edit that's known-safe and doesn't need pre-validation) and to preserve backward compatibility for any caller that still uses it.
- **FR-012**: The feature does NOT alter the spec 003 `template_correct_category_detection` webhook OR the spec 002 Direct Send dispatch gate. Both continue to operate exactly as today on the new `Version` rows this feature creates. A template can still be FLAGGED post-hoc by Meta's continuous-monitoring webhook even after a UTILITY-classified sample (the sample is a one-shot pre-check; Meta's ongoing detection is independent — per the document at `docs/direct-send-api-beta-integration.md:757-781`).
- **FR-013**: The feature does NOT support batch / multi-template sample submission. One POST = one template UUID = one sample. Operators submitting samples for N templates do N requests. (Rationale: Meta's `message_samples` endpoint is one-template-per-call; the document at `docs/direct-send-api-beta-integration.md:569` recommends 3-4 samples per business onboarding but those are independent calls.)
- **FR-014**: The feature does NOT change the request shape for the existing `PATCH` endpoint. Both endpoints continue to use `UpdateTemplateContentSerializer` (or, equivalently, the new endpoint uses a serializer that is byte-for-byte schema-compatible with the existing one); a frontend that calls both endpoints can serialize the same form-state to both call sites.

### Key Entities *(include if feature involves data)*

- **Template**: The local Retail-side representation of a WhatsApp message template. Persisted in `retail.templates.models.Template`. The endpoint READS the template by `uuid` (path param) and, when Meta classifies the sample as UTILITY, MUTATES TWO fields on the Template row in the same logical write: `metadata` (rewritten to the new content via the existing strategy) and `current_version` (advanced to the new Version row written by FR-006d). The `metadata` rewrite reuses the existing `UpdateNormalTemplateStrategy` / `UpdateCustomTemplateStrategy` machinery; the in-line `current_version` advance is the deviation from `PATCH /api/v3/templates/<uuid>/`, justified by Meta's sample-API verdict serving as the Direct Send approval signal (FR-006d).
- **Version**: The status-bearing row for a Template's lifecycle. Persisted in `retail.templates.models.Version`. The endpoint creates one new `Version` per UTILITY-classified sample via the existing `TemplateBuilderMixin._create_version` path — same `template_name` generation (`weni_<name>_<timestamp>`), same FK linkages — with the SINGLE deviation that the new Version's `status` is set to `"APPROVED"` (NOT the default `"PENDING"`) and `Template.current_version` is repointed to it in the same transaction (per FR-006d). The `versions` history retains the previous Version row(s) including any previously-FLAGGED rows; only the `current_version` pointer advances. No new column / index / constraint is added to the Version table by this feature.
- **IntegratedAgent**: The Project ↔ Agent assignment. Read through `Template.integrated_agent` to (a) gate the request via `config["direct_send"]` per FR-002a and (b) feed the existing `_sync_abandoned_cart_image_config` side effect inherited from `UpdateNormalTemplateStrategy` (when the extracted shared helpers per FR-006a include that side effect, the abandoned-cart agent's image-type config is synced on header changes). The endpoint does NOT otherwise mutate IntegratedAgent.
- **Project**: The tenant boundary. Read by `Project.uuid` (from `payload["project_uuid"]`) for the WABA-id resolution and the permission check. The endpoint does NOT mutate Project.
- **ProjectOnboarding**: The onboarding-progress row. Read by `ProjectOnboarding.project_id` to resolve `config["channels"]["wpp-cloud"]["channel_data"]["waba_id"]` — the WABA id Meta needs in the sample URL. The endpoint does NOT mutate ProjectOnboarding. (Source-of-truth path documented at `retail/projects/usecases/configure_wpp_cloud.py`.)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For a UTILITY-classified sample, the local `Template.metadata` is rewritten, a new `Version` row is written with `status = "APPROVED"`, and `Template.current_version` is advanced to the new row — all three writes are persisted before the endpoint returns HTTP 200, in 99.9% of well-formed requests. The total write latency is under 2 seconds in steady state. (The dominant latency is the outbound Meta call; 2 seconds covers the steady-state Meta response time plus the local single-row writes.)
- **SC-002**: For a NOT-UTILITY-classified sample, the local `Template.metadata` and `Version` rows are byte-for-byte identical before and after the endpoint call in 100% of cases. (No silent local mutation on non-UTILITY verdicts.)
- **SC-003**: For a Meta-side failure (HTTP 5xx / timeout / missing `category` field), the local `Template.metadata` and `Version` rows are byte-for-byte identical before and after the endpoint call in 100% of cases, AND `Template.current_version` is unchanged. (No corruption from partial / failed Meta calls. The "no Celery task enqueued" property is true by construction per A10 — this endpoint never enqueues `task_create_template` regardless of Meta's verdict.)
- **SC-004**: For a UTILITY-classified sample, the next Direct Send broadcast attempt against that template renders the new content with NO additional latency budget beyond the existing dispatch path (no cache lag, no waiting on a status-webhook callback, no asynchronous Integrations round-trip) — because `Template.current_version` was repointed in-line to an `APPROVED` Version per FR-006d and the Direct Send dispatcher reads local `metadata` directly, so `Broadcast.build_direct_send_message` returns the new content on its very next call. The legacy Flows broadcast path is not in play (FR-002a refuses non-Direct-Send-eligible templates upstream), so there is no asynchronous-convergence window to worry about.
- **SC-005**: The frontend can substitute the response's `template` field directly into its template-display state with NO additional API calls. (Schema-parity with `ReadTemplateSerializer` enforced by a frontend contract test in this feature's PR.)
- **SC-006**: Endpoint latency target: p99 response time under 3 seconds in steady state. The dominant contributor is the outbound Meta call; Retail's local hot path (validation + serialization + 2 single-row DB writes + optional S3 image upload) is bounded structurally by the database round-trip and the S3 upload time when applicable. No Celery enqueue is on the hot path (A10). The 3s ceiling accounts for Meta's `message_samples` p99 latency plus Retail-side overhead. Monitored post-deploy via standard per-endpoint p99 observability; no PR-time load test is warranted.
- **SC-007**: Audit log coverage is 100% on every code path: every well-formed endpoint invocation produces a `received` line at start AND a terminal outcome line on exit (one of `template_updated`, `update_skipped`, `meta_error`, `meta_invalid_response`, `waba_not_configured`, `not_direct_send_eligible`, `project_uuid_mismatch`, `local_update_failed_after_meta_approval`), AND every event token is emitted at the FR-008b-mandated log level (verified by per-event log-level assertions in the use-case and view test suites).
- **SC-008**: Cross-tenant isolation holds: in zero cases does a sample submitted by project A target the WABA of project B. Enforced by THREE structural checks: (a) `HasProjectPermission` on the view validates the `Project-Uuid` header against Connect's authorization API; (b) the serializer-layer FR-002b check refuses any request where the `Project-Uuid` header does not equal the body's `project_uuid` (the only field the use case uses for WABA resolution); (c) FR-005a's WABA lookup filters `ProjectOnboarding` by `project__uuid == dto.project_uuid`, so the WABA resolved is always the one that belongs to the verified-and-equal tenant identifier. The three checks together are pinned by an automated test that submits with mismatched header / body identifiers AND a second test that submits a request whose body `project_uuid` is the operator's authorized project but tries to load a Template belonging to a different project (verified to fail at the Template-load step because the Template's IntegratedAgent's project does not match).

## Assumptions

- **A1**: The `MetaClient` / `MetaService` / `MetaClientInterface` / `MetaServiceInterface` quartet is the right boundary for the new outbound call. The new `submit_template_sample(waba_id, sample_body)` method extends the existing interface and follows the same conventions as `create_flow` and `register_public_key` (same `_json_headers`, same `META_API_URL`, same `CustomAPIException` envelope). No new client / service is created.
- **A2**: The WABA id MUST be resolved per-call from `ProjectOnboarding.config["channels"]["wpp-cloud"]["channel_data"]["waba_id"]`. There is no `Project.waba_id` shortcut and no IntegratedAgent-level `waba_id` column. This is the same path `configure_wpp_cloud.py` reads — see `retail/projects/usecases/configure_wpp_cloud.py:96-114`. (If operational data later shows a per-call DB round-trip is too expensive, a `Project.config["waba_id"]` denormalization could be added in a follow-up; out of scope for v1.)
- **A3**: The frontend already has the form state that `UpdateTemplateContentSerializer` accepts. The new endpoint's request body schema MUST be byte-for-byte compatible with that serializer so the frontend can call the new endpoint by changing ONLY the URL path. No new required fields, no removed required fields, no renamed fields.
- **A4**: The conditional-update path reuses the existing metadata-adaptation + S3-upload + Version-creation machinery from `UpdateNormalTemplateStrategy` / `UpdateCustomTemplateStrategy` to guarantee schema parity between this feature's writes and the existing `PATCH /api/v3/templates/<uuid>/` writes on the `metadata` / `versions` shape. The Integrations push (`task_create_template.delay`) is the ONE step from the existing strategy that this endpoint does NOT inherit — see A10 for the rationale. The behavioral deviations from the existing `PATCH` path are: (a) skip the Integrations push entirely, (b) write the new Version with `status = "APPROVED"` (not `PENDING`), and (c) advance `Template.current_version` to the new Version in-line. The reuse boundary is best implemented by extracting the metadata-build and version-create work into smaller shared helpers (FR-006a) — both endpoints invoke the same metadata-adaptation + S3-upload + version-create code path but differ in (1) the terminal `status` passed to the version-create helper, (2) whether `current_version` is repointed, and (3) whether `_notify_integrations` is called afterwards. Any future change to metadata-building or S3 upload is automatically inherited by both call sites; the Integrations push is structurally absent from this endpoint's composition.
- **A5**: Meta's `message_samples` endpoint is the AUTHORITATIVE category-check surface. Retail does NOT implement a parallel offline classifier (e.g. keyword heuristics) — Meta's response is the only source of truth for the conditional update. (Rationale: Meta's classifier is the same one used by `template_correct_category_detection` to FLAG templates later; using a different classifier here would produce inconsistent verdicts between pre-check and post-detection.)
- **A6**: Meta's `message_samples` endpoint is idempotent for the operator's purposes — submitting the same sample body twice produces the same category verdict in practice. Retries after a transient Meta-side failure are safe.
- **A7**: Variables MUST be substituted BEFORE the sample is sent to Meta. Meta classifies the sample based on the RENDERED content — leaving `{{1}}`, `{{2}}`, etc. as raw placeholders in the body, header, footer, or button text/url would force Meta's classifier to evaluate placeholder-laden skeleton text rather than the real customer-facing string, which defeats the purpose of the pre-check (a `{{1}}` body would likely be classified as low-quality boilerplate by any sensible classifier). The endpoint MUST substitute every `{{N}}` placeholder in `template_body`, `template_header` (when TEXT), `template_footer`, and each button's `text` / `url` BEFORE building the Meta sample body, using the same `substitute_template_variables` helper that `Broadcast.build_direct_send_message` uses at dispatch time (`retail.agents.domains.agent_webhook.services.direct_send_payload_builder`). The variable values come from `template_body_params` (positional list mapped to `{{1}}`, `{{2}}`, ...) — the same field the existing `UpdateTemplateContentSerializer` already accepts; no new request field is added. Substitution is OUTBOUND-ONLY — the LOCAL persisted `metadata.body` / `metadata.header.text` / `metadata.footer` / `metadata.buttons[*].{text,url}` MUST retain raw `{{N}}` placeholders, NOT the substituted strings, so the existing Direct Send broadcast renderer can re-substitute against real per-contact variable values at dispatch time (per US2's lockstep guarantee). Missing indices substitute to empty string and emit a WARNING (matching the existing dispatch-time behavior); operators are expected to provide values for every placeholder they include.
- **A8**: The endpoint IS gated on Direct-Send-eligibility via `template.integrated_agent.config["direct_send"]` (FR-002a). The eligibility predicate is the SAME one `Broadcast.build_direct_send_message` reads at dispatch time per spec 002 (`retail/agents/domains/agent_webhook/services/broadcast.py:932`) and the SAME one `AssignAgentUseCase` writes at assignment time (`retail/agents/domains/agent_integration/usecases/assign.py:162`). The check costs zero extra queries because `template.integrated_agent` is loaded on the same Template `select_related("integrated_agent")` round-trip the existing `UpdateNormalTemplateStrategy._update_common_metadata` already needs. Two reasons motivate the gating: (a) Meta's sample API has its own quota / restriction surface (error code `2388341 — Samples API Access is restricted` per `docs/direct-send-api-beta-integration.md:977`); spending that quota on templates whose dispatch path is the legacy Flows broadcast (where Meta's sample-API verdict does NOT gate dispatch) produces no operational benefit. (b) FR-006d's "skip `PENDING`, write `APPROVED` directly" rule only holds when the dispatch path actually reads from local `metadata` — which is true for the Direct Send path but NOT for the legacy Flows path (where Integrations must independently approve the new template before it can be used). Gating on `config["direct_send"]` keeps FR-006d's contract consistent with the actual dispatch surface.
- **A9**: The S3 image-upload path (for IMAGE headers carrying base64-data-URIs) is shared with the existing `UpdateNormalTemplateStrategy` — the endpoint MUST upload the image to S3 BEFORE building the Meta sample body (so Meta's sample carries an `https://...` link that Meta can fetch), and MUST persist the same S3 URL on `metadata.header.text` on a UTILITY classification. The image upload happens once per request; on a non-UTILITY classification the S3 object becomes orphaned (acceptable trade-off — `template_headers/` bucket already accumulates orphans from failed `PATCH` calls today and is operationally bounded). A future cleanup feature MAY garbage-collect orphans.
- **A10**: The endpoint does NOT fire `task_create_template.delay(...)` and does NOT introduce any new Celery task. Three reasons motivate this: (1) Direct Send dispatch reads content directly from local `Template.metadata` and the template name from `Template.current_version.template_name` — the Integrations engine is NEVER consulted on the Direct Send dispatch path (`Broadcast.build_direct_send_message`), so pushing to Integrations would create a Meta-side legacy template no dispatch path actually uses. (2) Worse, firing the push would actively BREAK Direct Send dispatch — the legacy Meta template-create flow would create a non-Direct-Send template at Meta with the same `weni_<name>_<timestamp>` name as the new local Version, and the next Direct Send dispatch with `direct_send_config.template_name = "weni_<name>_<timestamp>"` would collide with that legacy template and Meta would return error 132021 "A template with the same name already exists" (per `docs/direct-send-api-beta-integration.md:976`). (3) Skipping the push aligns this endpoint with the precedent set by `AssignAgentUseCase` (`retail/agents/domains/agent_integration/usecases/assign.py:421-438`), which creates Direct Send templates locally with `status = "APPROVED"` and does not push to Integrations either — Direct Send templates are a self-contained local-state model, and this endpoint matches that pattern. The legacy `PATCH /api/v3/templates/<uuid>/` endpoint continues to fire the push because it serves non-Direct-Send templates whose dispatch path DOES depend on the Integrations engine; the divergence is intentional and follows the gating predicate from FR-002a.
- **A11**: The audit log lives in the same standard `logger = logging.getLogger(__name__)` channel as the rest of the templates module — no new structured-events sink, no new dashboard, no new alerting hook. Operators consume the `[TemplateSampleValidation]` tag via the same Datadog / Kibana log-search infrastructure used for `[DirectSendCategoryWebhook]` (spec 003), `[DirectSend]` / `[BroadcastDispatch]` (spec 002), and the existing `[CART_SERVICE]` / `[ORDER_STATUS]` / `[CONVERSION_TRACKING]` channels.
- **A12**: The endpoint accepts a `template_uuid` path parameter (per FR-001's URL recommendation `POST /api/v3/templates/<template_uuid>/sample/`). This is the SAME identifier the existing `PATCH /api/v3/templates/<uuid>/` endpoint accepts, so the frontend's template-detail view already has it in hand.

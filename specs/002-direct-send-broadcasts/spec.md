# Feature Specification: WhatsApp Direct Send Broadcasts (OrderStatus)

**Feature Branch**: `002-direct-send-broadcasts`

**Created**: 2026-05-20

**Status**: Draft

**Input**: User description: "We are going use a new method of sending the message templates to users, using a new WhatsApp beta feature called Direct Send. (...) Only the OrderStatus agent will have this feature for now. With this feature we do not need to manually create the template in Meta, also removing the need to send the template to Integrations service. We only need to persist the template information in the retail side, and use the template name as the identifier. (...) We will also need to handle the template variable substitution (...). We have 2 new template statuses: 'PAUSED' and 'FLAGGED'. Both should disable the template broadcast. An integrated agent should have a flag internally that tells if they have direct_send enabled or not, also, the flows channel will have this flag too. During the agent assignment flow we should check the flows channel direct_send flag to know if we should set the IntegratedAgent to true."

## Clarifications

### Session 2026-05-20

- Q: Where does Retail fetch the multilingual OrderStatus template content for the Direct Send path? → A: At agent-assignment time, fetch each template from Meta's library catalog in the project's resolved language and persist it locally. Push-time single-language validation stays unchanged. No template-creation requests are issued to Meta; only library-catalog reads.
- Q: How is the project's language resolved at agent-assignment time for the Meta library-catalog fetch? → A: From the project's locale auto-detected from its VTEX tenant during the assignment flow — the same signal already used by other agents (Abandoned Cart, Payment Recovery) to choose their template language. The agent's push-time language is not used for Direct Send-enabled assignments.
- Q: What happens when Meta cannot supply a template in the project's resolved language at agent-assignment time? → A: Fall back per-template to `pt_BR`. If the `pt_BR` fetch succeeds, the local Template is persisted in `pt_BR`, the IntegratedAgent stays Direct Send-enabled, and a warning is logged. The assignment fails atomically (no IntegratedAgent or Template is persisted) only when neither the project-language fetch nor the `pt_BR` fallback returns usable content (e.g., template missing in both languages, Meta unreachable, or malformed response).

### Session 2026-05-22

- Q: What is the canonical shape of `header` in the Meta library catalog response, and how should the Direct Send fetch-time adapter normalize and validate it? → A: Meta's library catalog ALWAYS returns `header` as a plain text string when present (never a dict). The fetch-time adapter MUST accept `header` only when absent or a plain text string, length-validate it against `MAX_HEADER_TEXT_LENGTH`, normalize it to `Template.metadata.header = {header_type: "TEXT", text: <string>}`, and treat any non-string, non-null `header` as a malformed-response failure per FR-003c → FR-003d (pt_BR retry, then atomic rollback). Image/media headers on Direct Send-path Templates arise EXCLUSIVELY from post-assignment edits to `Template.metadata.header.header_type` (e.g., an operator sets `header_type="IMAGE"` via the template-update endpoint to attach media); the dispatch path MUST continue to honor attachment-type headers via the existing `direct_send_payload_builder.build_direct_send_header` branch. The fetch path itself does NOT handle attachment headers. (FR-003e.)
- Q: For buttons in the Meta library catalog response, how should fetch-time validation handle types beyond `{URL, QUICK_REPLY}` (e.g., `PHONE_NUMBER`, `PAYMENT_REQUEST`, `ORDER_DETAILS`)? → A: Strict atomic-rollback at fetch time. Any button whose `type` is outside `{URL, QUICK_REPLY}` — including `PHONE_NUMBER`, `PAYMENT_REQUEST`, `ORDER_DETAILS`, `COPY_CODE`, `FLOW`, or any future Meta-curated value — MUST trigger a malformed-response failure that follows FR-003c → FR-003d (pt_BR retry, then atomic rollback). URL-button entries MAY arrive with either a flat `url` string (the dominant shape in real library-catalog responses) or the legacy nested `{base_url, url_suffix_example}` shape; both are normalized to a flat `url` string at persist time via the same `_ensure_protocol` + `_append_placeholder_if_needed` heuristic the push-path `ButtonTransformer` already applies, so `metadata.buttons` stores a single canonical shape regardless of upstream variance. The fetch-time count limits (≤1 `URL`, ≤3 `QUICK_REPLY`) and the label length limit (`MAX_BUTTON_LABEL_LENGTH` = 20 chars) remain enforced. The fetch-time policy mirrors Meta's Direct Send Beta v1 dispatchable set and pins the assumption "single CTA URL button, up to three reply buttons" to the validator. (FR-003f.)
- Q: Should the adapter persist Meta's auxiliary curation fields (`body_param_types`, `attributes`, `topic`, `usecase`, `industry`, `id`) on `Template.metadata` for future use, or drop them at fetch time? → A: Drop them at fetch time. The adapter persists only the dispatch-relevant subset (`{header, body, body_params, footer, buttons, category, language}`) plus the existing `direct_send` audit sub-object (`fetched_from_meta_library`, `fetched_at`, `requested_language`, `actual_language` per `data-model.md §3`). No additional fields are stored. Anti-YAGNI: future features that need AMOUNT/DATE-aware variable rendering or Meta-template-ID correlation will be free to extend the metadata shape additively under `metadata.direct_send` rather than ingesting unused fields preemptively.
- Q: When a Direct Send-enabled OrderStatus template carries a `URL` button (e.g. the `order_canceled_3` library template with `buttons: [{type: "URL", text: "Ver detalhes do pedido", url: "https://www.example.com"}]`), what is the canonical wire shape Retail MUST emit to Flows for that CTA URL? → A: Move CTA URL out of `msg.buttons` into a top-level discriminator + sibling sub-object on `msg` — `msg.interaction_type = "cta_url"` + `msg.cta_message = {"display_text": "<substituted>", "url": "<substituted>"}`. Drop the URL entry from `msg.buttons` entirely on the Direct Send path. The discriminator key spelling is `interaction_type` (matching the existing `msg.interaction_type = "order_details"` shape emitted by `Broadcast._apply_order_details` at `retail/agents/domains/agent_webhook/services/broadcast.py:216`), NOT `interactive_type`. The currently-implemented `msg.buttons[0] = {sub_type: "cta_url", display_text, url}` shape (`direct_send_payload_builder.build_direct_send_buttons` lines 156-171, `contracts/messaging-gateway-payload.md` §3.3 and §5.1) is a documentation/implementation error and MUST be corrected during US1 implementation. The new shape is pinned by FR-014a; the spec, not the contract artifact, is the canonical source. (FR-014a.)
- Q: How should Retail handle Meta library templates whose upstream button label exceeds `MAX_BUTTON_LABEL_LENGTH` in a specific language (e.g., `order_canceled_3` in `pt_BR`/`es`), which today triggers `DirectSendUnsupportedComponentError(component_type="buttons")` and routes through FR-003c → FR-003d, blocking every assignment in the affected locale? → A: Introduce a data-driven per-`(template_name, language)` button-label override map (`DIRECT_SEND_BUTTON_LABEL_OVERRIDES`) as a constant alongside `direct_send_constants.py`, consulted inside `adapt_meta_library_template_response` at fetch-time. The map is the single source of truth for these overrides and is trivially extended if Meta ships another long-label template; the adapter stays the only consumer (no per-call branch on `template_name`). (Pinned by FR-003g.)
- Q: Should the override map be applied unconditionally (whenever `(template_name, language)` is in the map) or conditionally (only when Meta's upstream label exceeds `MAX_BUTTON_LABEL_LENGTH`)? → A: Conditionally on overflow. The override is consulted ONLY when the upstream `text` would otherwise fail the existing `MAX_BUTTON_LABEL_LENGTH` length check in `_validate_and_normalize_buttons`; if Meta's upstream label already fits within the limit (≤20 chars), the upstream value is honored verbatim and the map is ignored. This preserves Meta as the source of truth whenever its content is dispatch-eligible and confines the override surface to the narrow "would otherwise atomic-rollback" cases. The map's `(template_name, language)` keys remain the authoritative substitute when overflow occurs. (Pinned by FR-003g.)
- Q: What components can the override map adjust — button label only, or any component (header/footer/body)? → A: Button label `text` only. Map values are plain strings (`{(template_name, language): "<localized button label>"}`), consumed exclusively inside `_validate_and_normalize_buttons`. Body / header / footer overflows are NOT addressable via this mechanism in v1 (`MAX_BODY_LENGTH=1024`, `MAX_HEADER_TEXT_LENGTH=60`, and `MAX_FOOTER_LENGTH=60` are generous and not currently reported as failure points). Future component overrides can extend additively by promoting the map value to a typed dict — the `(template_name, language)` key shape is already future-friendly. (Pinned by FR-003g.)
- Q: Within a template that may carry both URL and QUICK_REPLY buttons, which button gets the override applied? → A: The single URL button, identified by `type == "URL"`. The override fires only on the unique URL entry (FR-003f caps URL buttons at ≤1 per template), making positional identification unnecessary and avoiding brittleness against future field reorderings. QUICK_REPLY labels are NOT in scope for the override mechanism in v1; Meta's UX guidelines for QUICK_REPLY (short titles such as "Yes" / "No" / "Cancel") make overflow unlikely, and no QUICK_REPLY-overflow case has been reported. If a future template introduces a QUICK_REPLY overflow, the map value's shape can extend additively (per Q3's "additively typed dict" extension path). (Pinned by FR-003g.)
- Q: What observability does the override emit when it fires at fetch-time? → A: INFO-log only. The override site emits `logger.info` identifying `template_name`, `language`, the upstream (overflow) text, and the substituted override text; no audit flag is persisted on `Template.metadata.direct_send`. The log line is the only operator signal, queryable via log search; per-Template post-hoc inspection of "was an override applied?" is intentionally NOT supported in v1 (operators reconstruct it from logs if needed). This keeps the additive `Template.metadata.direct_send` audit sub-object minimal (continuing to carry only `fetched_from_meta_library`, `fetched_at`, `requested_language`, `actual_language` per `data-model.md §3`) and avoids a follow-up DB migration. (Pinned by FR-003g.)
- Q: When a Direct Send-enabled OrderStatus template carries `QUICK_REPLY` buttons, what is the canonical wire shape Retail MUST emit to Flows for those buttons? → A: Emit them as a top-level flat array of substituted title strings on `msg` — `msg.quick_replies = ["<title 1>", "<title 2>", ...]` (e.g. `"quick_replies": ["Yes", "No"]`) — NOT as objects inside `msg.buttons`. The Meta library catalog's `id` field is intentionally NOT carried on the wire; Flows derives any downstream payload identifier from the title (or from its own per-channel handling against Meta's Direct Send Beta endpoint). The fetch-time count limit of FR-003f (≤3 `QUICK_REPLY`) caps `msg.quick_replies` at 3 elements; the per-entry length limit `MAX_BUTTON_LABEL_LENGTH` (20 chars) is enforced post-substitution. The currently-implemented `msg.buttons = [{"sub_type": "reply", "id", "title"}]` shape (`direct_send_payload_builder.build_direct_send_buttons` lines 172-184, `contracts/messaging-gateway-payload.md` §3.3 and §5.1) is a documentation/implementation error and MUST be corrected during US1 implementation. Combined with FR-014a, `msg.buttons` is now LEGACY-ONLY on the wire — the Direct Send path NEVER emits a `msg.buttons` key. The new shape is pinned by FR-014b; the spec, not the contract artifact, is the canonical source. (FR-014b.)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Dispatch order-status broadcast through Direct Send (Priority: P1)

When the OrderStatus agent is configured with Direct Send and a customer transitions through an order state (e.g. invoiced, shipped, delivered, canceled), the system must reach that customer through WhatsApp **without** depending on a previously approved Meta template. Retail looks up the template locally, substitutes variables and components on its side, and dispatches the final message to the messaging gateway with a `direct_send` flag and the template name as the identifier.

**Why this priority**: This is the entire reason for the feature. Without it, none of the cost or speed-to-market gains of Direct Send are realised. It also unblocks businesses that currently wait days for Meta to approve their utility templates.

**Independent Test**: With a fixture OrderStatus IntegratedAgent that has Direct Send enabled and at least one locally-persisted template whose status is APPROVED, fire a VTEX order-status webhook for a contact. Verify that the messaging gateway receives a payload with the final substituted message body, the message components (header, footer, buttons, attachment), the Direct Send flag set, and the template name used as the Direct Send identifier. Verify the broadcast is recorded in the dispatch log.

**Acceptance Scenarios**:

1. **Given** an OrderStatus IntegratedAgent with Direct Send enabled and a Template whose current version is APPROVED with a body containing `{{1}}` and `{{2}}`, **When** the order-status webhook fires and the rule engine returns `{"1": "Maria", "2": "12345"}`, **Then** the messaging gateway receives a message whose body text already contains `Maria` and `12345` substituted in place, with the Direct Send flag set and the template name passed as the Direct Send identifier.
2. **Given** an OrderStatus IntegratedAgent with Direct Send enabled and a Template that includes an image header and a CTA URL button, **When** the order-status webhook fires and the rule engine returns the variable values plus the URL parameter, **Then** the messaging gateway receives the substituted body, the image attachment, and the CTA URL button — all built by Retail, with no template lookup performed by the messaging provider.
3. **Given** an OrderStatus IntegratedAgent with Direct Send enabled and a Template that includes only a body with no variables, **When** the order-status webhook fires, **Then** the messaging gateway receives the literal body text and no variable substitution is performed.

---

### User Story 2 - Onboard an OrderStatus agent without creating templates in Meta (Priority: P2)

When an operator assigns the OrderStatus agent to a project whose WhatsApp channel has Direct Send enabled, the system must skip every step that submits a template to Meta or to the Integrations service, and instead persist the templates locally with all their content (body, header, footer, buttons, language). The IntegratedAgent must reflect that Direct Send is in use so subsequent broadcasts pick the new dispatch path.

**Why this priority**: Direct Send removes the entire template-approval lifecycle. Failing to skip the legacy template-creation pipeline at assignment time would either submit duplicate/conflicting templates to Meta or block onboarding while waiting for an approval that is no longer required. This is what unlocks immediate availability for new customers.

**Independent Test**: Configure a project whose WhatsApp channel has Direct Send enabled. Assign the OrderStatus agent to that project. Verify that no calls are made to the template-creation pipeline (Meta or Integrations), that the resulting IntegratedAgent has Direct Send enabled, and that all the templates the agent ships with are persisted locally with their full content (body, header, footer, buttons, language) and a status of APPROVED — ready to be used for broadcasts immediately.

**Acceptance Scenarios**:

1. **Given** a project whose WhatsApp channel reports Direct Send enabled and whose VTEX-detected locale is fully covered by Meta's library catalog for every OrderStatus template, **When** the operator assigns the OrderStatus agent to it, **Then** no template-creation request is sent to Meta or to the Integrations service, every template is fetched from Meta's library catalog in the project's locale and persisted locally with its full content and status APPROVED, the IntegratedAgent has Direct Send enabled, and no fallback warning is logged.
2. **Given** a project whose WhatsApp channel reports Direct Send disabled, **When** the operator assigns the OrderStatus agent to it, **Then** the existing template-creation pipeline runs unchanged, and the IntegratedAgent has Direct Send disabled.
3. **Given** the channel's Direct Send flag cannot be retrieved (channel lookup fails), **When** the operator assigns the OrderStatus agent, **Then** the system defaults to Direct Send disabled (existing behavior is preserved) and a warning is logged.
4. **Given** a Direct Send-enabled channel and a project locale for which Meta's library catalog is missing one or more of the OrderStatus templates but has them in `pt_BR`, **When** the operator assigns the OrderStatus agent, **Then** the templates available in the project's locale are persisted in that locale, the missing-locale templates fall back to `pt_BR` and are persisted with the `pt_BR` content, a warning is logged for each fallback identifying the template and the originally-requested locale, and the IntegratedAgent has Direct Send enabled.
5. **Given** a Direct Send-enabled channel and a template that Meta cannot supply in either the project's locale or `pt_BR` (template missing in both, or Meta unreachable for both fetches), **When** the operator assigns the OrderStatus agent, **Then** the assignment fails atomically with an error identifying the failed template and reason, no `IntegratedAgent`, `Template`, or version is persisted, and the operator can retry once Meta has the content (or recovers).

---

### User Story 3 - Block broadcasts for paused or flagged templates (Priority: P3)

When a template's current version is in the new PAUSED or FLAGGED state, the system must not dispatch any broadcast that would use it. The broadcast attempt is skipped silently (with an audit log entry) so that paused or flagged content cannot reach end customers. When the template version returns to APPROVED, broadcasts resume on subsequent webhooks.

**Why this priority**: Meta pauses templates that show low quality and flags templates whose content does not meet utility guidelines; sending despite a paused/flagged status results in failed deliveries, customer-facing errors, and accelerates the path toward Direct Send access being revoked for the WABA. Blocking on Retail's side is a hard safety requirement before this feature can be released.

**Independent Test**: With a Direct Send-enabled OrderStatus agent and a Template whose current version is PAUSED, fire the order-status webhook for that template. Verify no payload is sent to the messaging gateway, no broadcast row is created, and an audit log entry records the skip with the reason. Repeat for status FLAGGED. Then transition the version back to APPROVED and verify broadcasts resume on the next webhook.

**Acceptance Scenarios**:

1. **Given** a Direct Send-enabled OrderStatus agent and a Template whose current version is PAUSED, **When** the order-status webhook fires for that template, **Then** no message is sent to the messaging gateway, no broadcast record is persisted, and the skip is recorded in the audit log with the template name and the PAUSED reason.
2. **Given** a Direct Send-enabled OrderStatus agent and a Template whose current version is FLAGGED, **When** the order-status webhook fires for that template, **Then** the broadcast is skipped under the same rules as PAUSED.
3. **Given** a previously-paused Template whose version returns to APPROVED, **When** the next order-status webhook fires for that template, **Then** the broadcast is dispatched normally through the Direct Send path.

---

### User Story 4 - Existing OrderStatus agents continue to broadcast unchanged (Priority: P4)

Customers whose WhatsApp channel does not have Direct Send enabled (the majority during the Beta) must keep receiving order-status messages exactly as they do today: through the legacy template-name-and-variables payload, with Meta performing the substitution. No regression in delivery rate, payload shape, or status tracking is acceptable.

**Why this priority**: Direct Send is a Meta Beta and is rolled out gradually per WABA. Most active customers will not have it enabled at release time. A regression in their flow would be a production incident; this user story is the safety net that protects them.

**Independent Test**: With an OrderStatus IntegratedAgent whose Direct Send is disabled (or a project whose channel does not report the flag), fire the same order-status webhook used in Story 1. Verify the messaging gateway receives the exact payload shape used today (template name + locale + positional variables, with components passed via the legacy fields), and that the Direct Send flag is not set in the payload.

**Acceptance Scenarios**:

1. **Given** an OrderStatus IntegratedAgent with Direct Send disabled and an APPROVED Template, **When** the order-status webhook fires, **Then** the messaging gateway receives the legacy payload (template name + locale + positional variables) and the Direct Send flag is absent from the request.
2. **Given** an OrderStatus IntegratedAgent with Direct Send disabled and a Template version in any pre-existing non-APPROVED state (PENDING, REJECTED, etc.), **When** the order-status webhook fires, **Then** the broadcast is skipped following the existing behavior, with no change introduced by this feature.

### Edge Cases

- **OrderStatus template missing in the project's locale at agent-assignment time**: Meta's library catalog returns no entry for one of the OrderStatus templates in the project's resolved language but has it in `pt_BR`. The assignment falls back to `pt_BR` for that template, persists the `pt_BR` content, keeps the IntegratedAgent Direct Send-enabled, and emits a warning so operators know that template will deliver in `pt_BR`. (FR-003c.)
- **Meta library catalog cannot supply a template in either the project's locale or `pt_BR`**: The agent assignment fails atomically — no IntegratedAgent or Template is persisted — and the operator receives an error identifying the failed template and reason. The operator can retry once Meta has the content or recovers. (FR-003d.)
- **Meta library template whose upstream URL-button label exceeds `MAX_BUTTON_LABEL_LENGTH` in one of the assignment-time languages** (e.g., `order_canceled_3` whose `pt_BR` upstream label is "Ver detalhes do pedido" = 22 chars and `es` upstream label is "Ver detalles del pedido" = 23 chars, both above the 20-char ceiling): the fetch-time adapter consults the per-`(template_name, language)` override map (FR-003g) and substitutes the URL button's `text` with the localized override before re-running the length check (`pt_BR` → "Detalhes do pedido" = 18 chars; `es` → "Detalles del pedido" = 19 chars). The Template persists with the override label; an INFO log records the substitution. If the same template's upstream label is later returned in a language NOT covered by the map AND still exceeds the limit, the existing FR-003f.c failure path applies (raise → FR-003c → FR-003d). (FR-003g.)
- **Direct Send-enabled agent with no local template for the rule**: The rule engine names a template that has not been persisted on Retail's side. The broadcast is skipped (same as the current "no APPROVED template found" path) and an audit log entry records the miss.
- **Variable values missing from the rule engine output**: The body contains `{{1}}` but the rule engine returned no value for index `1`. The system substitutes the placeholder with an empty string and logs a warning identifying the template and the missing index. The broadcast still proceeds.
- **Variable values for indices that do not exist in the template**: The rule engine returns extra indices that do not appear in the template body or components. Extra values are ignored silently.
- **Template name does not satisfy Meta's Direct Send naming rules**: The Direct Send identifier accepts only lowercase alphanumerics and underscores, max 512 characters. A template whose stored name violates the rule must not be dispatched through Direct Send; the broadcast is skipped silently with an audit log entry that includes the template name, the failing rule, and the originating order-status event. No normalization is attempted on the stored name (resolved at plan time — see research Decision 7; rationale: normalizing on the way out would silently desync Retail's local Template name from Meta's auto-created Direct Send identifier).
- **Channel Direct Send flag changes after agent assignment**: The channel is later flipped from Direct Send enabled to disabled (or vice-versa) outside this feature's flow. The IntegratedAgent's flag does not auto-resync; the snapshot taken at assignment time is the source of truth until the agent is re-assigned. (Out of scope for this feature; documented for awareness.)
- **Template current version is APPROVED but contains no body text**: The Direct Send beta requires a body component. A template with an empty body must not be dispatched; the broadcast is skipped with a clear audit entry. (Pre-existing constraint, restated here.)
- **Template content uses unsupported components for Direct Send**: Components outside the Direct Send-supported set (e.g. carousel, list message, catalog message, order_details, flow message) are out of scope for this feature. Templates that need such components must keep using the legacy path; the OrderStatus agent's templates do not require them.
- **Concurrent broadcasts targeting the same paused template**: All concurrent attempts within the same window must be skipped. The status read on Retail's side is authoritative; the system does not need to coordinate across workers.
- **Meta starts emitting `PAUSED` or `FLAGGED` to the existing template-status webhook before the future routing feature is delivered**: The existing handler does not know how to map the new Meta values; today it logs and drops unknown statuses, so the deploy is safe. When the future feature lands, the handler will start writing `PAUSED` / `FLAGGED` to `Version.status` and the dispatch gates introduced by FR-007 / FR-012 will start firing automatically. No coordination is required at this feature's release. (FR-023.)
- **Internal tooling sets `Version.status=PAUSED` or `FLAGGED` directly via the `update_template` endpoint**: The endpoint accepts the new values and persists them as-is on the named `Version`; the `current_version` promotion logic runs only on `APPROVED` and is unchanged. Subsequent broadcasts against that template are blocked by FR-012. (FR-026.)
- **Two order-status webhooks for DIFFERENT `current_state` for the same `order_id` arrive 50ms apart** (e.g. `invoiced` then `shipped`): both dispatch as SEPARATE logical broadcasts because the dedup tuple includes `current_state` (FR-030). The fact that two `BroadcastMessage` rows are persisted for the same `order_id` is expected; the `BroadcastConversion`'s `(project, order_id)` unique constraint (FR-033) still collapses any downstream conversion to one row via the last-touch attribution rule on the payment-recovery agent.
- **Two concurrent order-status webhooks for the SAME `(project, agent, order_id, current_state)` are processed by two different workers simultaneously**: `cache.add` is atomic across workers (Redis SETNX semantics); exactly one worker observes "event_registered_now == True" and proceeds, the other observes the existing key and skips with the `duplicate_skipped` audit log entry (FR-028). No coordination beyond the atomic `cache.add` is required.
- **The official OrderStatus agent AND a custom agent with `parent_agent_uuid` BOTH match the same project**: the official agent takes precedence per FR-031; the custom agent is silently ignored on that event. The `_lookup_order_status_agent` fallback chain is the canonical resolution.
- **The Lambda returns the SAME `template` name for two distinct `current_state` values** (e.g. operator misconfigures the rule engine and both `invoiced` and `shipped` resolve to `weni_order_invoiced`): each event dispatches because they are different logical broadcasts (FR-030); the customer receives two messages built from the same template with whatever variable values the Lambda produced for each event. This is an operator-misconfiguration class — the spec does not introduce a runtime guard against it because the dedup tuple intentionally separates the two events.
- **The Flows delivery-callback (RabbitMQ `template-status`) replays a `delivered` event after the consumer crashed mid-update**: the consumer is retry-safe per FR-035; the re-delivery looks up the row by `external_message_id`, observes the row is already in `DELIVERED`, and is rejected by `_is_out_of_order` (terminal/already-delivered → no further increment of `broadcasts_delivered`). The row's final state matches a single delivery.
- **PAUSED → APPROVED transition lands DURING the dedup window of an in-flight broadcast for the same template**: the version status is read AFTER `cache.add` returns "event_registered_now == True" — i.e. the dedup gate has already accepted the event — so the dispatch follows the version's status at READ time. Operators expecting an "unpause race" should be aware: an event that arrived during the dedup window of an earlier PAUSED-skip will NOT auto-replay; the next webhook is the trigger. The dedup window is intentionally narrow (default 60s) to keep this window short.
- **Direct Send Beta v3 error code 132021 (template-name collision delivered asynchronously via Meta webhook)**: Meta returns this code when a Direct Send dispatch references a template-name that collides with an existing previously-auto-generated template in the same WABA. This error reaches Retail asynchronously via Flows' status callback and surfaces as a `BroadcastMessage.status=FAILED` row with `error_message` carrying the upstream code. No Retail-side recovery is attempted (retry budget = 0); the operator is expected to rename the local Template before re-assigning. The OrderStatus templates today use snake_case names with WABA-unique suffixes (see Assumptions) so the cohort that would hit this error is empty at v1.
- **Flows returns 5xx and Retail records FAILED, but the message actually reached Meta and was delivered to the customer**: this is a Flows-to-Retail observability gap, NOT a Retail-side guarantee. `_record_failed_dispatch` persists `BroadcastMessage.status=FAILED` because the only signal Retail has is the 5xx; if a subsequent Flows `template-status delivered` event arrives for the same `broadcast_id`, the lifecycle-rank guard (FR-035) accepts the forward transition and the row converges to `DELIVERED`. Operators investigating "failed but delivered" should rely on this convergence rather than expect a Retail-side compensation flow. End-to-end exactly-once across Flows / Meta / device is explicitly NOT a Retail guarantee (see "Boundary" under the Exactly-Once Dispatch invariant).
- **Dedup-cache backend (Redis) unreachable mid-dispatch**: `cache.add` is delegated to Django's cache framework. If the backend is unreachable, Django's `redis` backend raises and the order-status entry point logs and re-raises — the trigger fails fast and no `BroadcastMessage` row is persisted. The fail-fast (fail-closed) behavior is the v1 default; tuning to fail-open ("allow dispatch on cache outage at the cost of potential duplicates") would be a separate spec change. The dedup-cache backend is documented in `plan.md` Constraints — Idempotency & retry safety.
- **`MarkBroadcastConvertedUseCase` is invoked twice for the same `(project, order_id)`** (broker re-delivery OR VTEX webhook duplicate after the first invoiced lookup completed): the second invocation observes the existing `BroadcastConversion` row via `get_or_create` and is logged as `conversion_already_recorded` with no side effects. (FR-036.)
- **Re-assignment of an OrderStatus agent after a previous assignment was marked `is_active=False`**: re-assignment is treated as a NEW assignment attempt — every template is re-fetched from Meta's library catalog at the project's current resolved language (FR-003a–FR-003d). Previously persisted Templates from the prior assignment are NOT re-used; this preserves the spec's "snapshot at assignment time" guarantee even when the channel-side Direct Send flag flipped between the two assignments. The orphaned legacy Templates remain in the database (harmless) and can be cleaned up by the existing template-deletion flows.
- **Two operators retry the same FR-003d failure concurrently against the same project**: each retry is its own `@transaction.atomic` assignment attempt. The slower attempt either (a) fails on the same template (e.g. Meta still unreachable) and rolls back without affecting the faster attempt, or (b) succeeds and races against the faster one on the database — Django's transaction isolation plus the `IntegratedAgent`'s existing `(project, agent)` uniqueness rule serializes the writes; the loser sees an `IntegrityError` and the operator retries one more time. The spec does not introduce a distributed lock for this race because the unique-row constraint is already the correct serialization point.
- **Operator with permission on project A passes project B's `app_uuid` (or `channel_uuid`) in the assignment request**: the cross-validation at the assignment surface (FR-043) catches the mismatch — the `IntegrationsService.get_channel_app(...)` response's `project_uuid` is compared to the `Project-Uuid` header, and on mismatch the assignment is DENIED with HTTP 403 and an audit log entry that includes both project UUIDs and the operator identifier. The persisted `IntegratedAgent.channel_uuid` is therefore guaranteed to belong to the same project that owns the IntegratedAgent. For v1 the validation is satisfied transitively via the trust boundary on Integrations Engine (HasProjectPermission + Integrations' own authorization on the channel-app endpoint); the explicit Retail-side cross-validation is captured as a defense-in-depth follow-up in `plan.md` Constraints — Tenant isolation.
- **Flows status callback misroutes a `delivered` event to the wrong tenant's `BroadcastMessage`**: structurally impossible at v1 because the consumer's project-less lookup keys on `broadcast_id` (FR-041) and Flows guarantees `broadcast_id` global uniqueness across all Weni tenants (see Assumptions). If Flows ever issues the same `broadcast_id` for two tenants the `broadcasts_broadcast_id_unique` constraint (FR-032) raises `IntegrityError` at insert time — a hard, observable failure rather than a silent cross-tenant leak. A `delivered` event whose payload references project B but whose `broadcast_id` matches a row in project A is processed against the matched row's project; the payload-vs-row project mismatch is logged at WARNING per FR-041.
- **Two projects share a VTEX account due to data error → order-status webhook routes to the wrong project**: the SECURITY BOUNDARY at the order-status entry point is a fail-closed `Project.MultipleObjectsReturned → return None` (`get_project_by_vtex_account` at `retail/agents/domains/agent_webhook/usecases/order_status.py:174`), which causes the webhook to drop the event with a logged ERROR (`[ORDER_STATUS] multiple_projects: vtex_account={...}`). Operators are expected to reconcile the duplicate `vtex_account` rows before traffic resumes; a "best-guess" tenant resolution is intentionally NOT implemented because the cost of a missed dispatch is observable and recoverable while a misrouted dispatch is user-facing and irrecoverable.
- **Lambda for project A returns a `template` name that happens to match project B's local Template name**: structurally safe because the per-IntegratedAgent template lookup `integrated_agent.templates.filter(name=...)` is the canonical resolution at `Broadcast.get_current_template`. The lookup is FK-chained to the project; a name collision across projects is therefore NOT a tenant boundary — each `IntegratedAgent` resolves the name against its own `templates` reverse-relation only. The Lambda function-name namespace itself is per-project (`Agent.project` FK + the `retail-setup-{hash_13_digits}` derivation, see Assumptions), so a Lambda invoked from project A cannot be the Lambda owned by project B.
- **Re-assignment after `is_active=False` re-attaches to a previous Template owned by a different project**: structurally impossible because re-assignment creates a new `IntegratedAgent` row (the previous one stays at `is_active=False` and is not reused — see Edge Case "Re-assignment of an OrderStatus agent..." earlier in this section). Templates from the previous assignment remain FK-linked to their original `IntegratedAgent` row and therefore to its original project, so they cannot be silently picked up by a re-assignment in a different project.
- **EDA event arrives whose payload references project B but whose lookup key (`broadcast_id` or `external_message_id`) matches a `BroadcastMessage` in project A**: the matched row's `project_id` is the canonical tenant for the subsequent write — every status transition, counter increment, and `BroadcastConversion` write uses `message.project_id`, never a project identifier from the event payload. The disagreement is logged at WARNING per FR-041 so operators can detect the upstream cross-tenant misroute. This guarantees that even a Flows-side or Meta-side cross-tenant identifier misroute (which Assumption: `broadcast_id` global uniqueness already forbids) cannot cause a Retail-side row in project A to be written with project B's `BroadcastConversion`.
- **Single Meta system token rate-limit affects ALL projects' assignments simultaneously**: the `META_SYSTEM_USER_ACCESS_TOKEN` is a CROSS-TENANT credential by design (see Multi-credential surface taxonomy in Tenant isolation). A Meta-side rate-limit applied to that token reduces every project's library-catalog GET success rate at the same time, and operator retries (per FR-003d) cannot independently isolate one tenant from another at the credential level. This is an accepted trade-off for v1: the alternative (per-tenant Meta access tokens) requires Beta-program-level cooperation from Meta and is well outside the scope of this feature. The blast radius is bounded because library-catalog reads are agent-assignment-time-only — there is no dispatch-time dependency on this token, so a rate-limit blocks new assignments but not existing tenants' broadcasts. Documented as `[Conflict]` resolution in `research.md` Decision 16 — Tenant isolation model.

## Requirements *(mandatory)*

### Functional Requirements

#### Agent assignment

- **FR-001**: The system MUST expose a per-agent Direct Send flag on the IntegratedAgent record (default: disabled) that controls which dispatch path applies at broadcast time. The flag MUST be stored as an optional key inside the existing `IntegratedAgent.config` JSON (`config["direct_send"]: bool`), NOT as a new column on `IntegratedAgent`. Absence of the key MUST be interpreted as `False`. **Rationale (JSON-key storage choice — see `data-model.md §1` Decision)**: storing the flag inside the existing `config` JSON yields zero schema change on `IntegratedAgent`, collapses the rollout footprint to a single migration scope (the `Version.STATUS_CHOICES` extension only), and makes US4's "byte-identical legacy-path" guarantee trivially provable — the IntegratedAgent column list is byte-identical to the pre-feature shape, so no persisted-row diff against a pre-feature baseline can drift on the legacy cohort regardless of which serializer or audit query inspects it. The choice is recorded as a Decision in `data-model.md §1`.
- **FR-002**: During OrderStatus agent assignment, the system MUST consult the WhatsApp channel's Direct Send flag (sourced from the Integrations service) and set the IntegratedAgent's Direct Send flag to the same value. For any other agent (subject to FR-019, which restricts Direct Send eligibility to OrderStatus in v1), the IntegratedAgent's Direct Send flag MUST remain `False` regardless of the channel's flag.
- **FR-003**: When the channel reports Direct Send enabled, the system MUST persist all templates required by the OrderStatus agent locally — with body text, header (text or image), footer, buttons, and language — without sending any template-creation request to Meta or to the Integrations service.
- **FR-003a**: When the channel reports Direct Send enabled, the system MUST fetch each OrderStatus template's content (body, header, footer, buttons, language) from Meta's library catalog at agent-assignment time, using the **project's resolved language** rather than the agent's push-time language. The fetched content is persisted on the local Template, and subsequent broadcasts MUST NOT depend on Meta at dispatch time. Meta library-catalog reads are GET requests for existing library templates; they are not template-creation operations and are explicitly permitted by FR-003.
- **FR-003b**: The "project's resolved language" referenced in FR-003a MUST be the project's locale auto-detected from its VTEX tenant during the assignment flow — the same signal already used by other agents (Abandoned Cart, Payment Recovery) to choose their template language. The agent's push-time language MUST NOT be used as the language for Direct Send-enabled assignments.
- **FR-003c**: When the Meta library-catalog fetch (FR-003a) returns no usable content for a template in the project's resolved language, the system MUST retry the fetch in `pt_BR`. "No usable content" covers three distinct failure modes and the system MUST treat them identically: (a) HTTP 200 with empty `data` or no exact-name match (translation truly missing in that locale); (b) HTTP 5xx / 4xx / network error (Meta unreachable or rate-limited); (c) HTTP 200 with malformed JSON or a schema the local adapter rejects. On a successful `pt_BR` retry the template is persisted locally with the `pt_BR` content, the IntegratedAgent's Direct Send flag remains enabled, and a warning MUST be logged identifying the template and the originally-requested locale so operators can act on the language gap.
- **FR-003d**: If both the project-language fetch (FR-003a) and the `pt_BR` fallback fetch (FR-003c) fail to return usable content for any required template — whether due to a missing translation, Meta unreachability (4xx/5xx), or a malformed response — the agent assignment MUST fail atomically. The three failure modes are treated as the same atomic-rollback class because the operator-facing outcome is identical: no `IntegratedAgent`, `Template`, or version is persisted, and the operator receives an error identifying the failed template(s) and the reason. The operator MAY retry the assignment; each retry is a NEW assignment attempt that re-fetches every template from scratch (no partial-batch results are cached between attempts — see Plan §Constraints — Idempotency).
- **FR-003e**: The Meta library catalog (FR-003a) ALWAYS returns `header` as a plain text string when present — never as a dict. The Direct Send fetch-time adapter MUST: (a) accept `header` only when absent or a plain text string; (b) length-validate the string against `MAX_HEADER_TEXT_LENGTH` (60 chars); (c) normalize the persisted form to `Template.metadata.header = {header_type: "TEXT", text: <string>}`; (d) treat any non-string, non-null `header` as a malformed-response failure and follow FR-003c → FR-003d (pt_BR retry, then atomic rollback). Image/media headers on Direct Send-path Templates arise EXCLUSIVELY from post-assignment edits to `Template.metadata.header.header_type` (e.g., an operator sets `header_type="IMAGE"` via the template-update endpoint to attach media); the dispatch path MUST continue to honor those attachment-type headers via the existing `direct_send_payload_builder.build_direct_send_header` branch. The fetch path itself does NOT handle attachment headers; FR-021's "additive `direct_send` sub-object" rule applies only to the fetch-time write, and post-assignment edits to `header.header_type` are governed by the existing `update_template` endpoint contract (FR-026). The `contracts/meta-library-catalog.md §2 / §5` examples that show `header` as a dict `{type, text}` are documentation errors and MUST be corrected to reflect the plain-string shape during implementation; the spec, not the contract artifact, is the canonical source for the header-shape requirement.
- **FR-003f**: For buttons in the Meta library catalog response, the Direct Send fetch-time adapter MUST: (a) reject any entry whose `type` is outside `{URL, QUICK_REPLY}` — including `PHONE_NUMBER`, `PAYMENT_REQUEST`, `ORDER_DETAILS`, `COPY_CODE`, `FLOW`, or any future Meta-curated value — by treating the response as malformed and following FR-003c → FR-003d (pt_BR retry, then atomic rollback); (b) accept URL-button entries with either a flat `url` string (the dominant shape in real library-catalog responses) or the legacy nested `{base_url, url_suffix_example}` shape (the push-path shape handled by `ButtonTransformer`), normalizing both to a flat `url` string at persist time via the same `_ensure_protocol` + `_append_placeholder_if_needed` heuristic so `metadata.buttons` stores a single canonical shape regardless of upstream variance; (c) length-validate `text` against `MAX_BUTTON_LABEL_LENGTH` (20 chars), subject to the per-`(template_name, language)` override of FR-003g; (d) enforce the count limits ≤1 `URL` button and ≤3 `QUICK_REPLY` buttons at fetch time. The fetch-time policy mirrors Meta's Direct Send Beta v1 dispatchable set and pins the spec assumption ("single CTA URL button, up to three reply buttons") to the validator. Templates whose Meta-side curation includes a non-dispatchable button type therefore cannot be ingested by the Direct Send path until Meta re-curates them; post-assignment edits to `Template.metadata.buttons` via `update_template` are governed by FR-026 and are NOT bound by this fetch-time policy.
- **FR-003g**: For specific Meta library templates whose upstream URL-button `text` exceeds `MAX_BUTTON_LABEL_LENGTH` (20 chars) in one or more languages — preventing FR-003a/FR-003c from ever returning a usable Template and causing FR-003d to atomic-rollback every assignment in the affected locale — the Direct Send fetch-time adapter MUST consult a single, data-driven per-`(template_name, language)` button-label override map and substitute the URL button's `text` before the FR-003f.c length check. The map is the single source of truth for these overrides:

  ```python
  DIRECT_SEND_BUTTON_LABEL_OVERRIDES: dict[tuple[str, str], str] = {
      ("order_canceled_3", "pt_BR"): "Detalhes do pedido",   # 18 chars; upstream "Ver detalhes do pedido" = 22
      ("order_canceled_3", "es"):    "Detalles del pedido",  # 19 chars; upstream "Ver detalles del pedido" = 23
  }
  ```

  Normative rules:
  (a) **Location**: the map MUST be declared as a module-level constant alongside the existing per-component length constants (e.g., next to `MAX_BUTTON_LABEL_LENGTH` in `retail/agents/domains/agent_webhook/services/direct_send_constants.py`, or in a sibling module imported by `_meta_library_template_fetch.py`). Adding entries to the map is the ONLY supported way to introduce a new override; per-call branching on `template_name` inside the adapter is forbidden.
  (b) **Trigger — conditional on overflow**: the adapter MUST consult the map ONLY when the upstream `button.text` for an entry of `type == "URL"` exceeds `MAX_BUTTON_LABEL_LENGTH`. If the upstream label already fits within the limit (≤20 chars), the upstream value is honored verbatim and the map is NOT consulted. This preserves Meta as the source of truth whenever its content is dispatch-eligible and confines the override surface to the narrow "would otherwise FR-003d-rollback" cases.
  (c) **Lookup**: when overflow is detected, the adapter MUST look up `(template_name, language)` in the map. On hit, `button.text` is replaced with the map value before the length re-check; the new value is the persisted `Template.metadata.buttons[*].text`. On miss, the existing FR-003f.c failure path is taken (raise `DirectSendUnsupportedComponentError(component_type="buttons")` → FR-003c → FR-003d).
  (d) **Targeting**: the override applies EXCLUSIVELY to the single URL button identified by `type == "URL"`. The override mechanism does NOT apply to `QUICK_REPLY` buttons in v1; a `QUICK_REPLY` whose `title` exceeds `MAX_BUTTON_LABEL_LENGTH` MUST continue to raise `DirectSendUnsupportedComponentError` per FR-003f.c. (Future QUICK_REPLY overflows can be addressed by extending the map value's shape additively without re-amending this requirement.)
  (e) **Scope**: the override mechanism is button-label-`text`-only in v1. Body, header, and footer overflows are NOT addressable via this mechanism; they continue to follow FR-003f.c / FR-003e / the body length check unchanged. Future component overrides MAY extend the map value to a typed dict (`{"button_text"?: str, "body"?: str, ...}`) additively.
  (f) **Audit**: the adapter MUST emit `logger.info` at the override site identifying `template_name`, `language`, the upstream (overflow) text, and the substituted override text — e.g., `direct_send_button_label_overridden: template=order_canceled_3 language=pt_BR upstream='Ver detalhes do pedido' (22 chars) override='Detalhes do pedido' (18 chars)`. No audit flag is persisted on `Template.metadata.direct_send`; the additive sub-object continues to carry only `fetched_from_meta_library`, `fetched_at`, `requested_language`, and `actual_language` per `data-model.md §3`. Operator reconstruction of "was the override applied to this Template?" is intentionally log-derived in v1; promoting it to a queryable persisted flag is a follow-up (`button_label_overridden: bool` on `Template.metadata.direct_send`) once a use case demands it.
  (g) **Initial map content**: the v1 map MUST contain exactly the two seed entries shown above (`order_canceled_3` × `pt_BR`, `order_canceled_3` × `es`). Any additional entries are out of scope for this feature and require a follow-up spec amendment to add a new `(template_name, language)` tuple plus the localized override text. The map MUST NOT carry an `en` (or any other-language) entry for `order_canceled_3` because the upstream `en` label ("View order details" = 18 chars) already fits within `MAX_BUTTON_LABEL_LENGTH`; the trigger of (b) ensures the upstream value is honored verbatim regardless.
  (h) **Validation interaction**: the post-substitution length re-check after override application MUST still pass. A misconfigured map entry whose override value itself exceeds `MAX_BUTTON_LABEL_LENGTH` MUST raise `DirectSendUnsupportedComponentError(component_type="buttons")` — the override is a remediation, not a length-check bypass.
- **FR-004**: When the channel reports Direct Send enabled, the system MUST mark the persisted templates' current version as APPROVED at assignment time so they are immediately eligible for broadcast.
- **FR-005**: When the channel reports Direct Send disabled or the lookup fails, the system MUST fall back to the existing OrderStatus template-creation pipeline (Meta + Integrations) without any change in behaviour.

#### Template status lifecycle

- **FR-006**: The system MUST extend the Template version status enumeration with two new values: `PAUSED` and `FLAGGED`.
- **FR-007**: The system MUST treat `PAUSED` and `FLAGGED` as broadcast-disabling statuses, equivalent in effect to the existing pre-`APPROVED` states for the purpose of dispatch decisions. (The dispatch-time behavior is fully specified by FR-012; FR-007 carries the enum-level normative statement so the two new values cannot be silently downgraded to "informational" in a future refactor.)
- **FR-008**: The system MUST NOT change existing transitions or behaviour for the pre-existing statuses (`APPROVED`, `IN_APPEAL`, `PENDING`, `REJECTED`, `PENDING_DELETION`, `DELETED`, `DISABLED`, `LOCKED`).
- **FR-009**: The mechanism by which a version transitions to `PAUSED` or `FLAGGED` is **out of scope** for this feature; the spec only requires the values to exist and to disable broadcasts when present.

#### Broadcast dispatch

- **FR-010**: When the rule engine names a template, the system MUST resolve the local Template by name and inspect its current version status before any dispatch attempt.
- **FR-011**: If the current version is in `APPROVED`, the system MUST proceed with dispatch.
- **FR-012**: If the current version is in any other state — including the new `PAUSED` and `FLAGGED` — the system MUST skip the dispatch, persist no broadcast record, and emit an audit log entry that includes the template name, the version status, and the originating order-status event.
- **FR-013**: When the IntegratedAgent has Direct Send enabled and the template is dispatchable, the system MUST substitute every variable referenced in the template's body, header, footer, and button parameters before sending the message — Retail performs the substitution, not Meta.
- **FR-014**: When the IntegratedAgent has Direct Send enabled and the template is dispatchable, the system MUST send the messaging gateway a payload containing the final, fully-substituted message — including body text, header (text or image), footer, and buttons (CTA URL or quick replies, where applicable) — with the Direct Send flag set, the template name carried as the Direct Send identifier, and the message category set to `utility`. The `utility` category is the only category supported by the Direct Send Beta v1 (any other category is out of scope for this feature); the exact wire shape, including the `category` key, is pinned by `contracts/messaging-gateway-payload.md` §3.1.
- **FR-014a**: For Direct Send-enabled dispatch, a CTA URL button (sourced from a Meta library template `buttons` entry of `type: "URL"`) MUST be emitted on the wire as a top-level discriminator + sibling sub-object on `msg`, NOT inside `msg.buttons`. The canonical shape is:

  ```jsonc
  {
    "msg": {
      "direct_send": true,
      "category": "utility",
      "template": {"name": "<template_name>", "locale": "<locale>"},
      "body": "<substituted body>",
      "interaction_type": "cta_url",
      "cta_message": {
        "display_text": "<substituted button label>",
        "url": "<substituted URL>"
      }
    }
  }
  ```

  Normative rules:
  (a) The discriminator key MUST be spelled `interaction_type` — matching the existing `msg.interaction_type = "order_details"` shape emitted by `Broadcast._apply_order_details` at `retail/agents/domains/agent_webhook/services/broadcast.py:216`. The spelling `interactive_type` is NOT valid.
  (b) `msg.buttons` MUST NOT contain a CTA URL entry on the Direct Send path; the previously-documented `{"sub_type": "cta_url", "display_text", "url"}` shape (`contracts/messaging-gateway-payload.md` §3.3 and §5.1) is a documentation error and is superseded by this requirement.
  (c) The fetch-time button policy of FR-003f remains unchanged — at most one `URL` button per template (see FR-003f(d) for the count-limit clause that supplies this structural guarantee). The dispatch-time emission accordingly produces at most ONE `msg.cta_message` sibling object; additional CTA URLs are structurally impossible because the fetch-time validator rejects them.
  (d) FR-013's variable-substitution rule applies to BOTH `cta_message.display_text` and `cta_message.url`; both MUST be substituted server-side and contain no `{{N}}` placeholders on the wire.
  (e) Length-validation of `cta_message.display_text` against `MAX_BUTTON_LABEL_LENGTH` (20 chars) remains enforced post-substitution by the existing `_exceeds_direct_send_length_limits` defensive last check (`retail/agents/domains/agent_webhook/services/broadcast.py:820-850`), which MUST be updated to read from `msg.cta_message.display_text` instead of `msg.buttons[*].display_text`.
  (f) The `contracts/messaging-gateway-payload.md` §3.3, §3.4, and §5.1 examples that show `msg.buttons[*].sub_type = "cta_url"` MUST be corrected to reflect this requirement during US1 implementation; the spec, not the contract artifact, is the canonical source for the wire shape (matching the precedent set by FR-003e for the `header` shape correction).
  (g) The legacy path (FR-015) is untouched — `IntegratedAgent.direct_send=False` MUST keep emitting `msg.buttons = [{"sub_type": "url", "parameters": [{"type": "text", "text": "<id>"}]}]` byte-for-byte per FR-020 and `contracts/messaging-gateway-payload.md` §2.2; this requirement applies exclusively to the Direct Send path.
- **FR-014b**: For Direct Send-enabled dispatch, Quick Reply buttons (sourced from Meta library template `buttons` entries of `type: "QUICK_REPLY"`) MUST be emitted on the wire as a top-level flat array of post-substitution title strings on `msg`, NOT as objects inside `msg.buttons`. The canonical shape is:

  ```jsonc
  {
    "msg": {
      "direct_send": true,
      "category": "utility",
      "template": {"name": "<template_name>", "locale": "<locale>"},
      "body": "<substituted body>",
      "quick_replies": ["<substituted title 1>", "<substituted title 2>", "<substituted title 3>"]
    }
  }
  ```

  Normative rules:
  (a) `msg.quick_replies` is a JSON array of strings; each element is the post-substitution `title` of a `QUICK_REPLY` button from the library catalog response. NO wrapping object, NO `sub_type`/`type`/`id` field is emitted; the Meta library catalog's `id` field (if present in the catalog response) is intentionally NOT carried on the wire. Flows derives any downstream payload identifier from the title or from its own per-channel handling against Meta's Direct Send Beta endpoint.
  (b) `msg.buttons` MUST NOT contain `QUICK_REPLY` entries on the Direct Send path; combined with FR-014a, `msg.buttons` is LEGACY-ONLY on the wire — the Direct Send path NEVER emits a `msg.buttons` key. The previously-documented `{"sub_type": "reply", "id", "title"}` shape (`contracts/messaging-gateway-payload.md` §3.3 and §5.1; `direct_send_payload_builder.build_direct_send_buttons` lines 172-184) is a documentation/implementation error and is superseded by this requirement.
  (c) The fetch-time button policy of FR-003f remains unchanged — at most 3 `QUICK_REPLY` buttons per template (see FR-003f(d) for the count-limit clause that supplies this structural guarantee). The dispatch-time emission accordingly produces at most THREE elements in `msg.quick_replies`; additional entries are structurally impossible because the fetch-time validator rejects them.
  (d) FR-013's variable-substitution rule applies to each `msg.quick_replies[*]` element; each title MUST be substituted server-side and contain no `{{N}}` placeholders on the wire. (In practice OrderStatus QUICK_REPLY titles are static strings without `{{N}}` placeholders, but the substitution path MUST run defensively in case a future library template introduces them.)
  (e) Length-validation of `msg.quick_replies[*]` against `MAX_BUTTON_LABEL_LENGTH` (20 chars per entry, post-substitution) is enforced by `_exceeds_direct_send_length_limits` (`retail/agents/domains/agent_webhook/services/broadcast.py:820-850`), which MUST be updated to iterate over `msg.quick_replies` (a flat list of strings) instead of `msg.buttons[*].title`.
  (f) `msg.interaction_type` is NOT set when only `QUICK_REPLY` buttons are present — the `interaction_type` discriminator from FR-014a is reserved for the CTA URL case. A template that carries BOTH a `URL` button AND `QUICK_REPLY` buttons emits BOTH `msg.interaction_type = "cta_url"` + `msg.cta_message = {...}` (FR-014a) AND `msg.quick_replies = [...]` (FR-014b) as PARALLEL sibling keys on `msg`; each surface is independent on the wire and neither suppresses the other. The combined case is structurally permitted by FR-003f's per-type count limits (`≤1 URL` AND `≤3 QUICK_REPLY`); upstream Direct Send Beta acceptance of the combined shape is a Flows / Meta-side concern that the Retail spec does not enlarge — if upstream rejects it, the broadcast settles as `BroadcastMessage.status=FAILED` via the standard FR-035 flow.
  (g) `contracts/messaging-gateway-payload.md` §3.3, §3.4, and §5.1 MUST be corrected during US1 implementation to reflect this shape; the spec, not the contract artifact, is the canonical source (same precedent as FR-003e for the `header` shape).
  (h) The legacy path (FR-015 / FR-020) is untouched — `IntegratedAgent.direct_send=False` keeps whatever button shape it emits today byte-for-byte per FR-020. Today's `build_broadcast_template_message` does not handle `QUICK_REPLY` buttons at all (only `sub_type: "url"` and `sub_type: "payment_request"`); this requirement applies exclusively to the Direct Send path.
- **FR-015**: When the IntegratedAgent has Direct Send disabled, the system MUST send the messaging gateway the existing template payload (template name + locale + positional variables) with no Direct Send flag.
- **FR-016**: The system MUST persist a broadcast record for every dispatched message, regardless of which path produced it, so dispatch history, conversion attribution, and status callbacks remain consistent across both paths. Audit-log shapes for dispatch refusals and admissions are catalogued separately by FR-039; FR-016 covers the persisted `BroadcastMessage` row only, not the audit-log line emitted alongside (the two surfaces are independent — a row is persisted only on dispatch, while audit-log lines are emitted on every refusal AND on the agent-resolution admission per FR-039).

#### Direct Send identifier rules

- **FR-017**: The Direct Send identifier sent to the messaging gateway MUST be the local template's name. The system MUST ensure the identifier respects the Direct Send beta naming rules (lowercase alphanumerics and underscores only, maximum 512 characters); if a template's stored name violates the rule, the broadcast MUST NOT be sent through Direct Send.
- **FR-018**: *(removed — folded into Assumptions; see "OrderStatus agent's templates are already named in the lowercase-alphanumeric-with-underscores pattern". The number is intentionally retired so downstream cross-references remain stable.)*

#### Scope guard

- **FR-019**: For the initial release, only the OrderStatus agent is eligible for Direct Send. Other agents (Abandoned Cart, Payment Recovery, etc.) MUST keep using the existing template-creation and dispatch pipeline regardless of the channel's Direct Send flag.

#### Idempotency & retry safety

**Named Invariant — Exactly-Once Dispatch**: For every "single logical broadcast" (defined below), the system MUST issue at most one outbound POST to the messaging gateway AND persist at most one `BroadcastMessage` row, even when (a) the trigger event is delivered twice (VTEX webhook duplicate or upstream replay), (b) any background task retries, or (c) any inbound EDA event is re-delivered by the broker. This invariant applies to BOTH the Direct Send path and the legacy path; the dedup mechanism is unchanged across paths so the two cohorts cannot drift apart.

**Single logical broadcast (canonical idempotency tuple)**: for the order-status dispatch path, the tuple is `(Project, IntegratedAgent.uuid, OrderStatusDTO.orderId, OrderStatusDTO.currentState)` where `Project` is the project identifier — the implementation MAY serialize it as `Project.id` (the integer FK column on `IntegratedAgent`) or as `Project.uuid` (the public identifier), since both uniquely identify the same row; the current implementation uses `IntegratedAgent.project_id` (the FK integer) for cache-key length and to avoid an extra column read. The four-COMPONENT shape is what is normative (FR-029); the serialization of the project component is an implementation detail. Two events that disagree on any element of this tuple are DIFFERENT logical broadcasts (e.g. `invoiced` and `shipped` for the same `order_id` are two broadcasts; two `invoiced` webhooks for the same `(project, agent, order_id)` collapse into one). Abandoned Cart and Payment Recovery dispatch paths define their own tuples and are out of scope for this feature; the invariant statement here is scoped to OrderStatus.

**Boundary**: "exactly once" is a Retail-internal guarantee — no duplicate `BroadcastMessage` rows, no duplicate Flows POSTs from Retail. End-to-end exactly-once through Flows / Meta / device is NOT a Retail-side guarantee; Flows is at-least-once on its side and Meta provides its own at-most-once/at-least-once semantics. This spec does not enlarge those guarantees and does not require Retail to compensate when Flows reaches Meta but Retail records FAILED (see Edge Cases).

**Retry budget**: Retail does not retry on its own (retry budget = 0 for the order-status dispatch hot path). Retail IS retry-safe — i.e. tolerant of upstream/downstream retries — but never originates a retry from inside the dispatch use case. This is the explicit answer to the "Sisyphean vs. stoic" question: Retail is stoic. The only Retail-side retry surface is operator-initiated re-assignment after an FR-003d failure (which is a fresh `AssignAgentUseCase.execute` invocation with its own atomic block, not a Celery-level retry). All other "retries" originate upstream — VTEX re-firing an order-status webhook within or after the dedup window, the broker re-delivering an unacked EDA message — and are absorbed by the per-use-case idempotency contracts (FR-035, FR-036). Celery's `task.retry(...)` mechanism is NOT used by this pipeline (FR-038).

##### Duplicate trigger suppression

- **FR-028**: The system MUST suppress duplicate order-status webhooks within a configurable time window. The dedup mechanism uses an atomic compare-and-set primitive keyed on the canonical idempotency tuple `(project, integrated_agent.uuid, order_id, current_state)` (FR-029 — same components as the "single logical broadcast" tuple). The `project` component MAY be serialized as the FK integer (`Project.id` / `IntegratedAgent.project_id`) or as `Project.uuid`; both uniquely identify the project and the choice is an implementation detail (see "Single logical broadcast (canonical idempotency tuple)" above). When the key is already present, the event is skipped with an audit log entry; processing is otherwise unchanged. This applies identically to the Direct Send path and the legacy path — neither cohort is treated more strictly than the other. (The concrete primitive — Django's `cache.add` over Redis in production, LocMemCache under tests — is documented in `plan.md` Constraints — Idempotency & retry safety.)
- **FR-029**: The dedup window length MUST be configurable at the requirement level via the existing `ORDER_STATUS_DUPLICATE_WINDOW_SECONDS` environment variable (default: 60 seconds). The exact value is operational, not normative — tuning the window does not require a spec amendment. The dedup key COMPONENTS, however, ARE normative: a future change to the four-component shape (e.g. dropping `current_state`, or replacing `integrated_agent.uuid` with the project identifier alone, or adding a fifth component) IS a spec amendment and is forbidden without one. Switching the SERIALIZATION of an existing component — for instance, swapping `project_id` (FK integer) for `project_uuid` in the cache-key string — is NOT a spec amendment because the component identity is preserved (both uniquely identify the same project row); such a serialization change SHOULD still be reviewed for cache-key-collision risk against the live key namespace before merging.
- **FR-030**: Two order-status webhooks that differ ONLY in `current_state` (e.g. `invoiced` immediately followed by `shipped` for the same `(project, agent, order_id)`) MUST both be dispatched as separate logical broadcasts. The dedup key carries `current_state` for exactly this reason.
- **FR-031**: When the order-status agent resolution returns BOTH an official agent (matched on `settings.ORDER_STATUS_AGENT_UUID`) AND a custom agent with `parent_agent_uuid` for the same project, the official agent MUST take precedence and the custom agent MUST be ignored on that event. The `_lookup_order_status_agent` fallback chain (official → parent-flagged) is the canonical resolution and a single event MUST NEVER dispatch through both agents simultaneously.

##### Persistence-level idempotency keys

- **FR-032**: The system MUST enforce persistence-level idempotency on the `BroadcastMessage` table via two conditional unique constraints: `broadcasts_broadcast_id_unique` on `broadcast_id` (when not null) and `broadcasts_external_message_id_unique` on `external_message_id` (when not null). These constraints — already shipped on the live table — are restated here as REQUIREMENTS so a future migration cannot silently drop them.
- **FR-033**: The system MUST enforce persistence-level idempotency on the `BroadcastConversion` table via the unique constraint `(project, order_id)` (`broadcast_conversions_project_order_unique`). A single order produces at most one conversion row regardless of how many broadcasts targeted it during the order's lifecycle. When multiple broadcasts precede the `invoiced` event (e.g. `invoiced` Direct Send broadcast followed by a `shipped` broadcast on the same `order_id`), the LAST-TOUCH eligible broadcast (most recent non-FAILED dispatch from the payment-recovery agent — see `MarkBroadcastConvertedUseCase._select_last_touch_broadcast`) is the one credited; order-status broadcasts are NOT eligible for conversion attribution and never trigger a `BroadcastConversion` row on their own.
- **FR-034**: The `IntegratedAgent.broadcasts_delivered` counter is atomic-but-not-idempotent: it uses `F("broadcasts_delivered") + 1` and is incremented exactly once per FIRST DELIVERED transition (gated by `_apply_status_transition`'s `is_first_delivery` check that compares the incoming status with `message.previous_status`). Subsequent `DELIVERED` redeliveries on the SAME `BroadcastMessage` row MUST NOT re-increment the counter. The counter therefore stays consistent across broker re-deliveries of the `template-status` topic without requiring a separate idempotency token.

##### Background task / consumer retry safety

- **FR-035**: The inbound RabbitMQ status-update consumer (`BroadcastSendConsumer` on `retail.template-send`, `BroadcastStatusConsumer` on `retail.template-status`) MUST be safe to invoke repeatedly with the same event payload. The retry-safety is achieved by (a) looking up the target row with `select_for_update` so concurrent updates serialize, (b) keying lookups on `broadcast_id` / `external_message_id` (both bound by FR-032's unique constraints), and (c) the lifecycle-rank guard in `_is_out_of_order` that rejects backwards transitions. A re-delivery of the SAME event after a crash-before-ack MUST produce the same final row state as a single delivery would have.
- **FR-036**: `MarkBroadcastConvertedUseCase` MUST be safe to invoke repeatedly for the same `(project, order_id)`. Idempotency is enforced at write time by `BroadcastConversion.objects.get_or_create(project, order_id)`: a duplicate insert hits the unique constraint and is recovered as a logged warning (`conversion_already_recorded`) without raising. A re-delivery of the SAME `invoiced` event after a crash-before-ack MUST produce zero side effects beyond the warning.
- **FR-037**: `RecordBroadcastSentUseCase` is NOT retry-safe on its own — invoking it twice with the same DTO will produce two `BroadcastMessage` rows. The callers (`Broadcast.send_message` happy path and `Broadcast._record_failed_dispatch`) MUST therefore invoke it at most once per dispatch attempt. The `RuntimeError` / DB-error paths inside `send_message` and `_record_failed_dispatch` swallow exceptions exactly to preserve this "called at most once" invariant; do not change this without revisiting FR-032.
- **FR-038**: Every background task in the OrderStatus pipeline (currently `task_order_status_update`, `task_mark_broadcast_converted`, `handle_purchase_event_task`) is ONE-SHOT — Celery's `task.retry(...)` mechanism is NOT used and `acks_late` is NOT enabled (the project's Celery defaults to early-ack). Re-delivery of an event comes from one of two sources: (a) the broker (RabbitMQ) re-delivers an unacked message to the inbound EDA consumer — `BroadcastConsumer.consume` (`retail/broadcasts/consumers/broadcast_status_consumer.py:48`) acks ONLY after the use case method returns successfully, so a crash-before-ack triggers a redelivery that the use case absorbs idempotently (FR-035); (b) the upstream caller — VTEX webhook for `task_order_status_update`, scheduled trigger for housekeeping tasks — re-fires the entry-point task, which is absorbed by `MarkBroadcastConvertedUseCase`'s idempotency contract (FR-036) plus the trigger-side dedup (FR-028). A poison-message DLX is NOT a v1 requirement; if a transient error swallows work inside a Celery task, the upstream retrigger is the recovery mechanism. This stance is consistent with "retry budget = 0 on the dispatch hot path" (Exactly-Once Dispatch §Retry budget) and is the EXISTING behaviour of the codebase — no Celery retry configuration is added or removed by this feature.

##### Audit log shapes (distinguishable by skip reason)

- **FR-039**: The system MUST emit audit log entries for every dispatch refusal AND for the order-status agent-resolution admission (so log consumers can correlate dispatch decisions with which agent was selected), and the entry shapes MUST be DISJOINT so consumers can route on each class independently. Every shape MUST carry the project identifier as a TOP-LEVEL key — either `project_uuid={...}` (Retail-internal UUID) or `vtex_account={...}` (external account, used by the dedup and agent-resolution shapes because the upstream entry point keys on it) — so operators can filter by tenant (`checklists/tenant-isolation.md` CHK042). The six shapes are:
  - **dedup skip**: `[ORDER_STATUS] duplicate_skipped: vtex_account={...} agent_uuid={...} current_state={...} order_id={...}` (FR-028) — INFO level.
  - **Dispatch-gate skip (unified shape)**: `[BroadcastDispatch] skipped_due_to_status: project_uuid={...} vtex_account={...} agent={...} template={...} version_status={NOT_FOUND|PAUSED|FLAGGED|PENDING|REJECTED|IN_APPEAL|LOCKED|DISABLED|DELETED|PENDING_DELETION} event={data}` (FR-012) — WARNING level. Emitted by `Broadcast.get_current_template` for EVERY non-APPROVED skip class at this gate, including the no-row case (`version_status=NOT_FOUND`). The `version_status` discriminator pins the skip class so log consumers can route on each class independently. `project_uuid` and `vtex_account` are both top-level structured fields (FR-044) so operators can filter by either tenant identifier.
  - **Direct Send validation skip**: `[BroadcastDispatch] skipped_due_to_direct_send_validation: project_uuid={...} agent={...} template={...} reason={naming_rule|empty_body|component_length_limit} event={data}` (FR-017, research Decision 7) — WARNING level.
  - **Legacy downstream miss**: preserved bit-for-bit per FR-027 — WARNING level, existing shape. The live code emits ONE WARNING line under this class, originating in `retail/agents/domains/agent_webhook/services/broadcast.py` at `Broadcast.build_message` AFTER `get_current_template` returns `None`: `WARNING "Template not found or has no approved current version. Project: {project_uuid}, VTEX Account: {vtex_account}, Template: {template_name}, Data: {data}"`. FR-027 forbids changing the level, the wording, or the trailing field order on this line. Note that this line is somewhat redundant with the unified "Dispatch-gate skip" line emitted upstream by `get_current_template` (which now covers the no-row case via `version_status=NOT_FOUND`) — operators will see both lines per miss event. The downstream line is intentionally preserved to avoid touching the higher-level orchestration log shape; a future PR MAY drop it as a separate observability cleanup once log consumers have migrated to the unified shape. **Historical note**: the per-name miss line previously emitted by `Broadcast.get_current_template` (`"Template {template_name} does not exist in database or has no approved version. Project: ... VTEX Account: ... Data: ..."`) has been CONSOLIDATED into the unified "Dispatch-gate skip" shape above as of US3 (T031); FR-027's Exception clause documents this single legacy-line break.
  - **Direct Send assignment atomic failure**: `[DirectSend] assignment_failed_atomic: project_uuid={...} agent={...} template={...} requested_language={...} fallback_language=pt_BR reason={missing_translation|meta_unreachable|malformed_response}` (FR-003d) — ERROR level.
  - **Order-status agent resolution (admission)**: `[ORDER_STATUS] agent_resolved: vtex_account={...} agent_uuid={...} source={official|parent_agent}` (FR-031) — INFO level. Emitted once per resolved event by `_lookup_order_status_agent`; the `source` discriminator pins the official-vs-parent-flagged decision so a regression that silently routed Direct Send-enabled traffic through the parent-agent fallback (or vice versa) is observable in the audit log without having to re-derive the resolution from row state. Tenant-key uses `vtex_account` because the resolution log line is emitted at the entry point alongside the dedup shape; per FR-027 this existing line is preserved bit-for-bit and `project_uuid` MAY be added additively without breaking consumers (FR-044).

#### Tenant isolation

**Named Invariant — Tenant Isolation**: A project MUST NEVER dispatch through, read, or write data scoped to another project. Every database query that touches tenant-scoped models (`IntegratedAgent`, `Credential`, `Template`, `Version`, `BroadcastMessage`, `BroadcastConversion`), every EDA event, and every outbound HTTP call that consumes per-tenant credentials MUST be keyed by the canonical tenant identifier (directly or via a documented FK chain). Cross-tenant data ingress is structurally guarded by FK constraints; cross-tenant data egress is structurally guarded by request-validation gating at the assignment surface and by per-tenant credential scoping at dispatch time.

**Canonical tenant identifier**: `Project.uuid` (Retail-internal UUID). It is the public identifier emitted in logs, EDA payloads, observability tags, and API responses. The integer FK `Project.id` is the internal storage key used for joins and cache-key compactness; both uniquely identify the same row, so any tenant-scoping requirement that says "scoped by project" is satisfied by either serialization. `Project.vtex_account` is the EXTERNAL tenant identifier (the VTEX-side account name) used at the order-status webhook entry to resolve the project; a duplicate `vtex_account` value across two `Project` rows is a SECURITY BOUNDARY: the entry point logs `[ORDER_STATUS] multiple_projects` and returns `None` (no project resolved → no dispatch fired) rather than risk routing to the wrong tenant. Other identifiers (`IntegratedAgent.uuid`, `Agent.uuid`, `Template.uuid`, `Version.uuid`, `BroadcastMessage.uuid`, `BroadcastConversion.uuid`, `channel_uuid`) are NOT tenant identifiers — they are PER-PROJECT object identifiers and resolve to a single tenant via their FK chain (see "Tenant FK chain" below).

**Tenant boundary**: ONE project = ONE tenant. An (Agent, Project) pair is NOT a finer-grained tenant — every per-project record (`IntegratedAgent`, `Credential`, `Template`, `Version`, `BroadcastMessage`, `BroadcastConversion`) inherits its tenant from the project the agent is integrated into, never from the agent itself. A multi-agent project (e.g. OrderStatus + AbandonedCart + PaymentRecovery on the same project) shares the project's tenant boundary; the agents do not isolate from each other, only the project boundary isolates.

**Tenant FK chain (canonical scoping)**:

```
BroadcastMessage      ──► project (FK to Project)
BroadcastConversion   ──► project (FK to Project)
IntegratedAgent       ──► project (FK to Project)
Credential            ──► integrated_agent ──► project (transitive)
Agent                 ──► project (FK to Project; per-project Lambda boundary)
Template              ──► integrated_agent ──► project (transitive)
Version               ──► template ──► integrated_agent ──► project (transitive)
                          AND
                      ──► project (FK to Project, direct — see "dual-path scoping")
```

Every read or write that touches one of these tables MUST be scoped through this chain (directly via `project_uuid` / `project_id`, or transitively via the FK that owns the project).

**Dual-path scoping on `Template` / `Version`**: `Template.integrated_agent.project` (transitive) and `Version.project` (direct FK on `Version`) MUST always agree for every (Template, Version) pair. The redundancy is deliberate — `Version.project` exists so the dispatch-time queryset at `Broadcast.get_current_template` can filter by project without joining through `Template → IntegratedAgent`. The invariant `Template.integrated_agent.project_id == Version.project_id` is enforced by the assignment-time write logic; a future migration that breaks the invariant (e.g. allowing a Template to be re-pointed at a different IntegratedAgent without re-pointing its Versions) MUST also re-establish the invariant.

**Multi-credential surface taxonomy**: the colloquial label "Meta credentials" maps to FOUR distinct credential surfaces in this codebase, and the tenant boundary is different for each:

| Credential surface                                | Tenant scope                  | Where it is used                                                                                  | Scoping mechanism                                                                                  |
| ------------------------------------------------- | ----------------------------- | ------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `META_SYSTEM_USER_ACCESS_TOKEN` (env)             | **CROSS-tenant** (deliberate) | Library-catalog GET at agent-assignment time only (FR-003a). Reads a globally-curated public catalog. | Token grants read-only access to a Meta-curated public resource; no per-tenant data is exposed by reads against it (see Library-catalog tenant-isolation observation in `contracts/meta-library-catalog.md`). |
| Channel-side WhatsApp Cloud credentials           | **PER-tenant**                | Direct Send DISPATCH (POST to Flows → POST to `<PHONE_NUMBER_ID>/messages`). Owned by the project's WhatsApp Cloud channel. | Held by Integrations Engine; resolved per-channel via the `channel_uuid` snapshot on `IntegratedAgent` (see `contracts/messaging-gateway-payload.md` §8). |
| Flows internal-auth token                         | **CROSS-tenant** (deliberate) | Every Retail → Flows POST (broadcast dispatch).                                                  | Internal Weni-to-Weni service-account token; Flows enforces tenant scoping on its side via the `project` field in the request body (see `contracts/messaging-gateway-payload.md` §1). |
| Per-agent `Credential` rows (DB-backed)           | **PER-tenant**                | Lambda invocation parameters (e.g. VTEX app key/token).                                          | FK-chained via `Credential.integrated_agent.project`.                                              |

A statement of the form "Retail uses Meta credentials" without naming the surface is AMBIGUOUS and MUST be resolved against this table.

##### Tenant-isolation requirements

- **FR-040**: Every DB query that touches `IntegratedAgent`, `Credential`, `Template`, `Version`, `BroadcastMessage`, or `BroadcastConversion` MUST be scoped by the canonical tenant identifier (directly or via the FK chain documented above). The scoping MUST be enforced by ORM filters that include `project=...`, `project_uuid=...`, `project_id=...`, OR a join through one of the chained FKs (`integrated_agent__project=...`, `template__integrated_agent__project=...`). A queryset that returns rows from these tables WITHOUT applying tenant scoping (directly or transitively) is a regression and MUST fail code review. An explicit project-less lookup is permitted ONLY when the lookup key is itself globally unique across tenants (e.g. `BroadcastMessage.broadcast_id`, see FR-041); the global-uniqueness assumption that justifies the project-less lookup MUST be stated explicitly as an Assumption.
- **FR-041**: Inbound EDA event consumers (`BroadcastSendConsumer` on `retail.template-send`, `BroadcastStatusConsumer` on `retail.template-status`, the order-status webhook entry point) MUST resolve the target tenant deterministically. The supported resolution mechanisms are: (a) `Project.uuid` carried in the event payload (preferred when present); (b) `BroadcastMessage.broadcast_id` lookup (safe ONLY because Flows guarantees `broadcast_id` global uniqueness across all Weni tenants — see Assumptions); (c) `BroadcastMessage.external_message_id` lookup (safe ONLY because Meta guarantees `external_message_id` global uniqueness on its side); (d) `Project.vtex_account` lookup with the documented "duplicate → return None" SECURITY BOUNDARY. A consumer that resolves the tenant by ANY OTHER mechanism (e.g. `agent_uuid` alone, `channel_uuid` alone, customer URN alone) MUST justify that the chosen identifier is globally unique across tenants; without that justification the resolution is forbidden. When an event payload's project reference disagrees with the matched row's `project_id`, the matched row's tenant is canonical for any subsequent write (the event payload is informational, not authoritative); the disagreement MUST be logged at WARNING with both project identifiers so operators can detect a Flows-side cross-tenant misroute.
- **FR-042**: Outbound EDA / datalake events (`weni_datalake_sdk` / `CommerceWebhookPath`) MUST carry `project=str(integrated_agent.project.uuid)` as a required field so downstream analytics never aggregates across tenants. A future event-shape change that drops the `project` key OR replaces it with an identifier that is not globally unique across tenants is a forbidden regression.
- **FR-043**: The assignment surface MUST treat the `app_uuid` query parameter and the `Project-Uuid` header as a TENANT-BINDING PAIR. The required cross-validation is: the `project_uuid` field on the `IntegrationsService.get_channel_app(...)` response MUST equal the `Project-Uuid` header on the request; on mismatch, the assignment MUST be DENIED with HTTP 403 and an audit log entry that includes both project UUIDs and the operator identifier. The persisted `IntegratedAgent.channel_uuid` snapshot inherits the validated channel-to-project binding, so two `IntegratedAgent` rows in different projects MUST NEVER share the same `channel_uuid` (modulo the upstream guarantee that Integrations Engine never issues the same channel UUID for two channels — Assumption). For v1, this requirement is satisfied by the existing trust boundary on Integrations Engine (the channel-app endpoint's authorization layer plus DRF's `HasProjectPermission` on the assignment view); the explicit Retail-side cross-validation is documented in `plan.md` Constraints — Tenant isolation as a defense-in-depth follow-up. The requirement is restated here so that future Retail-side code MUST treat the cross-validation as the canonical defense and so that any regression in the Integrations-side guarantee is a known-broken precondition rather than a silent leak.
- **FR-044**: Every audit log entry, error log entry, and observability span tag emitted by the OrderStatus pipeline (assignment + dispatch + EDA consumer) MUST carry the project identifier as a top-level structured field — `project_uuid={...}` once the project is resolved, OR `vtex_account={...}` at the entry point where the project has not yet been resolved. The six audit-log shapes pinned by FR-039 already comply; FR-044 restates the rule as a cross-cutting requirement so any NEW log line introduced by future code MUST include the field. **Legacy preservation rule (interaction with FR-027)**: pre-existing log lines REMAIN COMPLIANT under this requirement regardless of how they spell the project identifier; FR-027 forbids removing or renaming the existing field shape and FR-044 does NOT mandate retroactive replacement on legacy lines. The compliant pre-existing shapes explicitly recognised by this rule are (non-exhaustive; new pre-existing shapes discovered by code review SHOULD be enumerated here as additive PR clarifications so the legacy cohort is fully catalogued over time): (a) lines that key on `vtex_account={...}` only — even when emitted after the project has been resolved (e.g. `agent_resolved`, `official_agent_not_found`, `no_agent_found`, `multiple_parent_agents` inside `_lookup_order_status_agent`); (b) lines that key on the human-readable form `Project: {project_uuid}, VTEX Account: {vtex_account}, …` instead of the `project_uuid={...}` / `vtex_account={...}` structured form (notably the downstream "Template not found or has no approved current version" WARNING line emitted by `Broadcast.build_message` AFTER `get_current_template` returns `None` — see FR-039 entry "Legacy downstream miss"). The per-name miss line previously emitted by `Broadcast.get_current_template` itself (under either name) is NO LONGER in this legacy cohort: it has been consolidated into the unified `[BroadcastDispatch] skipped_due_to_status: …` shape with `version_status=NOT_FOUND` per FR-027's Exception clause and FR-039's "Dispatch-gate skip (unified shape)" entry. Adding `project_uuid` additively (or upgrading the shape from human-readable to structured) on a still-legacy line is permitted (purely additive change) but NOT required by this feature; doing so MUST keep every existing field present and unchanged. Sentry / Elastic APM tags on every span SHOULD include `project_uuid` whenever the project is in scope; the legacy preservation rule (FR-027) does NOT permit removing existing tags but DOES permit adding `project_uuid` when it is not already present.
- **FR-045**: The local uniqueness scope of `Template.name` AND `Version.template_name` MUST be PER-INTEGRATED-AGENT — two `Template` rows in different `IntegratedAgent` records (whether in the same project or different projects) MAY hold the same `name`, and two `Version` rows belonging to different IntegratedAgents (or to different Templates within the same IntegratedAgent) MAY hold the same `template_name`. Today this is enforced by the ABSENCE of any `unique=True` constraint on `Template.name` / `Version.template_name` (the constraint was removed in migrations `templates/0007_alter_template_name.py` and `templates/0015_alter_version_template_name.py`); the requirement is restated here so a future migration that re-introduces a global `unique=True` on either column is a forbidden regression. Such a global constraint would silently break Direct Send for any second tenant that adopts the same Meta library template (which is the legitimate steady state — Meta's Direct Send Beta requires per-WABA template-name uniqueness, and each project has its own WABA, so per-WABA uniqueness on Meta's side maps to per-IntegratedAgent uniqueness on Retail's side). The dispatch-time lookup `integrated_agent.templates.filter(name=...)` is the canonical resolution and is FK-scoped to the tenant; a name collision across two projects' IntegratedAgents is therefore not a tenant-isolation issue.
- **FR-046**: The dispatch-time `PAUSED` / `FLAGGED` skip (FR-012) is per-tenant by construction (the FK chain `Version → Template → IntegratedAgent → Project` scopes the lookup at `Broadcast.get_current_template`). Flipping a Version's status in project A's Template MUST NEVER affect any other project's broadcasts. A future refactor that performs a project-less status lookup (e.g. `Version.objects.filter(template_name=..., status="APPROVED")` without a tenant join) is a forbidden regression.

#### Backward compatibility (Story 4 anchor)

**Named Invariant — Legacy-Path Byte-For-Byte Preservation**: For any `IntegratedAgent` whose `direct_send=False`, every aspect of dispatch and assignment behaviour MUST be byte-for-byte unchanged from the pre-feature baseline. This is the explicit normative form of the guarantee that User Story 4, FR-005, FR-015, and SC-004 collectively imply.

**Cohort the invariant applies to**: every existing `IntegratedAgent` at deploy time (because the Direct Send flag is stored as an optional key inside the existing `IntegratedAgent.config` JSON and absence of the key is interpreted as `False` — legacy rows are therefore structurally on the legacy path with zero schema change required), every newly-assigned `IntegratedAgent` against a non-Direct-Send channel (FR-005), every newly-assigned non-OrderStatus `IntegratedAgent` regardless of channel (FR-019), and every `IntegratedAgent` whose channel-side flag was flipped after assignment (the snapshot is taken at assignment time only — see Edge Cases).

**Public surface (the closure of the invariant)**: model fields exposed via DRF serializers; HTTP endpoint paths, methods, and required headers / query params; inbound webhook payload schemas (order-status, template-status, agent-assignment, send-test-template); outbound HTTP request shapes (Flows broadcasts, Integrations Engine template creation, Meta Graph); EDA and datalake event payloads; settings and environment-variable names / types / defaults; audit log line shapes; and Sentry / Elastic APM span tags. The detailed enumeration of every preserved field per model, per serializer, per outbound call, and per settings key lives in `plan.md` Constraints.

**"Byte-identical" definition**: for the legacy outbound HTTP path, "byte-identical" means semantic JSON equivalence — same set of keys, same values, same value types, and same array order; key order MAY differ since both producers and consumers serialize through `json.dumps`. The snapshot tests pinned at plan time (`tasks.md` T033 for dispatch and T034 for assignment) are the regression guard.

- **FR-020**: For every `IntegratedAgent` whose `direct_send=False`, the system MUST NOT change the outbound payload sent to Flows, the outbound calls to the Integrations service, the Meta template-creation traffic, the persisted broadcast / template / version row shape, or the existing audit log line shapes compared with the pre-feature baseline. The existing datalake event payloads (`weni_datalake_sdk` / `CommerceWebhookPath`) keep the same keys and value types.
- **FR-021**: The system MUST NOT rename, remove, or change the type of any existing field on `IntegratedAgent`, `Template`, `Version`, `IntegratedAgent.config`, or any DRF serializer that exposes them. The upstream `Agent` and `PreApprovedTemplate` models are not modified by this feature. The only model-level changes are: (a) two additive values appended to `Version.STATUS_CHOICES` (`PAUSED`, `FLAGGED`); (b) an additive optional sub-object `direct_send` inside `Template.metadata`, present only on Direct Send-path templates. The Direct Send flag itself adds NO new column to `IntegratedAgent` — it is stored as an optional key (`direct_send: bool`) inside the existing `IntegratedAgent.config` JSON; absence of the key is interpreted as `False` (FR-001). The `IntegratedAgent` column list is therefore byte-identical to the pre-feature shape. Adding a new optional read-only field (`direct_send`) to `ReadIntegratedAgentSerializer` is purely additive and does NOT count as a backward-compatibility break, given the assumption that downstream consumers ignore unknown JSON fields (see Assumptions); the field is computed at serialization time from `obj.config.get("direct_send", False)` so the wire shape is unchanged from US1's first implementation while the underlying storage moved from a column to the existing `config` JSON.
- **FR-022**: The system MUST NOT add any new required field to any inbound payload (order-status webhook, agent-assignment endpoint, send-test-template endpoint, template-status webhook, or any other endpoint touched directly or transitively by this feature). New fields on outbound payloads (`msg.direct_send`) are emitted only when `IntegratedAgent.direct_send=True` and never reach the legacy cohort. No URL path, HTTP method, required header, or required query parameter is renamed or removed.
- **FR-023**: The existing template-status webhook handler MUST NOT be modified by this feature. Its URL path, request shape, response codes, and downstream side effects on `IntegratedFeature.config` and `Version.status` are preserved. The future feature that delivers Meta's `PAUSED` / `FLAGGED` events to that handler will register the mapping; until that feature lands the handler treats unknown statuses by its current behaviour (log and drop), which is safe. No new Integrations Engine subscription is required at deploy time.
- **FR-024**: The system MUST NOT introduce any new environment variable or settings key. Deploying this feature without any settings change MUST be safe; legacy behaviour is preserved everywhere with no operational toggle.
- **FR-025**: The single migration introduced by this feature MUST be data-safe by construction — the `Version.STATUS_CHOICES` extension is purely additive so every existing row's status (one of the eight pre-existing values) remains valid. No new `IntegratedAgent` migration ships with this feature: the Direct Send flag is stored as an optional key inside the existing `IntegratedAgent.config` JSON (FR-001), so legacy rows need no backfill — absence of the key is interpreted as `False`, which is exactly the legacy behaviour. No data backfill is required for any row. The migration MUST be reversible by Django's default `migrate <app> <prev>` rollback. Reverting only the feature code (without rolling back the migration) is also safe — the new enum values stay unused once the dispatch gate code is removed, and the `config["direct_send"]` key stays unread on every row.
- **FR-026**: The `update_template` endpoint MUST accept `PAUSED` and `FLAGGED` as valid `status` input values and persist them as-is on the named `Version`. These statuses MUST NOT trigger the `current_version` promotion logic that runs only when `status=APPROVED`. Existing behaviour for every other status value is unchanged.
- **FR-027**: Existing observability surfaces that are NOT dispatch-time skip lines MUST be preserved bit-for-bit on the legacy path. This explicitly covers: Sentry / Elastic APM tags on the dispatch span (no rename or removal; an optional `direct_send` tag MAY be added and is omitted (or `False`) on the legacy path so existing dashboards continue to work unchanged); the downstream "Template not found or has no approved current version" WARNING line in `Broadcast.build_message` (FR-039 entry "Legacy non-APPROVED skip" item (ii)) which fires AFTER `get_current_template` returns `None` and is independent of the per-name miss class; every audit / error / info log line emitted OUTSIDE the dispatch-gate decision in `Broadcast.get_current_template`. **Exception — dispatch-gate consolidation in `Broadcast.get_current_template`**: the per-name miss WARNING line at `Broadcast.get_current_template` (previously `WARNING "Template {template_name} does not exist in database or has no approved version. Project: ... VTEX Account: ... Data: ..."`) is INTENTIONALLY consolidated with the PAUSED/FLAGGED audit shape into the unified `[BroadcastDispatch] skipped_due_to_status: ...` line catalogued by FR-039, so a single log shape covers every skip class at this gate (NOT_FOUND, PAUSED, FLAGGED, PENDING, REJECTED, IN_APPEAL, LOCKED, DISABLED, DELETED, PENDING_DELETION). The `version_status` discriminator makes the skip class observable without parser changes between classes. Log consumers that previously routed on the legacy per-name miss prefix `"Template ... does not exist ..."` MUST switch to routing on the audit prefix `[BroadcastDispatch] skipped_due_to_status:` with a `version_status=NOT_FOUND` filter; this is the only legacy-line break introduced by this feature and is justified by the unified observability surface across the dispatch gate.

### Key Entities *(include if feature involves data)*

- **IntegratedAgent**: The per-project record that binds a project to one of the agents (e.g. OrderStatus). Gains a Direct Send flag set at assignment time from the WhatsApp channel's flag. The flag determines which dispatch path is used for every broadcast originated by this agent.
- **Template (and its current Version)**: The local record of an outbound message template. Existing for both paths, but now must be persisted with full content (body, header, footer, buttons, language) when the IntegratedAgent has Direct Send enabled — because Meta no longer holds it. The version's status enumeration is extended with `PAUSED` and `FLAGGED`, both of which disable broadcast.
- **WhatsApp Channel (referenced from Integrations)**: An external record owned by the Integrations service that exposes a Direct Send flag. Retail reads this flag at agent-assignment time only; it is not re-fetched during the broadcast hot path.
- **Broadcast record**: The per-dispatch audit and tracking row already persisted today. Continues to be persisted for both paths (legacy and Direct Send), preserving downstream conversion attribution and status callback wiring.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After release, OrderStatus agents assigned to Direct Send-enabled channels can dispatch their first broadcast within minutes of assignment, without waiting for any Meta template-approval step (current baseline: hours to multiple days). **Measurement**: observable post-deploy metric only — no automated test ships in this PR because the time-to-first-broadcast collapses to "the existing assignment + dispatch hot paths run end-to-end without an inserted Meta-approval cycle", which the structural removal of the template-creation submission already guarantees (SC-003 + the absence of any new blocking step). The operator-facing happy-path walkthrough in `quickstart.md §0–§3` is the runnable end-to-end validation; staging dispatch latency on a Direct Send-enabled fixture is the production confirmation.
- **SC-002**: For Direct Send-enabled OrderStatus agents, 100% of broadcast attempts that target a `PAUSED` or `FLAGGED` template result in a skipped dispatch (zero messages reach the messaging gateway for those templates) and the skip is observable in the audit log. **Measurement under status-flip races**: when a version's status flips from `PAUSED` to `APPROVED` mid-dedup-window, the status read at dispatch time (AFTER the `cache.add` admit) is authoritative for that event. The 100% guarantee therefore applies to "events whose status read returned PAUSED/FLAGGED", not to "events that COULD have been APPROVED by the time the dispatch fired". A measurable corollary is: sum over a fixed window of (skipped_due_to_status entries for `PAUSED`/`FLAGGED`) MUST equal the count of "version was in PAUSED/FLAGGED at dispatch read time" — both observable from the audit log.
- **SC-003**: Assigning an OrderStatus agent to a Direct Send-enabled channel results in zero outgoing requests to the template-creation pipeline (Meta + Integrations) for that agent.
- **SC-004**: Assigning an OrderStatus agent to a non-Direct-Send channel produces the exact same template-creation traffic and the same final IntegratedAgent state as before this feature was released (zero regression on the legacy path). **Measurement**: snapshot tests pinned to fixtures (`tasks.md` T033 for the dispatch payload, T034 for the assignment outbound calls) record the exact outbound HTTP request bodies (Flows broadcast, Integrations Engine template-creation calls) and the persisted `IntegratedAgent` / `Template` / `Version` row diff against the pre-feature baseline. Any drift fails CI.
- **SC-005**: For every dispatched broadcast (Direct Send or legacy), a broadcast record exists with the correct status, template name, and contact identifier — no observability gap is introduced by the new path. **Measurement under retry / replay**: "every dispatched broadcast" means "every Flows POST that returned a `broadcast_id`". The conditional unique constraint `broadcasts_broadcast_id_unique` (FR-032) ensures the relation is 1:1 even under replays — a measurable corollary is: `COUNT(DISTINCT broadcast_id WHERE broadcast_id IS NOT NULL) == COUNT(BroadcastMessage WHERE broadcast_id IS NOT NULL)`. A re-delivery of the `template-send` topic observes the same row (via the `broadcast_id` lookup in `_link_message_to_broadcast`) and converges to the same `external_message_id` without producing a second row.
- **SC-006**: When a template's current version transitions back from `PAUSED` or `FLAGGED` to `APPROVED`, the next eligible order-status event results in a normal dispatch with no manual intervention.
- **SC-007**: The legacy-path no-regression guarantee holds **per IntegratedAgent** (every individual project's outbound payload and persisted-row shape are byte-identical with the pre-feature shape) AND **per fleet** (aggregate template-creation traffic across the Beta period is unchanged). The snapshot tests enforce the per-IntegratedAgent guarantee directly; the per-fleet guarantee follows because every IntegratedAgent independently satisfies it.
- **SC-008**: After deploy, the existing log lines, Sentry / APM tags, and datalake events emitted on the legacy dispatch path keep the same keys, value types, and emission frequency — measurable by replaying a baseline batch through staging and diffing log / span / event captures against the pre-feature baseline.
- **SC-009**: Exactly-once dispatch (the named invariant introduced in Requirements §Idempotency & retry safety) is measurable Retail-internally via the following invariants over any continuous time window: (a) for every unique `(project, integrated_agent.uuid, order_id, current_state)` tuple observed in the order-status webhook stream — where `project` is identified by either its FK integer or its UUID per FR-028's serialization rule — there is AT MOST ONE `BroadcastMessage` row whose `created_at` falls within `ORDER_STATUS_DUPLICATE_WINDOW_SECONDS` of the first observation; (b) `COUNT(DISTINCT broadcast_id WHERE broadcast_id IS NOT NULL) == COUNT(BroadcastMessage WHERE broadcast_id IS NOT NULL)` (FR-032); (c) `COUNT(DISTINCT external_message_id WHERE external_message_id IS NOT NULL) == COUNT(BroadcastMessage WHERE external_message_id IS NOT NULL)` (FR-032); (d) `COUNT(DISTINCT (project_id, order_id)) == COUNT(BroadcastConversion)` (FR-033); (e) `IntegratedAgent.broadcasts_delivered == COUNT(BroadcastMessage WHERE integrated_agent_id=... AND status IN (DELIVERED, READ))` after a quiescent period (FR-034). The equality holds because FR-035's lifecycle-rank guard rejects backward transitions, so the only ingress to `READ` is through `DELIVERED` — every row currently in `READ` was counted exactly once at its prior `DELIVERED` transition. If a future change relaxes the lifecycle-rank guard so that `READ` can be reached without a prior `DELIVERED` (e.g. consuming a Meta-side `read` event for a message that never delivered), this invariant MUST be restated against `previous_status` transition history rather than current `status`. End-to-end exactly-once across Flows / Meta / device is OUT OF SCOPE for this success criterion and is not measured Retail-side (see "Boundary" in the Exactly-Once Dispatch invariant).
- **SC-010**: "No cross-project data leakage" (the Tenant Isolation named invariant) is objectively measurable Retail-internally via the following SQL invariants over any continuous time window: (a) for every `BroadcastMessage` row where `integrated_agent_id IS NOT NULL`, `BroadcastMessage.project_id == BroadcastMessage.integrated_agent.project_id`; (b) for every `BroadcastConversion` row where `integrated_agent_id IS NOT NULL`, `BroadcastConversion.project_id == BroadcastConversion.integrated_agent.project_id`; (c) for every `(Template, Version)` pair where `current_version_id IS NOT NULL`, `Template.integrated_agent.project_id == Version.project_id` (the dual-path scoping invariant); (d) for every `Credential` row, `Credential.integrated_agent.project_id` resolves to a single `Project` row. Any violation of (a) / (b) / (c) is a hard tenant-isolation regression and MUST fail an audit query in CI. **Measurement under cross-project test fixtures**: SC-003 and SC-004 are verifiable PER PROJECT — each snapshot fixture (`tasks.md` T033 / T034) is scoped to a single project so a regression that mixes traffic between two test projects produces a fixture mismatch (not an averaged-out pass); the dispatch-payload snapshot includes `msg.project` as a literal field that the assertion compares byte-for-byte. The audit query for invariant (a) is materialized in the test suite as `tasks.md` T035c (a two-project cross-tenant regression guard that also asserts the dedup cache key, the datalake event payload, and the per-IntegratedAgent template lookup remain tenant-scoped); invariants (b)–(d) are spot-checked by integration tests because the FK constraints already make the runtime violation impossible.

## Assumptions

- The WhatsApp channel's Direct Send flag is exposed by the Integrations service and is the single source of truth at agent-assignment time; Retail reads it via the existing Integrations channel-app endpoint.
- Template content for Direct Send-enabled OrderStatus agents is sourced from Meta's library catalog at agent-assignment time, in the project's resolved language (FR-003a). The push-time single-language validation flow stays unchanged; the integration-time fetch is what materializes the language-correct content used at dispatch. After assignment, the local Template carries everything required to render the final message, and Meta is no longer consulted at broadcast time.
- The rule engine continues to return template variables in the existing positional shape (`{"1": "...", "2": "...", ...}`) plus the existing special keys (`button`, `image_url`); the Direct Send path consumes the same shape and only changes how the values are turned into a payload.
- The set of message components in scope for the OrderStatus Direct Send flow is the intersection of (a) Meta's Direct Send Beta-supported set — body text, optional header (text or image), optional footer, single CTA URL button, up to three reply buttons, optional TTL — and (b) the components the OrderStatus templates use today (body text with positional variables, optional image header, single URL button). Components outside this intersection (carousel, list, catalog, order_details, flow message) are out of scope for v1. **Header-shape source taxonomy** (FR-003e): at fetch time, Meta's library catalog ALWAYS delivers the `header` field as a plain text string; image headers on Direct Send-path Templates are introduced EXCLUSIVELY by post-assignment edits to `Template.metadata.header.header_type` via the `update_template` endpoint. The dispatch path therefore handles both `TEXT` and `IMAGE` header types, but the fetch path only handles `TEXT`.
- The IntegratedAgent's Direct Send flag is a snapshot captured at assignment time. Re-syncing the flag if the channel later changes is out of scope for this feature; operators re-assign the agent if they need to switch paths.
- **Direct Send flag storage (JSON-key storage — deliberate spec correction; see `data-model.md §1` Decision)**: the IntegratedAgent's Direct Send flag is stored as an optional key (`direct_send: bool`) inside the existing `IntegratedAgent.config` JSONField rather than as a new column on `IntegratedAgent` (FR-001, FR-021). Absence of the key is interpreted as `False` (FR-005). This choice is a deliberate spec correction relative to the original first-implementation draft (which added a new boolean column) and was made for three reasons: (a) zero schema change on `IntegratedAgent` collapses the rollout footprint to a single migration scope (the additive `Version.STATUS_CHOICES` extension only); (b) US4's "byte-identical legacy-path" guarantee is trivially provable because the `IntegratedAgent` column list is byte-identical to the pre-feature shape — no persisted-row diff against a pre-feature baseline can drift on the legacy cohort; (c) the existing `config` JSON is already the documented home for per-IntegratedAgent feature configuration (`abandoned_cart`, `payment_recovery`, `delivered_order_tracking_config`, etc.), so adding one more optional key is the smallest possible change consistent with the spec's "purely additive" rule. The wire shape exposed by `ReadIntegratedAgentSerializer` is unchanged from US1's first implementation: the field stays as a top-level read-only `direct_send` key, computed at serialization time from `obj.config.get("direct_send", False)`. The choice is recorded as a Decision in `data-model.md §1`.
- The OrderStatus agent's templates created through the assignment flow are already named in the lowercase-alphanumeric-with-underscores pattern (e.g. `weni_order_shipped_<timestamp>` from the legacy push flow, or simpler forms such as `weni_order_invoiced` for fixtures); the timestamp suffix is the historical default, not a hard requirement — any name that satisfies `^[a-z0-9_]+$` and is ≤ 512 chars is valid (FR-017). No additional naming work is required for them at v1.
- The mechanism for transitioning a template version into `PAUSED` or `FLAGGED` (e.g. inbound webhook from Integrations forwarding Meta's template-status events) will be delivered in a separate feature; this spec only requires the statuses to exist and to disable broadcasts.
- The order-status event sampling, deduplication, conversion attribution, and broadcast-record persistence behaviours are unchanged from today; the Direct Send flag only changes the payload shape sent to the messaging gateway and the variable-substitution responsibility.
- TTL configuration on Direct Send messages is not in scope for v1; if added later, it would be a small extension to the new payload builder.
- Downstream consumers of the `ReadIntegratedAgentSerializer` response (frontend dashboards, BI exports, support tools) ignore unknown JSON fields. The new read-only `direct_send` field is therefore additive and safe to ship without coordinating a frontend deploy. (FR-021.)
- Downstream consumers of `Version.status` (frontend dashboards, BI / datalake, support tools) tolerate unknown values — they either render them verbatim or fall through to a default "unknown" branch. No consumer is expected to crash on `PAUSED` / `FLAGGED`. Coordinating a frontend deploy to relabel the new statuses is a follow-up activity outside this feature's scope.
- The Integrations Engine keeps its existing template-creation pipeline (`fetch_templates_from_user`, `notify_integrations`, `create_template_message`, `create_template_translation`, `create_library_template_message`) unchanged for the duration of the Direct Send Beta. Any change on the Integrations side to those endpoints would silently break the legacy cohort regardless of this feature.
- The order-status webhook inbound payload schema (`Domain`, `OrderId`, `State`, `LastState`, `Origin.Account`, `Origin.Sender`), the agent-assignment endpoint contract (query params `app_uuid`, `channel_uuid`; header `Project-Uuid`; body `credentials`, `include_templates`), the `send_test_template` endpoint contract, and the Lambda response shape (`template`, `template_variables`, `contact_urn`, optional `language`, `image_url`, `button`, `order_details`, `payment_buttons`, `status`, `error`) are documented in the project's existing API documentation and remain unchanged. FR-022 forbids adding a required field to any of them.
- The existing template-status webhook handler today only persists the statuses delivered by the current Integrations subscription (`APPROVED`, `IN_APPEAL`, `PENDING`, `REJECTED`, `PENDING_DELETION`, `DELETED`, `DISABLED`, `LOCKED`). If Meta starts emitting `PAUSED` / `FLAGGED` before the future webhook-routing feature is delivered, the handler's existing behaviour for unknown statuses (log and drop) is preserved by this feature; no additional safeguards are required.
- **Retry budget = 0**: Retail does NOT retry on its own at the dispatch hot path. The system is retry-SAFE (tolerant of upstream Flows / Meta retries, broker re-deliveries, and operator-initiated re-assignments) but never originates a retry from inside the order-status dispatch use case or from inside `AssignAgentUseCase`. The only Retail-originated retry surface is operator-initiated re-assignment after an FR-003d failure (a fresh `AssignAgentUseCase.execute` invocation, not a Celery-level retry). Celery's `task.retry(...)` mechanism is NOT used by this pipeline (FR-038); re-delivery comes from the broker (RabbitMQ re-delivering unacked messages, absorbed by FR-035) or from the upstream caller re-firing the entry-point task (absorbed by FR-028's dedup + FR-036's `get_or_create`). This stance is the explicit answer to "is Retail Sisyphean or stoic?" — stoic.
- **AWS Lambda rule engine deterministic output**: the Lambda invoked by the order-status webhook is assumed to return DETERMINISTIC output for the same input — same `template`, same `template_variables`, same `contact_urn`, same `image_url`/`button`/`order_details` across replays of the same event. Without determinism, a transient Lambda retry could change the variable values mid-event and produce a non-equivalent message on a second attempt; Retail does not currently detect this. This is an upstream contract Retail trusts; the assumption is recorded so a future Lambda regression that breaks determinism is owned by the Lambda owners, not absorbed silently by Retail.
- **Flows broadcast endpoint idempotency**: Retail does NOT rely on Flows being idempotent on the same `(project, urn, msg)` payload — i.e. Retail does not send the same payload twice expecting Flows to deduplicate. The exactly-once boundary lives entirely in Retail (cache-based dedup on the webhook + `BroadcastMessage` unique constraints on `broadcast_id` / `external_message_id`); whatever Flows does internally is its responsibility. Conversely, Retail tolerates Flows replaying the SAME status callback (a `delivered` event for the same `external_message_id` arriving twice) via the lifecycle-rank guard in `_apply_status_transition` (FR-035).
- **Broker (RabbitMQ) delivery guarantee**: the inbound `retail.template-send` / `retail.template-status` topics are at-least-once. The consumer (`BroadcastConsumer.consume`) acks AFTER the use case completes, so a crash-before-ack triggers a re-delivery. The use cases are idempotent (FR-035) so re-deliveries are safe. Out-of-band: poison messages exceeding the Celery retry budget land on the broker's DLX (FR-038).
- **`BroadcastMessage.broadcast_id` global uniqueness across tenants**: Retail relies on Flows to allocate `broadcast_id` values that are globally unique across all Weni tenants. This is the precondition that makes the project-less lookup in `HandleStatusUpdateUseCase._link_message_to_broadcast` safe — without it, two projects could share a `broadcast_id` and a status callback would race-condition into the wrong row. The assumption is documented here and in `checklists/tenant-isolation.md` CHK014 / CHK038 as a Flows-side dependency.
- **Multi-broadcast-per-order conversion attribution (last-touch on payment-recovery)**: when multiple broadcasts target the same `(project, order_id)` over the course of the order's lifecycle (e.g. order-status broadcasts on `invoiced` / `shipped` / `delivered` followed by a payment-recovery broadcast on `payment-pending`), the `BroadcastConversion` row at `invoiced` is attributed to the LAST-TOUCH NON-FAILED payment-recovery broadcast. Order-status broadcasts are NEVER attributed; abandoned-cart is handled via VTEX-side UTM and is also NEVER attributed via this flow. This is the canonical attribution rule and is restated as FR-033.
- **Manual re-dispatch flow for failed broadcasts**: when a `BroadcastMessage` row settles in `FAILED`, there is no automated Retail-side re-dispatch (retry budget = 0). The implicit recovery path is the next order-status webhook for the same `(project, agent, order_id, current_state)` AFTER the dedup window expires — VTEX or the operator's tooling is the retrigger. Operators who need to recover a specific failed broadcast without waiting for a new webhook MUST do so through internal-tooling channels outside this feature; the spec intentionally does not introduce a dispatch retry endpoint.
- **No correlation identifier is added by this feature**: the existing identifiers — `OrderStatusDTO.orderId`, `IntegratedAgent.uuid`, `BroadcastMessage.uuid`, `broadcast_id`, `external_message_id` — already let operators stitch a single logical dispatch across logs, EDA events, and retry attempts. Adding a separate `trace_id` / `event_id` would be a cross-cutting observability change and is out of scope. Log lines on the dispatch path include `agent_uuid` and `order_id` precisely so this correlation is possible without a new identifier; see FR-039 for the canonical log-shape catalogue.
- **Dedup-cache backend (Redis)**: the dedup `cache.add` in `_is_duplicate_event` uses Django's default cache backend, which is Redis in production. The expected durability and failure semantics are documented in `plan.md` Constraints — Idempotency & retry safety. A Redis outage triggers fail-closed behaviour (the trigger fails fast, no `BroadcastMessage` is persisted) rather than fail-open (allow possible duplicate). Tests that exercise the dedup path MUST `@override_settings(CACHES={...LocMemCache...})` to isolate from real Redis (Constitution Principle III).
- **Cache keys are project-scoped** (Tenant isolation): every cache key that materializes tenant-scoped state MUST include a project component. The keys touched by this feature are `order_status_event:{integrated_agent.project_id}:{integrated_agent.uuid}:{order_id}:{current_state}` (the order-status dedup key — FR-028) and `project_by_vtex_account_{vtex_account}` / `project_by_uuid_{project.uuid}` / `project_domain_{project.uuid}` (the project-resolution caches). A future cache-key change that drops the project component or replaces it with a non-globally-unique component is a forbidden regression — the canonical key shapes are pinned in `plan.md` Constraints — Tenant isolation.
- **Datalake event payloads carry the canonical tenant identifier**: every `weni_datalake_sdk` / `CommerceWebhookPath` event emitted by the OrderStatus pipeline includes `project=str(integrated_agent.project.uuid)` as a top-level required field (FR-042). Downstream analytics consumers (BI, dashboards, audit pipelines) MAY rely on this field as the canonical tenant key — it is globally unique by virtue of `Project.uuid`'s `unique=True` constraint and never aliases across tenants.
- **AWS Lambda function-name namespace is per-project**: the per-agent Lambda is invoked by name `retail-setup-{hash_13_digits}` where the hash is `SHA256(agent.name + agent.uuid.hex)` (`retail/agents/domains/agent_management/usecases/push.py:112-123`); the agent is per-project (`Agent.project` FK), so the function name itself is per-project by construction — different agent UUIDs across two projects produce different SHA256 prefixes, and the IAM role attached to Retail is scoped so that only function names following this naming convention can be invoked. A function-name collision across two projects' agents is therefore impossible by construction. This assumption is the precondition that makes "the rule engine for project A cannot accidentally invoke project B's Lambda" structurally true (FR-040 — DB scoping; the Lambda invocation itself is FK-scoped via `IntegratedAgent.agent.lambda_arn` which derives from the per-project agent's UUID).
- **VTEX `defaultLocale` per tenant never aliases as a tenant identifier**: the locale-driven Meta library fetch at agent-assignment time keys on the project's resolved language (FR-003a / FR-003b). Two projects' fetches MAY legitimately resolve to the same locale (e.g. both `pt_BR`), which is fine because the library catalog is GLOBAL (a Meta-curated public resource — see `contracts/meta-library-catalog.md` §9 and FR-003a) and there is no per-tenant content to cross-pollinate. The locale is NOT a tenant identifier; per-project content materialization at assignment time is what isolates tenants — the local `Template.metadata` is FK-chained to the project via `Template.integrated_agent.project`, and project A's persisted `pt_BR` content cannot be read or overwritten by project B's assignment.
- **Channel-app endpoint enforces channel-to-project ownership upstream**: Retail TRUSTS Integrations Engine to be the source of truth for the channel-to-project binding — the `IntegrationsService.get_channel_app(...)` response's `project_uuid` is the canonical project for the channel, and Integrations' own authorization layer prevents an operator on project A from reading project B's channel by its `app_uuid`. This trust is the upstream precondition that makes the v1 assignment flow tenant-safe before the Retail-side explicit cross-validation (FR-043) lands. A regression in Integrations' authorization layer that lets project A read project B's channel-app row would break the precondition and is documented as a known dependency in `contracts/integrations-channel-app.md` §9.
- **Meta library catalog is a GLOBAL Meta-curated public resource**: the library-catalog GET at agent-assignment time (FR-003a) reads templates that Meta curates for ALL Weni tenants — there is no per-tenant content in the catalog and no per-tenant data is exposed by reads against it. This is the precondition that justifies using a SINGLE `META_SYSTEM_USER_ACCESS_TOKEN` for all projects' library-catalog reads (Multi-credential surface taxonomy in Tenant isolation): a cross-tenant credential is acceptable here because the resource it grants access to is itself cross-tenant by design. This assumption is restated in `contracts/meta-library-catalog.md` §9 as a tenant-isolation observation; if Meta ever introduces per-WABA library-catalog content the assumption breaks and the credential surface MUST be re-evaluated.
- **Flows guarantees `broadcast_id` global uniqueness across all Weni tenants** (re-stated for tenant isolation): already documented under "Idempotency & retry safety" assumptions; restated here because it is the precondition that makes the project-less consumer lookup safe (FR-041). Flows is the authority on `broadcast_id` allocation and MUST NOT issue the same value for two distinct broadcasts in any project — see `contracts/messaging-gateway-payload.md` §7.3.
- **Image attachments use a fixed `image/jpeg:` MIME prefix** (legacy parity): the existing `Broadcast.build_broadcast_template_message` builds `attachments[*]` as `f"image/jpeg:{image_url}"` regardless of the actual content type of the underlying URL (PNG, WebP, etc.). The Direct Send path inherits this shape verbatim — `Broadcast.build_direct_send_message` (T013) emits the same `f"image/jpeg:{image_url}"` literal so the legacy snapshot tests (T033) do not drift and FR-020's byte-for-byte preservation rule is satisfied. A future content-type-aware refactor that derives the MIME prefix from the URL or from a header response would be a backward-compatibility break for the legacy cohort and is therefore explicitly out of scope for this feature; if delivered later it MUST update both builders symmetrically and re-snapshot T033/T035a so the two cohorts stay aligned.

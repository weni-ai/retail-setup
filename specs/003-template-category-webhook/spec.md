# Feature Specification: Direct Send Template Incorrect-Category Webhook

**Feature Branch**: `003-template-category-webhook`

**Created**: 2026-05-24

**Status**: Draft

**Input**: User description: "We are now going to implement a webhook to handle a direct send template incorrect category detection. (...) Integrations will send us the project_uuid, app_uuid, template_name, template_category, template_correct_category, with this data we should fetch the IntegratedAgents that are linked to this app and update their template statuses to FLAGGED if the category and correct_category do not match or if the category is not 'UTILITY'."

## Clarifications

### Session 2026-05-24

- Q: Should the webhook follow the existing `TemplatesStatusWebhook`'s unprefixed free-form logging pattern, or align with the newer `[TAG] event_name: k=v` pattern used by `[CART_SERVICE]` (`retail/webhooks/vtex/services_cart_abandonment_unified.py`), `[ORDER_STATUS]` / `[CONVERSION_TRACKING]` (`retail/vtex/tasks.py`), and spec 002's broadcast audit trails (`[BroadcastDispatch]`, `[DirectSend]`, `[AssignAgent]`)? → A: Adopt the `[TAG] event_name: k=v` pattern (Pattern B). The new webhook MUST emit every line with the literal `[DirectSendCategoryWebhook]` prefix, a snake_case `event_name` token drawn from a closed enumeration (`received`, `flagged`, `flag_replay_noop`, `no_action_required`, `auto_demoted`, `no_matching_integrated_agent`, `template_not_found`, `template_has_no_current_version`, `completed`, `unexpected_error`), and `k=v` context pairs. A `reason=<sub_reason>` k=v MUST be carried on `flagged` events (`reason ∈ {category_mismatch, category_not_utility, category_mismatch_and_not_utility}`). Log-level discipline mirrors `[ORDER_STATUS]`: INFO for happy-path / state transitions / replays, WARNING for "expected but unhappy" lookup misses, ERROR with `exc_info=True` for unexpected exceptions only. Each request MUST emit a `received` line at start, zero-or-more per-IntegratedAgent outcome lines, and a `completed` line on the success path — OR an `unexpected_error` line on the caught-exception path. The legacy unprefixed format used by `TemplatesStatusWebhook` is NOT to be used by this webhook. (FR-009 / FR-009a / FR-009b / FR-009c / FR-009d.)

### Session 2026-05-25

- Q: When the webhook's matched `current_version.status` is already `FLAGGED` and a subsequent webhook arrives with `template_category == template_correct_category == "UTILITY"` (a corrected-category signal), should the webhook auto-demote the Version back to `APPROVED`, or remain one-directional (operator-only recovery via `UpdateTemplateUseCase`)? → A: Auto-demote. The webhook now bidirectionally tracks Meta's category determination: `FLAGGED` is set when any clause of FR-006 fires, and `APPROVED` is restored when both clauses are false on an already-`FLAGGED` Version. A new `auto_demoted` token joins the FR-009a `event_name` enumeration; the previously-defined `no_action_required_already_flagged` token is **removed** because its scenario now writes (`status = "APPROVED"`) instead of being a no-op. FR-014's "demotion is OUT OF SCOPE" framing is removed: the operator-driven `UpdateTemplateUseCase` path remains supported as a manual recovery channel, and this webhook becomes a second, automatic recovery channel. US2 AS2 is rewritten accordingly. (FR-006c / FR-007c / FR-007d / FR-008 / FR-009a / FR-009b / FR-009c / FR-009d / FR-014.)
- Q: Should the auto-demote fire regardless of how the prior `FLAGGED` was set (this webhook vs. operator-driven `UpdateTemplateUseCase` with `status="FLAGGED"`), or only when the prior `FLAGGED` was attributable to this webhook? → A: Always demote, regardless of flag source. The demote condition is the simple two-part check (`current_version.status == "FLAGGED"` AND the FR-006 flagging condition evaluates to false); no `Version.flagged_source` column is added and no migration is introduced (preserving the plan.md "No new database migration (A10)" commitment). Operators who manually set `FLAGGED` via `UpdateTemplateUseCase` for reasons unrelated to category mismatch accept that a subsequent corrected-category webhook will demote that `FLAGGED` to `APPROVED`; the same operator path can be used to re-flag. This trade-off is recorded as Assumption A11.
- Q: Under the prior FR-006 / FR-006c rules, a payload like `{template_category: "MARKETING", template_correct_category: "UTILITY"}` arriving against an already-`FLAGGED` Version produced a `flag_replay_noop` (no demote) because the demote was gated on the strict `template_category == template_correct_category == "UTILITY"` corrected-category signal — both clauses of the two-clause FR-006 condition needed to be false. Should this strict demote signal be relaxed so that **any** payload Meta sends with `template_correct_category == "UTILITY"` demotes an already-`FLAGGED` Version, and (symmetrically) the flag rule simplifies to be governed by the same single field? → A: Adopt a single-field eligibility model. `template_correct_category` is the sole source of truth for Direct-Send eligibility — it carries Meta's content-determined "this template should be UTILITY" determination, which is the only signal that matters for whether the template can be dispatched on the Direct Send path. The flag rule simplifies to `template_correct_category != "UTILITY"` (FR-006 rewritten), and the demote rule (against an already-`FLAGGED` Version) becomes the symmetric inverse `template_correct_category == "UTILITY"` (FR-006c rewritten). Consequences: (a) `template_category` becomes a **purely diagnostic / audit-log field** — it is captured on every `k=v` payload (FR-009d) so operators can see Meta's current label, but it never participates in the flag-or-demote decision. (b) FR-006b's composite reason `category_mismatch_and_not_utility` is removed because there is now only one flag clause; the FR-009a `reason` sub-enumeration collapses to the single value `correct_category_not_utility`. (c) Replays converge idempotently regardless of prior state because flag and demote are exact inverses on the same field — no payload-vs-prior-state oscillation is possible. (d) A `<non-UTILITY>/UTILITY` payload arriving against an `APPROVED` Version is now `no_action_required` — Retail trusts Integrations' `template_correct_category` signal as the eligibility gate; if Meta's actual current category later rejects the dispatch, that surfaces through Meta's own response on the broadcast path, not through this webhook. Dispatch-time gating remains `Version.status != "APPROVED"` per spec 002's FR-007 / FR-012; no change is required there. (FR-006 / FR-006a / FR-006b / FR-006c / FR-007c / FR-007d / FR-008 / FR-009a / FR-009d / FR-014 / A11.)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Block dispatch when Integrations detects an incorrect template category (Priority: P1)

When the Direct Send pipeline detects that a template's Meta-determined correct category is not `UTILITY` (the only category supported by the Direct Send Beta v1), the Integrations service notifies Retail by calling a dedicated internal webhook. Retail then locates every IntegratedAgent linked to that WhatsApp Cloud app inside the named project, finds the local template whose name matches the payload, and flips its current version status to `FLAGGED`. Any subsequent broadcast that would have used that template is silently skipped by the existing dispatch gate (per feature 002's FR-007 / FR-012), preventing customer-visible deliveries of policy-violating content. The reverse signal — Integrations later notifying that the template's correct category is `UTILITY` — auto-demotes a `FLAGGED` Version back to `APPROVED` and re-admits it on the next dispatch attempt.

**Why this priority**: This is the entire reason for the feature. Without it, an incorrectly-categorized template keeps broadcasting through the Direct Send path until an operator manually intervenes — exposing the WABA to Meta's enforcement actions (rate-limit reductions, paused message templates, possible Direct Send Beta access revocation) and delivering policy-violating content to end customers. The webhook is the only automated stop-loss; making this work correctly is the v1 bar.

**Independent Test**: With (a) a project that has at least one IntegratedAgent whose templates were created against a specific WhatsApp Cloud `app_uuid`, and (b) one of those templates having a current version in `APPROVED`, fire the webhook with that `(project_uuid, app_uuid, template_name)` and a `template_correct_category` value of `MARKETING` (or any non-`UTILITY` value); the `template_category` field can carry any value (it does not affect the decision). Verify that the template's `current_version.status` transitions to `FLAGGED`, that a subsequent order-status webhook for that template is skipped by the existing dispatch gate, and that the operation is recorded in the audit log.

**Acceptance Scenarios**:

1. **Given** an IntegratedAgent in project `P` whose `current_version` of template `T` is `APPROVED` and whose versions were created against `app_uuid=A`, **When** Integrations posts the webhook with `{project_uuid: P, app_uuid: A, template_name: T, template_category: "MARKETING", template_correct_category: "MARKETING"}`, **Then** the template's `current_version.status` is updated to `FLAGGED` (because `template_correct_category != "UTILITY"`), the response is HTTP 200, and an audit log entry records the template, the IntegratedAgent, both category values, and the reason (`correct_category_not_utility`).
2. **Given** the same setup as scenario 1, **When** the webhook is posted with `{template_category: "UTILITY", template_correct_category: "MARKETING"}`, **Then** the template's `current_version.status` is updated to `FLAGGED` (because `template_correct_category != "UTILITY"`), the response is HTTP 200, and the audit log entry's reason is `correct_category_not_utility`. (The `template_category="UTILITY"` value is captured on the audit-log `k=v` payload but does not affect the decision — see FR-006a.)
3. **Given** the same setup as scenario 1, **When** the webhook is posted with `{template_category: "MARKETING", template_correct_category: "UTILITY"}` or `{template_category: "UTILITY", template_correct_category: "UTILITY"}` (or any payload with `template_correct_category == "UTILITY"`), **Then** the template's `current_version.status` is unchanged (it remains `APPROVED`), the response is HTTP 200, and the audit log entry records that no flagging condition was met (event `no_action_required`).
4. **Given** two IntegratedAgents in the same project — IA-1 and IA-2 — both of which created templates named `T` against `app_uuid=A`, **When** the webhook fires with `(P, A, T)` and a flagging condition is true (`template_correct_category != "UTILITY"`), **Then** the `current_version.status` of template `T` is set to `FLAGGED` on both IA-1 and IA-2, and the audit log records one update per IntegratedAgent.

### User Story 2 - Webhook is safe to retry and replay (Priority: P2)

The Integrations service treats this notification as part of a courier-driven pipeline that can deliver the same event more than once (transient network errors, broker re-delivery, manual operator replays). Retail must therefore process this webhook idempotently — replays must converge on the same final state (`FLAGGED`) without raising errors, without producing duplicate audit-log entries that look like independent events, and without overwriting an already-`FLAGGED` template version's status with anything else.

**Why this priority**: Without idempotency, retries from the upstream courier pipeline would either (a) fail the operator with 4xx/5xx responses that don't reflect the true state, or (b) repeatedly mutate the template version and pollute the audit log. The first breaks the courier's retry semantics; the second breaks operator observability. Idempotency is a hard requirement for any webhook on the Direct Send pipeline.

**Independent Test**: Fire the webhook for a `(project, app, template)` that triggers a flagging condition, observe the response and the resulting `FLAGGED` state. Fire the exact same payload again immediately. Verify that the response is HTTP 200, that the template's `current_version.status` is still `FLAGGED` (no change), and that the audit log records the second call as a no-op replay (separate from the first call's update entry).

**Acceptance Scenarios**:

1. **Given** the same setup as US1's scenario 1, **When** the webhook is posted twice with the same payload, **Then** both responses are HTTP 200, the template's `current_version.status` is `FLAGGED` exactly once (no re-write), and the audit log shows one `flagged` entry and one `flag_replay_noop` entry.
2. **Given** a template whose `current_version.status` is already `FLAGGED` (set by a prior call), **When** the webhook is posted with any payload whose `template_correct_category == "UTILITY"` (e.g. `{template_category: "MARKETING", template_correct_category: "UTILITY"}` — the corrected-category signal Integrations sends after Meta confirms the template is genuinely UTILITY-eligible — or `{template_category: "UTILITY", template_correct_category: "UTILITY"}`), **Then** the template's status is auto-demoted to `APPROVED` (the corrected-category signal is honored — see FR-006c / FR-007d), the response is HTTP 200, and the audit log records `auto_demoted` with `previous_status=FLAGGED`, `new_status=APPROVED`. The `template_category` value is captured on the audit-log `k=v` payload but does not affect the decision.

### User Story 3 - Misrouted or stale webhooks fail closed without partial writes (Priority: P3)

When the webhook payload references a project, app, or template that Retail cannot locate (e.g. typo in `app_uuid`, project that was deleted, template that was never persisted on Retail's side), Retail must not raise a server error to Integrations and must not partially-write any state. The webhook returns HTTP 200 (so the upstream courier does not retry indefinitely) and records the miss in the audit log so operators can investigate the upstream cause.

**Why this priority**: A 5xx response would trigger upstream retries that would never succeed; a partial write (e.g. flagging a template that happens to match `template_name` in an unrelated project) would corrupt cross-tenant data. Failing closed with a logged miss is the only safe behavior. This is a P3 because the dominant flow is the success path; misrouted events are exceptional but must be handled correctly.

**Independent Test**: Fire the webhook with a `(project_uuid, app_uuid, template_name)` tuple that does not match any existing IntegratedAgent + Template pair (e.g., random UUIDs and a non-existent template name). Verify the response is HTTP 200, no template's status changes in the database, and the audit log records a "no matching IntegratedAgent / template" miss with the full payload values.

**Acceptance Scenarios**:

1. **Given** a webhook payload whose `project_uuid` matches an existing project but whose `app_uuid` does not correspond to any Version's `integrations_app_uuid` for that project's IntegratedAgents, **When** the webhook fires, **Then** the response is HTTP 200, no Template or Version row is mutated, and the audit log records `no_matching_integrated_agent` with all five payload values.
2. **Given** a webhook payload whose `(project_uuid, app_uuid)` correctly identifies one or more IntegratedAgents but whose `template_name` does not match any Template on any of those IntegratedAgents, **When** the webhook fires, **Then** the response is HTTP 200, no Template or Version row is mutated, and the audit log records `template_not_found` per IntegratedAgent inspected.
3. **Given** a webhook payload whose matched Template has no `current_version` set (an inconsistent local state — never observed in production but defensively guarded), **When** the webhook fires, **Then** the response is HTTP 200, no Version row is mutated, and the audit log records `template_has_no_current_version`.

### Edge Cases

- **Webhook payload missing one or more required fields** (e.g. `project_uuid` not supplied, or `template_correct_category` empty): the request is rejected at the serializer level with HTTP 400 and a field-by-field validation error. No state is mutated and no audit entry is written (DRF's standard 400 logging is sufficient).
- **Webhook payload includes `template_correct_category` values outside the Meta-known set** (e.g. lowercase `"utility"` or an unknown string like `"PROMO"`): values are compared as-is (case-sensitive). A lowercase `"utility"` is NOT equal to the literal `"UTILITY"` and therefore triggers the flagging condition (`reason=correct_category_not_utility`). Integrations is the contracting party for the value namespace; Retail does not normalize. Unknown / invalid non-blank values still produce a deterministic flag-or-no-flag outcome (typically flag, because they fail the `template_correct_category == "UTILITY"` check). The `template_category` field is captured on every audit-log `k=v` payload for diagnostic visibility but does not participate in the decision (see FR-006a). Empty-string values are NOT handled at this layer — they are rejected one layer earlier at the serializer (HTTP 400 per FR-003 / FR-010a; see the "missing one or more required fields" Edge Case row above), so the use case never observes an empty value for any of the five required fields.
- **`template_correct_category` equals `"UTILITY"`** (regardless of `template_category` — e.g. `{template_category: "MARKETING", template_correct_category: "UTILITY"}` OR `{template_category: "UTILITY", template_correct_category: "UTILITY"}`): this is the "Meta confirms the template is UTILITY-eligible" case. Against an `APPROVED` Version, no flagging condition fires, the template's status is unchanged, and the audit log records `no_action_required` (US1 scenario 3). Against a `FLAGGED` Version, the auto-demote condition fires (FR-006c / FR-007d) and the Version transitions to `APPROVED`.
- **Multiple IntegratedAgents in the same project share the same `(app_uuid, template_name)` pair** (e.g. an old IntegratedAgent with `is_active=False` and a new one with `is_active=True`): the webhook updates ALL matching IntegratedAgents' Templates, including inactive ones. The audit log entries identify which IntegratedAgent's template was updated. (Filtering on `is_active=True` would silently desync historical IntegratedAgents' template metadata — see Assumptions.)
- **Template's `current_version` is already in a non-`APPROVED`, non-`FLAGGED` state when the webhook fires** (e.g. `PAUSED`, `PENDING`, `REJECTED`, `DELETED`): the webhook still transitions the version to `FLAGGED` if a flagging condition is met. `FLAGGED` is a terminal-for-dispatch state per spec 002's FR-007 — it dominates over `PENDING`/`REJECTED` because the user-facing risk (incorrect category being broadcast) outweighs the existing non-`APPROVED` reason. The audit log records the previous status so operators can investigate the override.
- **Template's `current_version.status` is already `FLAGGED` when a NEW flagging payload arrives** (i.e. another `template_correct_category != "UTILITY"` payload, possibly with a different `template_category` value than the one that originally caused the flag): the status remains `FLAGGED` (idempotent) and the second call emits a `flag_replay_noop` audit line that carries the new category values via the `template_category` / `template_correct_category` k=v fields per FR-009d — the formal `reason=` token is reserved for `flagged` events (FR-009a) and is intentionally absent on `flag_replay_noop`. The `current_version` is NOT replaced; only the audit log's category-value fields differentiate.
- **Template's `current_version.status` is already `FLAGGED` when a CORRECTED-category payload arrives** (any payload with `template_correct_category == "UTILITY"`, regardless of `template_category` — e.g. `MARKETING/UTILITY` after Meta confirms the template's content is genuinely UTILITY-eligible, OR `UTILITY/UTILITY`): the Version is auto-demoted to `APPROVED` (per FR-006c / FR-007d) and the call emits an `auto_demoted` audit line carrying `previous_status=FLAGGED`, `new_status=APPROVED`. The dispatch gate (spec 002's FR-007 / FR-012) re-admits the template on the next broadcast attempt with no operator action required. This is the **only** path by which this webhook writes a non-`FLAGGED` status to a Version. Auto-demote fires regardless of how the prior `FLAGGED` was set (this webhook OR operator-driven `UpdateTemplateUseCase`) per Assumption A11.
- **Multi-step flag-then-correct flow** (Meta re-categorizes a template after a flag): T0 — Version `V1` is `APPROVED`. T1 — a category webhook with `template_correct_category="MARKETING"` arrives → `V1.status` becomes `FLAGGED`, audit `flagged` emitted with `reason=correct_category_not_utility`. T2 — Meta re-evaluates the template's content (e.g. operator edits the template on the Meta side, or Meta's classifier reconverges) and Integrations re-fires the webhook with `template_correct_category="UTILITY"` → `V1.status` is auto-demoted to `APPROVED`, audit `auto_demoted` emitted. The dispatch gate immediately re-admits the template. State can oscillate as Meta-side detection flips back and forth (`correct_category=MARKETING → correct_category=UTILITY → correct_category=MARKETING → …`); each transition is fully audit-logged. This is the intended recovery path for the auto-demote channel — see FR-014.
- **Project exists but has no IntegratedAgents at all** (e.g. agents were unassigned): the webhook returns HTTP 200 and the audit log records `no_matching_integrated_agent` (same as US3 scenario 1).
- **`app_uuid` value matches a `Version.integrations_app_uuid` for an IntegratedAgent in a DIFFERENT project than the one named in `project_uuid`**: cross-tenant isolation is enforced — the IntegratedAgent is NOT considered a match because the IntegratedAgent's `project_id` must equal `project_uuid`. The audit log records `no_matching_integrated_agent` for the named project (the audit line does NOT leak any reference to the IntegratedAgent in the other project).
- **`template_name` differs in case from the locally-persisted Template name** (e.g. payload says `"weni_order_invoiced"` but local row is `"Weni_Order_Invoiced"`): comparison is case-sensitive (DB-native string equality). Integrations is the contracting party for the casing of `template_name`; Retail does not normalize. A casing mismatch results in `template_not_found` (US3 scenario 2). Assumption: the OrderStatus agent's locally-persisted names are lowercase snake_case (see Assumptions) so this edge case is operationally improbable.
- **Concurrent webhook calls for the same `(project_uuid, app_uuid, template_name)`**: the underlying status update is a single-row UPDATE against `Version.status`; Postgres' implicit row-level UPDATE lock serializes the writes (Django's default isolation level is `READ COMMITTED`, which does not by itself serialize transactions, but the per-row exclusive lock taken by `UPDATE` is sufficient for this single-column write). Both concurrent calls converge on the same final state (`FLAGGED`), each produces its own audit log entry, and no application-level lock or distributed coordination is needed. This convergence property is structurally enforced by Postgres' row-level locking under Django's default per-statement transaction model and is verified operationally via the SQL parity check in `quickstart.md §10.1` (no `FLAGGED` Version without a corresponding audit-log entry); no concurrency unit test is added because Django's `TestCase` runs every test inside a single connection's transaction, which makes simulating two interleaved connections brittle and high-maintenance for a property already structurally enforced at the database level.
- **`project_uuid` references a project marked `is_blocked=True`**: the webhook still processes the event and updates the template's status. Blocking a project (`Project.is_blocked`) gates outbound flows (broadcasts, billing) — it does not gate inbound state-correctness signals like this category-detection webhook. Blocking a project must not prevent Retail from honoring a Meta-side category determination.
- **Integrations replays the same webhook hours/days later (e.g. broker queue re-delivery after an outage)**: idempotent per US2 — the second call observes the existing `FLAGGED` state and records a `flag_replay_noop` audit entry (when the replay carries a flagging payload — `template_correct_category != "UTILITY"`) or auto-demotes the Version to `APPROVED` and records an `auto_demoted` audit entry (when the replay carries a corrected-category payload — `template_correct_category == "UTILITY"`, per FR-006c / FR-007d). No additional cooldown or dedup window is introduced for this webhook (the existing order-status dedup mechanism is dispatch-time only and is not relevant here).
- **Heterogeneous-status fan-out under a corrected-category payload**: when the fan-out matches multiple IntegratedAgents whose Versions are in different starting states (e.g. IA-1's `current_version.status == "APPROVED"`, IA-2's `current_version.status == "FLAGGED"`) and the payload's `template_correct_category == "UTILITY"`, each IntegratedAgent's Version is processed independently on its own branch: IA-1 follows the `no_action_required` path (no write, audit line `no_action_required`), and IA-2 follows the auto-demote path (writes `APPROVED`, audit line `auto_demoted` per FR-006c / FR-007d). The response `detail` collapses to `"Mixed outcomes."` whenever the per-IA outcomes disagree (mapping pinned in `data-model.md §5.2`); the per-IA detail lives in the audit log, not the HTTP body. `templates_updated` counts only the demote write (`= 1` in this example) per the FR-010 direction-agnostic counter rule.

## Requirements *(mandatory)*

### Functional Requirements

#### Webhook surface

- **FR-001**: The system MUST expose a new internal HTTP POST webhook endpoint that accepts JSON payloads from the Integrations service when an incorrect template category is detected on a Direct Send template. The endpoint MUST be reachable under the existing `retail.webhooks.templates` URL namespace (alongside the existing `templates-status/api/notification/` webhook) so operators can find both Direct Send-related webhooks in one place.
- **FR-002**: The webhook MUST require the same internal-communication authentication used by the existing `TemplatesStatusWebhook` (DRF `CanCommunicateInternally`). No project-level permission check is performed because the caller is a trusted internal service (Integrations) and the payload itself carries the `project_uuid` that gates the lookup.
- **FR-003**: The webhook MUST accept exactly five required fields in the JSON body: `project_uuid` (UUID), `app_uuid` (UUID), `template_name` (non-empty string, the local Template's `name`), `template_category` (non-empty string, the category Meta reports the template currently has), and `template_correct_category` (non-empty string, the category Meta says the template SHOULD have). Missing or empty fields MUST return HTTP 400 with a field-level error. The webhook MUST NOT accept any other fields (unknown keys are ignored silently for forward-compatibility, matching DRF's default serializer behavior).
- **FR-003a**: The webhook MUST NOT impose any allow-list on the values of `template_category` / `template_correct_category` at the validator level. Values are compared as-is for the flagging-condition checks defined in FR-006. (Rationale: Integrations is the contracting party for the category-value namespace, and Meta may extend the category set in the future; pinning Retail's validator to today's known categories would force a follow-up release to accept any new Meta-side category.)

#### Lookup

- **FR-004**: For each incoming webhook, the system MUST locate every `IntegratedAgent` row where (a) `IntegratedAgent.project_id` equals the project whose `uuid` matches the payload's `project_uuid`, AND (b) at least one `Version` linked to one of that IntegratedAgent's Templates has `integrations_app_uuid` equal to the payload's `app_uuid`. This is the canonical interpretation of "IntegratedAgents linked to this app" in the user's description: the `(project, app)` link is materialized via the `Version.integrations_app_uuid` column, NOT via a separate column on `IntegratedAgent` (which does not store `app_uuid` directly — its `channel_uuid` is a different identifier).
- **FR-004a**: The IntegratedAgent lookup MUST include rows whose `is_active=False`. (Rationale: an old IntegratedAgent whose templates were created against the same `app_uuid` still has those templates in the database; an incorrect-category determination is a property of the template content, not of the IntegratedAgent's activation status. Filtering on `is_active=True` would silently desync historical templates and break the "FLAGGED is terminal-for-dispatch" guarantee from spec 002.)
- **FR-004b**: When the lookup in FR-004 returns no matches, the system MUST return HTTP 200 (so the upstream courier does not retry) and record an audit log entry tagged `no_matching_integrated_agent` with the full payload values. No Template or Version row is mutated.
- **FR-005**: For each matching IntegratedAgent, the system MUST locate the `Template` whose `name` equals the payload's `template_name` and whose `integrated_agent_id` equals the matched IntegratedAgent's primary key. The lookup MUST be case-sensitive and exact-match (no `ILIKE`, no normalization). When no Template is found for an IntegratedAgent, the system MUST skip that IntegratedAgent silently and record an audit log entry tagged `template_not_found` with the IntegratedAgent's identifier and the requested `template_name`.
- **FR-005a**: When the matched Template's `current_version` is `NULL` (an inconsistent local state — never observed in production but defensively guarded), the system MUST skip the update and record an audit log entry tagged `template_has_no_current_version`. No Version row is mutated.

#### Flagging condition

- **FR-006**: For each matched Template whose `current_version` is non-`NULL`, the system MUST evaluate the flagging condition: `template_correct_category != "UTILITY"`. If true, the template MUST be flagged per FR-007. If false (i.e. `template_correct_category == "UTILITY"`), the system MUST leave the Version's status unchanged and record an audit log entry tagged `no_action_required` — UNLESS the matched Version's status is already `"FLAGGED"`, in which case FR-006c's auto-demote condition fires. The `template_category` field is NOT consulted for the flag-or-demote decision; it is captured on every audit-log `k=v` payload (FR-009d) for diagnostic visibility but does not participate in the rule. (Pinned by Clarifications session 2026-05-25 — supersedes the prior two-clause `(template_category != template_correct_category) OR (template_category != "UTILITY")` rule.)
- **FR-006a**: The flagging condition MUST be evaluated with strict string equality (`==` / `!=`) on the raw `template_correct_category` payload value against the literal `"UTILITY"`. Case folding, whitespace trimming, and value normalization are explicitly forbidden — Integrations is responsible for sending the value in the agreed-upon case (the literal string `"UTILITY"` for the only Direct Send-eligible category). The `template_category` field is NOT compared against `template_correct_category` and is NOT compared against `"UTILITY"`; it is captured verbatim for audit logging only.
- **FR-006b**: When the FR-006 flagging condition fires (`template_correct_category != "UTILITY"`), the system MUST flag the template exactly once and record the audit reason as the single value `correct_category_not_utility`. The single-clause rule has only one reason and produces no composite reasons. (Pinned by Clarifications session 2026-05-25 — supersedes the prior `category_mismatch` / `category_not_utility` / `category_mismatch_and_not_utility` three-value enumeration.)
- **FR-006c**: When the matched Template's `current_version.status` is already `"FLAGGED"` AND the FR-006 flagging condition evaluates to **false** (i.e. `template_correct_category == "UTILITY"` — the corrected-category signal, regardless of `template_category`), the system MUST auto-demote the Version per FR-007d. This is the inverse of the FR-006 flagging condition (both rules are governed by the same single field — `template_correct_category` — making them exact symmetric inverses) and is the **only** path by which this webhook writes a non-`"FLAGGED"` status value. The auto-demote fires regardless of how the prior `"FLAGGED"` was originally set (this webhook OR operator-driven `UpdateTemplateUseCase`) — see Assumption A11.

#### Status update

- **FR-007**: When the flagging condition fires, the system MUST update the matched Template's `current_version.status` to the literal value `"FLAGGED"`. The update MUST be performed via the existing `Version` model's status field; no new table, no new column, and no new version row are introduced by this feature.
- **FR-007a**: The update MUST NOT change the Template's `current_version` pointer (the Template still points at the same Version row; only the status string is updated). This preserves the precedent set by spec 002's FR-026 / `update_template.py`: `current_version` is promoted only on `APPROVED` transitions.
- **FR-007b**: The update MUST persist via a single `UPDATE` on the Version row (Django's `save(update_fields=["status"])` or equivalent). No transaction wrapping is required because the update is a single-row write per Template; the iteration across multiple matched IntegratedAgents is best-effort sequential and does not require atomicity (one IntegratedAgent's success is independent of another's failure).
- **FR-007c**: When the `current_version.status` is already `"FLAGGED"` before processing, the system MUST dispatch to one of two paths based on the FR-006 flagging condition: (a) if the flagging condition is **true** (`template_correct_category != "UTILITY"` — a replay of a prior flag, possibly with a different `template_category` value), the system MUST NOT issue a redundant `UPDATE` and the audit log entry MUST be tagged `flag_replay_noop` so operators can distinguish a fresh flag from a replay; (b) if the flagging condition is **false** (`template_correct_category == "UTILITY"` — the corrected-category signal, regardless of `template_category`), the system MUST auto-demote the Version per FR-007d and emit an `auto_demoted` audit log entry. This is the single dispatch point that distinguishes the two replay paths and is the only place the demote write originates.
- **FR-007d**: When the auto-demote condition fires (FR-006c / FR-007c clause (b)), the system MUST update the matched Template's `current_version.status` to the literal value `"APPROVED"`. The update MUST be performed via the same single-row `UPDATE` mechanism as FR-007b (Django's `save(update_fields=["status"])`). The Template's `current_version` pointer MUST NOT be changed (only the status string is updated; the precedent set by FR-007a is preserved). No new Version row is created. The demote is auditable via the `auto_demoted` log line (FR-009a) which carries `previous_status=FLAGGED` and `new_status=APPROVED`.

#### Idempotency

- **FR-008**: The webhook MUST be idempotent at the Version level in both directions. Replays with identical flagging payloads (`template_correct_category != "UTILITY"`) MUST converge on the same final `FLAGGED` state; replays with identical corrected-category payloads (`template_correct_category == "UTILITY"`) against a `FLAGGED` Version MUST converge on the same final `APPROVED` state; subsequent replays of the same corrected-category payload (now against an `APPROVED` Version) follow the `no_action_required` path (FR-006) because FR-006c requires `status == "FLAGGED"` to trigger the demote. Because flag and demote are exact symmetric inverses on the single `template_correct_category` field, the same payload arriving twice in a row CANNOT oscillate the Version's state — a property structurally enforced by the single-field rule (Clarifications session 2026-05-25). Replays MUST NOT raise errors. The replay audit entries MUST be distinctly tagged — the canonical triple is `flagged` (first time a flagging condition fired and the Version transitioned to `FLAGGED`), `flag_replay_noop` (subsequent flagging payloads observed `FLAGGED` and did nothing), and `auto_demoted` (the first corrected-category payload after a `FLAGGED` transition that demoted the Version back to `APPROVED`).
- **FR-008a**: The webhook MUST NOT introduce a dedup cache, distributed lock, or sliding-window suppression. Idempotency is achieved purely by the two-path dispatch in FR-007c. (Rationale: the upstream courier already debounces; introducing a Retail-side cache would expand the cache-backend surface for a webhook whose writes are themselves idempotent single-row UPDATEs.)

#### Audit log

- **FR-009**: Every log line emitted by this webhook MUST follow the structured format `[DirectSendCategoryWebhook] <event_name>: <key=value> <key=value> ...` — the same `[TAG] event_name: k=v` shape used by `[CART_SERVICE]` (`retail/webhooks/vtex/services_cart_abandonment_unified.py`), `[ORDER_STATUS]` / `[CONVERSION_TRACKING]` (`retail/vtex/tasks.py:113-177`), and spec 002's broadcast audit trails (`[BroadcastDispatch]`, `[DirectSend]`, `[AssignAgent]`). The literal `[DirectSendCategoryWebhook]` tag is the stable dashboard-filter token; the `event_name` is the snake_case operation discriminator drawn from the closed enumeration in FR-009a; the `key=value` pairs carry the per-event context defined in FR-009d. The unprefixed free-form pattern used by the legacy `TemplatesStatusWebhook` (`retail/webhooks/templates/usecases/template_status_update.py`) and `OrderStatusUseCase` (`retail/webhooks/vtex/usecases/order_status.py`) MUST NOT be used by this webhook. (Pinned by Clarifications session 2026-05-24.)
- **FR-009a**: The `event_name` token (the snake_case discriminator immediately after the `[TAG]` and before the colon) MUST be exactly one of:
  - `received` — request entry; emitted exactly once at request start with the full payload.
  - `flagged` — a Version transitioned to `FLAGGED`; emitted once per matched Template whose flagging condition fired.
  - `flag_replay_noop` — the matched Version is already `FLAGGED`; no write performed.
  - `no_action_required` — `template_correct_category == "UTILITY"` (regardless of `template_category`) AND the matched Version is **not** already `FLAGGED`; no write performed.
  - `auto_demoted` — the matched Version was `FLAGGED` and a corrected-category payload (`template_correct_category == "UTILITY"`, regardless of `template_category`) arrived; the system wrote `status = "APPROVED"` per FR-006c / FR-007d.
  - `no_matching_integrated_agent` — the lookup of FR-004 returned zero IntegratedAgents for `(project_uuid, app_uuid)`.
  - `template_not_found` — a matched IntegratedAgent has no Template whose `name` equals `template_name` (per FR-005).
  - `template_has_no_current_version` — a matched Template's `current_version` is `NULL` (per FR-005a).
  - `completed` — request exit on the success path; emitted exactly once with the same counter values the HTTP 200 response carries.
  - `unexpected_error` — caught unexpected exception; emitted exactly once on the HTTP 500 path with `exc_info=True` per FR-009b.

  This enumeration is the contract surface for operator dashboards and MUST be treated as additive-only: new event names MAY be added in future PRs, but existing names MUST NOT be renamed or removed. For `flagged` events specifically, the `k=v` payload MUST additionally include a `reason=<flagging_reason>` field whose value is the single token `correct_category_not_utility` (per FR-006b — the only reason under the single-clause flag rule pinned by Clarifications session 2026-05-25). The `reason` sub-enumeration is additive-only; future flag clauses (e.g. a future Meta-side category disqualification beyond `correct_category != "UTILITY"`) MAY add new reason values but MUST NOT rename or remove `correct_category_not_utility`.
- **FR-009b**: Log-level discipline MUST be:
  - **INFO** for `received`, `flagged`, `auto_demoted`, `no_action_required`, `flag_replay_noop`, and `completed`. These cover the happy path, the two directional state transitions (`flagged` and `auto_demoted`), and idempotent replays — operator dashboards treat them as normal traffic.
  - **WARNING** for `no_matching_integrated_agent`, `template_not_found`, and `template_has_no_current_version`. These are "expected but unhappy" cases that complete the request successfully but never result in a `FLAGGED` write; an upstream typo, stale data, or local-state inconsistency typically explains them, and they MAY trigger an operator follow-up. This mirrors the legacy `TemplatesStatusWebhook`'s `logger.warning(f"No IntegratedFeature found...")` discipline.
  - **ERROR with `exc_info=True`** for `unexpected_error` ONLY (the HTTP 500 path of FR-010b). Domain-level misses (FR-004b, FR-005, FR-005a) are NOT ERROR-level — they are WARNING per the clause above. This mirrors `retail/vtex/tasks.py:174-177` (`logger.error(f"[ORDER_STATUS] unexpected_error: ...", exc_info=True)`).
- **FR-009c**: Each well-formed webhook invocation MUST emit log lines in this sequence:
  1. EXACTLY ONE `received` line at request start, BEFORE any database lookup, carrying the five payload values (`project_uuid`, `app_uuid`, `template_name`, `template_category`, `template_correct_category`).
  2. ZERO-OR-MORE outcome lines after the lookup completes — either one `no_matching_integrated_agent` line when FR-004 returns zero matches, or one line per inspected IntegratedAgent tagged with the appropriate event name (`flagged`, `flag_replay_noop`, `auto_demoted`, `no_action_required`, `template_not_found`, or `template_has_no_current_version`).
  3. EXACTLY ONE `completed` line at request exit on the success path, carrying `templates_updated=<N>` and `integrated_agents_inspected=<N>` — the same two counters returned in the HTTP 200 response body per FR-010.

  On the caught-exception path, the system MUST emit EXACTLY ONE `unexpected_error` line with `exc_info=True` and MUST NOT emit a `completed` line. The `received` line MAY be present or absent depending on where the exception fired.
- **FR-009d**: The `k=v` payload on each event MUST include AT LEAST these fields (additional context fields MAY be added as needed):
  - **`received`**: `project_uuid`, `app_uuid`, `template_name`, `template_category`, `template_correct_category`.
  - **`flagged`**: `project_uuid`, `app_uuid`, `template_name`, `template_category`, `template_correct_category`, `integrated_agent_uuid`, `template_uuid`, `version_uuid`, `previous_status`, `new_status=FLAGGED`, `reason=correct_category_not_utility` (the single-value reason under FR-006b — see FR-009a). The `template_category` value is captured verbatim for diagnostic visibility but does not appear in the `reason` token.
  - **`flag_replay_noop`**: same as `flagged` minus `new_status` and `reason` (the existing `FLAGGED` state is reported via `previous_status=FLAGGED`).
  - **`no_action_required`**: `project_uuid`, `app_uuid`, `template_name`, `template_category`, `template_correct_category`, `integrated_agent_uuid`, `template_uuid`, `version_uuid`, `previous_status` (always the pre-call status — the call is a no-op).
  - **`auto_demoted`**: `project_uuid`, `app_uuid`, `template_name`, `template_category`, `template_correct_category`, `integrated_agent_uuid`, `template_uuid`, `version_uuid`, `previous_status=FLAGGED`, `new_status=APPROVED`.
  - **`no_matching_integrated_agent`**: `project_uuid`, `app_uuid`, `template_name`, `template_category`, `template_correct_category` (the five payload values verbatim — no IntegratedAgent / Template / Version identifiers because none were resolved).
  - **`template_not_found`**: `project_uuid`, `app_uuid`, `template_name`, `integrated_agent_uuid` (the IntegratedAgent that was inspected and lacked a Template with the given name).
  - **`template_has_no_current_version`**: `project_uuid`, `app_uuid`, `template_name`, `integrated_agent_uuid`, `template_uuid`.
  - **`completed`**: `project_uuid`, `app_uuid`, `template_name`, `templates_updated`, `integrated_agents_inspected`.
  - **`unexpected_error`**: `project_uuid`, `app_uuid`, `template_name`, `error=<str(exc)>`. All three payload identifiers are always present because `unexpected_error` is emitted only from inside the use case's outer `try / except`, after DTO construction (FR-010b's HTTP 500 path is downstream of the serializer's `is_valid(raise_exception=True)` and the view's `DirectSendCategoryDTO(**validated_data)` build, both of which complete successfully before the use case is invoked). The full traceback is delivered via `exc_info=True` (Python logging's stdlib mechanism), NOT serialized into the `k=v` payload.
- **FR-009e**: The audit log MUST NOT include any free-form operator-supplied content (the webhook caller is Integrations, a trusted internal service, not a human operator). The entire log surface is bounded by the five payload fields plus Retail-internal identifiers (IntegratedAgent / Template / Version `uuid`s and previous/new status values), so PII redaction is not required and no field MAY be truncated or hashed for privacy reasons.

#### Response shape

- **FR-010**: The webhook MUST return HTTP 200 (with a small JSON body summarizing the outcome) for every well-formed request — including the "no matching IntegratedAgent / template" cases, the "no action required" case, and the auto-demote case (FR-006c / FR-007d). The response body MUST include at minimum: `detail` (a short human-readable summary), `templates_updated` (the count of Version rows whose `status` was transitioned by this request — either to `"FLAGGED"` per FR-007 OR to `"APPROVED"` per FR-007d; every status write originating from this webhook is counted, regardless of direction), and `integrated_agents_inspected` (the count of IntegratedAgents the system iterated over). Operators distinguish the two write directions via the audit-log `event_name` token (`flagged` vs `auto_demoted` per FR-009a) rather than via the response counter, which is intentionally direction-agnostic so a single number captures total churn per request. Replays and no-action paths (`flag_replay_noop`, `no_action_required`) increment `templates_updated` by `0` because no write occurred. The `templates_updated` and `integrated_agents_inspected` values MUST equal the values reported on the FR-009c `completed` log line for the same request (the HTTP response body and the `completed` audit-log line are two views of the same counters). Optional additional fields (e.g. a per-template breakdown, or a per-direction `flagged_count` / `demoted_count` split) MAY be included.
- **FR-010a**: The webhook MUST return HTTP 400 with a field-level error body for malformed requests (missing required field, wrong UUID format, empty `template_name` / `template_category` / `template_correct_category`). HTTP 401 / 403 are returned by the DRF permission layer for authentication failures (no custom handling).
- **FR-010b**: The webhook MUST return HTTP 500 ONLY for genuinely unexpected exceptions (e.g. database connection lost mid-request). Domain-level "couldn't find X" cases MUST be HTTP 200 per FR-004b, FR-005, and FR-005a — they are not server errors.

- **Version**: The status-bearing row for a Template's lifecycle. Persisted in `retail.templates.models.Version`. Carries the `status` field whose allowed values include `FLAGGED` (added by spec 002's FR-006). The webhook mutates exactly one column on one row per matched Template: `Version.status` is updated to either `"FLAGGED"` (when a flagging condition fires per FR-006 against a non-`FLAGGED` Version) or `"APPROVED"` (when the auto-demote condition fires per FR-006c against an already-`FLAGGED` Version). ## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: When the webhook fires for a template whose category is incorrect (per FR-006), the template's `current_version.status` is `FLAGGED` within 1 second of the webhook returning HTTP 200, in 99.9% of well-formed requests. (The single-row UPDATE is bounded by the DB write latency; 1 second is a generous ceiling that accounts for connection-pool contention.)
- **SC-002**: After a template is flagged by this webhook, the next broadcast attempt against that template (via the existing order-status dispatch path) is skipped 100% of the time — verified by checking that no new `BroadcastMessage` row is persisted for that template until its status is restored to `APPROVED` (either automatically by this webhook on a subsequent corrected-category payload per FR-006c / FR-007d, or operator-driven via `UpdateTemplateUseCase` per FR-014).
- **SC-003**: For an incoming webhook whose payload references a `(project_uuid, app_uuid, template_name)` tuple that matches at least one IntegratedAgent + Template, the response is HTTP 200 in 99.9% of cases. The remaining 0.1% (genuinely unexpected exceptions like DB connection loss) return HTTP 500; the upstream courier's retry policy reconverges these eventually.
- **SC-004**: Idempotency holds: when the same payload is replayed N times (N ≤ 100, the courier's maximum retry budget), the final Version state is the same as after the first call, and the database receives exactly one `UPDATE` against `Version.status` (no churn from redundant writes).
- **SC-005**: Audit log coverage is 100%: every well-formed webhook invocation produces at least one audit log line whose `event_name` token is drawn from the FR-009a enumeration, and the per-request sequence pinned by FR-009c is honored (a `received` line at start, outcome line(s) after the lookup, and a `completed` line at end on the success path — or an `unexpected_error` line on the caught-exception path). Zero invocations produce no log line at all.
- **SC-006**: Cross-tenant isolation holds: in zero cases does a webhook for project A flag a template owned by project B, even when the `app_uuid` value coincidentally appears on a Version row in project B. (This is enforced by FR-004's `IntegratedAgent.project_id == project_uuid` clause.)
- **SC-007**: Per-request latency target: p99 response time under 500 ms in steady state. The realistic traffic profile is sparse — ≤1 webhook per minute across the entire fleet in steady state, with bursts of ≤10 webhooks per second during a rare Meta-driven mass re-categorization event (operator-driven, not user-driven). Because the endpoint's hot path is two reads plus N≤2 single-row `UPDATE`s (no outbound HTTP, no synchronous broadcast), latency is bounded structurally by the database round-trip; no PR-time load test is warranted, and the SLO is monitored post-deploy via standard per-endpoint p99 observability rather than via a release-gate benchmark.

## Assumptions

- **A1**: The Integrations service is the only caller of this webhook. The internal-communication authentication (`CanCommunicateInternally`) is sufficient; no separate API key or HMAC verification is added (consistent with the existing `TemplatesStatusWebhook` at `retail/webhooks/templates/views/template_status_update.py`).
- **A2**: The Integrations service is responsible for upstream filtering. Per the user's description, this webhook is invoked specifically for "direct send template incorrect category detection". Retail does NOT additionally gate on `IntegratedAgent.config["direct_send"] == True` — if Integrations sends an event for a non-Direct-Send template (which would be an upstream bug), Retail still processes it as written. The dispatch-time gate from spec 002's FR-012 already handles `FLAGGED` for both Direct Send and legacy paths consistently.
- **A3**: The `template_category` value namespace is owned by Integrations / Meta. Retail does not maintain its own enumeration of valid categories and does not normalize the value (case, whitespace). The literal `"UTILITY"` (uppercase, no trimming) is the only category value the flagging condition compares against; any other value triggers the "not UTILITY" clause.
- **A4**: The `app_uuid` → IntegratedAgent linkage is materialized exclusively via `Version.integrations_app_uuid`. There is no separate column on `IntegratedAgent` storing `app_uuid` — `IntegratedAgent.channel_uuid` is a different identifier (the WhatsApp channel UUID, not the WhatsApp Cloud app UUID). The lookup in FR-004 reflects this constraint.
- **A5**: An IntegratedAgent may be linked to a single `app_uuid` (the one stored on every Version of every Template it owns, set at template-creation time per `retail.templates.usecases._base_template_creator._create_version`). Cross-app linkage (multiple `app_uuid` values across one IntegratedAgent's Versions) is operationally improbable and is not specially handled — the lookup uses `EXISTS (SELECT 1 FROM Version WHERE integrations_app_uuid = :app_uuid)` semantics, which correctly handles the rare cross-app case.
- **A6**: Email notification (the diagram's `N → O` node) is out of scope for this feature (see FR-011). The audit log lines from FR-009 are the v1 operator observability surface. A follow-up feature MAY add email if operational data shows the audit log alone is insufficient.
- **A7**: This feature depends on spec 002's FR-006 (the `FLAGGED` value being a valid choice on `Version.STATUS_CHOICES`) and FR-007 / FR-012 (the dispatch gate that treats `FLAGGED` as broadcast-disabling). Both have already landed in `retail/templates/models.py` and `retail/agents/domains/agent_webhook/services/broadcast.py`; no further model migration or dispatch-path change is required by this feature.
- **A8**: The OrderStatus agent's locally-persisted Template `name` values follow lowercase snake_case (per `retail.templates.usecases._base_template_creator._create_template`, which writes `template.name = payload["template_name"]` verbatim from upstream where the convention is snake_case). The webhook's case-sensitive Template lookup (FR-005) is consistent with this convention.
- **A9**: The webhook's lookup may match multiple IntegratedAgents for the same `(project_uuid, app_uuid)` pair (e.g. an inactive old assignment plus an active new one, or two distinct agent types — OrderStatus + a custom agent — that share the same app). The behavior is "fan out and update all": the webhook processes each matched IntegratedAgent's Template independently. Operators reading the audit log see one entry per IntegratedAgent processed.
- **A10**: The webhook does NOT introduce a new database migration. The `FLAGGED` status value already exists on `Version.STATUS_CHOICES` (added by spec 002's migration `0017_alter_version_status_paused_flagged.py`). No new column, table, index, or constraint is needed.
- **A11**: Operators who manually set `FLAGGED` via `UpdateTemplateUseCase` for reasons unrelated to a Meta-side category determination accept that a subsequent corrected-category webhook payload (`template_correct_category == "UTILITY"`, regardless of `template_category`) will auto-demote that Version to `APPROVED` per FR-006c / FR-007d. The webhook does not track flag source (no new `Version.flagged_source` column is added, preserving Assumption A10's "no new database migration" commitment) and treats every `FLAGGED → APPROVED` transition triggered by a payload whose `template_correct_category == "UTILITY"` as a valid corrected-category recovery. Operators needing a non-recoverable broadcast block should use a different `Version.status` value (e.g. `PAUSED` or `REJECTED`) which is NOT affected by this webhook: the auto-demote condition (FR-006c) explicitly requires `current_version.status == "FLAGGED"`, so operator-set `PAUSED` / `REJECTED` / `PENDING` / `DELETED` Versions are never written by this webhook (the FR-006-fires branches against a `PAUSED` / `REJECTED` / `PENDING` / `DELETED` Version still transition to `FLAGGED` per the spec.md Edge Cases row on non-`APPROVED`, non-`FLAGGED` starting states; the auto-demote branch never fires against any non-`FLAGGED` starting state). (Pinned by Clarifications sessions 2026-05-25.)
- **A12**: The fan-out cardinality bound — the number of IntegratedAgents returned by the FR-004 lookup for any single `(project_uuid, app_uuid)` pair — is operationally bounded by the assignment cadence (typically 1; multi-IntegratedAgent fan-out is exceptional and reflects historical inactive assignments per A9). The webhook does NOT impose an explicit cap, does NOT page the queryset, and does NOT chunk the audit-log emissions: every matched IntegratedAgent is processed and one audit line per inspected agent is emitted in the same request. A pathological case (e.g. 1000+ IntegratedAgents on the same `app_uuid` after a bulk re-assignment) would produce 1000+ audit lines and 1000+ single-row UPDATEs in one request, which the webhook would still handle correctly — the throughput remains bounded by the DB write rate and the audit-log shipping rate, not by an in-process limit. Pagination / batching is OUT OF SCOPE for v1; a future feature MAY introduce a cap if operational data shows the unbounded fan-out is a latency or cost driver.
